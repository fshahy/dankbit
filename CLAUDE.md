# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Running the Stack

```bash
# Start all services
docker compose up -d

# Rebuild after code changes
docker compose up -d --build

# Rebuild a single service (e.g. after editing dankbit_ws_batch.py)
docker compose up -d --build dankbit_ws

# Tail logs
docker compose logs -f
docker compose logs -f dankbit_ws

# Restart just the Odoo addon (no rebuild needed for Python changes to my_addons)
docker compose restart web
```

After first startup, install the Dankbit addon via Odoo UI: Apps → search "Dankbit" → Install.

**Required `.env` variables:**
```
DANKBIT_POSTGRES_DB=<db_name>   # any name; db1 is the development default
DERIBIT_KEY=<key>
DERIBIT_SECRET=<secret>
```
The `UID` and `GID` env vars are also needed so the `web` container writes files as the host user (dev default: `1000`/`1000`). `docker-compose.yml` sets `user: "${UID}:${GID}"` on the `web` service and `chown`s the Odoo data dir to that UID/GID before starting — the values must match whichever host user owns the bind-mounted directories, so Odoo's writes (filestore, logs) land with an owner the host can actually manage. Check with `ls -ln <bind-mounted dir>` if unsure. Setting it to `0`/`0` (root) is correct only when the production host directory itself is root-owned — it also means Odoo runs as root inside the container, a broader security surface than an unprivileged UID.

> **Production note:** `config/odoo.conf` contains `dbfilter = ^db1$` — update this to match the actual database name.
> **Production note:** production's `.env` uses `UID=0`/`GID=0` (root), unlike the dev default of `1000`/`1000` — see above for when that's the correct choice.

## Architecture

Two Docker services talk to a shared PostgreSQL database:

```
Deribit WS API ──► dankbit_ws (Python asyncio) ──► PostgreSQL
                                                        │
                   Deribit REST API ◄── Odoo cron       │
                                                        ▼
                                               Odoo 18 (web:8069)
                                                        │
                               ┌───────────────────────┤
                               │                       │
                        PNG charts                TradingView
                  /BTC-<EXPIRY>, /i/BTC-<EXPIRY>  /chart/BTC
```

**Trade data enters two ways:**
1. **WebSocket (primary):** `dankbit_ws_service/dankbit_ws_batch.py` — connects to Deribit, authenticates, fetches all BTC+ETH option instruments, subscribes to `trades.<instrument>.raw` channels in chunks of 400, inserts rows directly into PostgreSQL with `ON CONFLICT IGNORE` on `deribit_trade_identifier`.
2. **REST backfill:** `Trade.get_last_trades()` — runs every minute via cron; ensures no trades were missed by paging through the last few days of Deribit REST API history.

**PNG chart rendering:** Per-expiry routes like `/BTC-7MAY26` aggregate all trades for that expiry, compute a Black-Scholes portfolio delta and dollar-gamma (GEX = Γ × S²) curve, and return a PNG via matplotlib's Agg backend (`Figure + FigureCanvas`, never `pyplot` — server-safe). Overlaid lines: gamma peaks and bottoms (dashed black), delta=0 crossings (solid — green for "supply" [delta negative below, positive above], red for "demand" [delta positive below, negative above]).

**Zones chart (`/<instrument>/zones`):** a distinct plot mode — separates that instrument's trades since 00:00 UTC into buy-side and sell-side (trade domain uses `("name", "=ilike", f"{instrument}-%")`, an anchored/left-prefix match — not a bare `ilike` substring match — so a query for one expiry can never pull in another instrument's trades), builds two independent `OptionStrat` payoff curves via `options.build_zone_curves()` (`long_call`/`long_put` for buys, `short_call`/`short_put` for sells — same payoff math as the combined-portfolio routes, just accumulated into two separate objects, plus the crossing-based auto-zoom below — same function `dankbit.zones.extrema` calls, see Models), and plots them together via `OptionStrat.plot_zones()`: green "Longs" curve, red "Shorts" curve, blue vertical line at index price — `plot_zones()` itself draws no text overlay. The `Seller Max Profit (SMP)`/`Buyer Max Loss (BML)`/`High Zone`/`Low Zone`/`Middle Zone` info (previously drawn inside the PNG via `ax.text()`) is now rendered as page HTML instead, alongside further per-expiry stats: `chart_png_zones` calls `options.zone_summary(STs, longs_curve, shorts_curve)` (module-level function, same extrema/box-boundary definitions as `dankbit.zones.extrema` — see Models) for `seller_max_profit`/`buyer_max_loss`/`high_zone`/`low_zone`/`middle_zone`/`high_resistance`/`low_support`, labeled on the page as `Seller Max Profit (SMP)`/`Buyer Max Loss (BML)`/`High Zone`/`Low Zone`/`Middle Zone`/`High/Resistance`/`Low/Support` (the `(SMP)`/`(BML)` parenthetical added to match Thales's own "Buyer Max Loss / BML"/"Seller Max Profit / SMP" labeling, same convention the Gamma/Delta/Theta/Vega headers below already use). Current price is not a factor in any of these: `high_zone`/`low_zone` (renamed from `top_box`/`bottom_box`) are each curve's own highest/lowest zero-crossing (the Shorts curve's and Longs curve's max feed `high_zone`, their min feeds `low_zone`) rather than "the nearest crossing above/below index price" — a zone forms as soon as *either* curve has at least one crossing (a lone curve's crossing yields a degenerate same-price pair, rendered as a single price rather than a dashed range); only `None`/`"n/a"` if neither curve ever crosses zero. `middle_zone` is a third zone bounded by `seller_max_profit`/`buyer_max_loss` (min/max of the two) rather than any zero-crossing — always defined, since those two values are argmax/argmin over the full curve and don't depend on a crossing existing, unlike `high_zone`/`low_zone`. Then restricts that route's own `trades` to the single nearest expiry (mirroring `dankbit.zones.extrema`'s "next expiry only" restriction) and delegates the actual per-leg gamma/delta/theta/vega extrema computation to `options.per_leg_greeks()` — the single source of truth for this also used by `dankbit.zones.extrema`'s `gamma_band`/`delta_band` and by `forecast3.per_leg_greeks()`, so this page, the persisted zones-extrema record, and the Thales Forecast engine can never quietly disagree on these numbers for the same trades. Gamma extrema are preceded in `zone_info_lines` by a violet-colored `Gamma` section-header line: `Buyer Call Gamma (BCG)`/`Buyer Put Gamma (BPG)` (price where the long-call/put gamma curve peaks, `argmax`) and `Seller Call Gamma (SCG)`/`Seller Put Gamma (SPG)` (price where the short-call/put gamma curve bottoms out, `argmin` — short positions carry negative gamma, so the relevant extremum is a trough, not a peak — renamed from `Long Call Gamma Peak`/`Long Put Gamma Peak`/`Short Call Gamma Bottom`/`Short Put Gamma Bottom` to match Thales's own BCG/BPG/SCG/SPG terminology, the same abbreviations `dankbit.forecast3.snapshot`'s `bcg_price`/`bpg_price`/`scg_price`/`spg_price` fields already used) — plus, in their own block right below, the gamma value (not dollar-formatted, unlike the price lines) at each of those same 4 extrema (`{curve}[argmax/argmin index]`, the y-value at the point already located for the price line, not a separate computation) — shown as an absolute value divided by 1,000,000 and rounded (the sign is already conveyed by which side of the section it's on: peaks are long/positive, bottoms are short/negative), since these raw dollar-gamma numbers run into the hundreds of millions and the user copies them by hand into another tool — labeled `BCG Abs.`/`BPG Abs.`/`SCG Abs.`/`SPG Abs.` (renamed from `Long Call Gamma Peak Value`/`Long Put Gamma Peak Value`/`Short Call Gamma Bottom Value`/`Short Put Gamma Bottom Value`). Also computes each leg's delta-saturation price via `options.delta_saturation_price()` (calls saturate ITM at high S, puts at low S, independent of long/short, `fraction=DELTA_SATURATION_FRACTION`=0.9 — see below), preceded by a green-colored `Delta` header, labeled `Buyer Call Delta (BCD)`/`Buyer Put Delta (BPD)`/`Seller Call Delta (SCD)`/`Seller Put Delta (SPD)` (renamed from `Long Call Delta`/`Long Put Delta`/`Short Call Delta`/`Short Put Delta` to match Thales's own BCD/BPD/SCD/SPD terminology, the same abbreviations `dankbit.forecast3.snapshot`'s `bcd_price`/`bpd_price`/`scd_price`/`spd_price` fields already used), plus — in their own block right below — the portfolio-delta value at that same price, labeled `BCD Abs.`/`BPD Abs.`/`SCD Abs.`/`SPD Abs.` (renamed from `Long Call Delta Value`/`Long Put Delta Value`/`Short Call Delta Value`/`Short Put Delta Value`) — shown as an absolute value divided by 10 and rounded to a whole number, same reasoning as the Gamma Value lines above (dividing by 1,000,000 there, 10 here, since delta values are already two to three orders of magnitude smaller). That value is interpolated off the underlying `delta.portfolio_delta()` curve at the saturation price rather than assumed to equal `fraction * curve edge` exactly, since `delta_saturation_price()` falls back to the `STs` edge (not a true 90%-threshold crossing) when the curve never reaches that threshold within this window — interpolating keeps the reported value correct in that fallback case too. Also computes, per leg, the price where portfolio theta ($/day — see `theta.py`, a new module mirroring `delta.py`/`gamma.py`'s `bs_*`/`portfolio_*` pattern; `bs_theta()` returns the annualized Black-Scholes theta ÷ 365, the convention traders quote, not the raw per-year value) is most extreme over the same price grid gamma/delta use — sign-flipped from gamma's peak/bottom split: long positions carry negative theta (decay cost), so their extremum is the trough (`argmin`, worst decay); short positions carry positive theta (decay gain), so theirs is the peak (`argmax`, best decay) — against the same next-expiry-only `long_calls`/`long_puts`/`short_calls`/`short_puts` recordsets gamma/delta reuse, with the same `r=0.0` convention as this page's other Greeks. Preceded by an orange-colored `Theta` header, labeled `Buyer Call Theta (BCT)`/`Buyer Put Theta (BPT)`/`Seller Call Theta (SCT)`/`Seller Put Theta (SPT)` (renamed from `Long Call Theta`/`Long Put Theta`/`Short Call Theta`/`Short Put Theta` to match Thales's own BCT/BPT/SCT/SPT terminology, the same abbreviations `dankbit.forecast3.snapshot`'s `bct_price`/`bpt_price`/`sct_price`/`spt_price` fields already used) — plus, in their own block right below, the theta value ($/day) at each of those same 4 extrema (`{curve}[argmin/argmax index]`, same pattern as the gamma/delta value blocks), labeled `BCT Abs.`/`BPT Abs.`/`SCT Abs.`/`SPT Abs.` (renamed from `Long Call Theta Value`/`Long Put Theta Value`/`Short Call Theta Value`/`Short Put Theta Value`) — shown as an absolute value divided by 10,000 and rounded, same reasoning as the Gamma/Delta Value lines (dividing by 1,000,000 for gamma, 10 for delta, 10,000 here, since raw $/day theta values sit in between those two scales). Also computes, per leg, the price where portfolio vega (`vega.py`, a module mirroring `gamma.py`'s `bs_*`/`portfolio_*` pattern rather than `theta.py`'s — vega has no call/put distinction in Black-Scholes, so `bs_vega()` takes no `option_type`, same as `bs_gamma()`; returns raw vega / 100, the $-per-1-IV-point convention traders quote, mirroring `bs_theta()`'s /365) is most extreme over the same price grid — same peak/bottom split as gamma, not theta's flipped one: vega is always positive for a long option and always negative for a short one, so long legs peak (`argmax`) and short legs bottom out (`argmin`) — against the same next-expiry-only `long_calls`/`long_puts`/`short_calls`/`short_puts` recordsets and `r=0.0` convention as this page's other Greeks. Preceded by a blue-colored `Vega` header, labeled `Buyer Call Vega (BCV)`/`Buyer Put Vega (BPV)`/`Seller Call Vega (SCV)`/`Seller Put Vega (SPV)` (renamed from `Long Call Vega`/`Long Put Vega`/`Short Call Vega`/`Short Put Vega` to match Thales's own BCV/BPV/SCV/SPV terminology, the same abbreviations `dankbit.forecast3.snapshot`'s `bcv_price`/`bpv_price`/`scv_price`/`spv_price` fields already used) — plus, in their own block right below, the vega value ($ per 1 IV point) at each of those same 4 extrema, labeled `BCV Abs.`/`BPV Abs.`/`SCV Abs.`/`SPV Abs.` (renamed from `Long Call Vega Value`/`Long Put Vega Value`/`Short Call Vega Value`/`Short Put Vega Value`) — shown as an absolute value divided by 100 and rounded, same reasoning as the other Value lines (raw portfolio-vega values sit in the low thousands, an order of magnitude below theta's ten-thousands and well above delta's tens, so /100 keeps the displayed number in the same easy-to-copy 2-3 digit range). Unlike every block above Theta, Theta and Vega are kept in a *separate* `right_info_lines` list rather than appended to `zone_info_lines` (passed into the shared template under the `theta_info_lines` context var name, kept for backward compatibility even though it now carries both Greeks), and rendered in its own top-right `.zone-info-right` overlay (mirrors `.zone-info`'s styling, just `right: 12px` instead of `left: 12px` for the box position — text inside stays left-aligned, same as `.zone-info`) — so it sits to the right of the PNG instead of stacking underneath Gamma/Delta on the left. All of this is assembled into `zone_info_lines`, where every line is a `{text, color}` dict rather than a plain string (color `None` for the default black styling; only section headers set one), rendered by the template via `t-att-style` on that per-line `<div>` (grouped into 7 blocks — price extrema; boxes; intersections; gamma peak/bottom prices; gamma peak/bottom values; delta-saturation prices; delta-saturation values — separated by blank spacer lines, a `" "` non-breaking space since a truly empty/whitespace-only `<div>` can collapse to zero height) and passed to the shared `dankbit_page` template's `zone_info_lines` context var, rendered as a fixed top-left overlay div (`.zone-info`, `user-select: text` so it's copyable, only shown when that var is present, so every other PNG route using `dankbit_page` is unaffected). A separate bottom-left annotation (added by `chart_png_zones` itself, still drawn inside the image via `ax.text()`, not moved) shows the long/short trade counts since 00:00 UTC. Restores a feature this repo had (and removed) in its early history (`d416bcd` "add zones" → removed in `6c293d6`), reimplemented against the current architecture rather than reusing the old pyplot-based code. **Auto-zoomed x-axis:** the configured `from_price`/`to_price` settings are only used for an initial wide pass to locate where the Longs/Shorts curves cross (linear interpolation on the sign change of `longs.payoffs - shorts.payoffs`, same technique as the delta=0 crossing finders); the curves are then rebuilt over `[leftmost_crossing - margin_below, rightmost_crossing + margin_above]` at the same `steps` resolution, so the rendered chart is tightly framed around the relevant region instead of the full (much wider) configured range. Margin is asset-dependent: BTC uses ±$2000; ETH uses ±$100 (ETH's much smaller price scale made the ±$2000 BTC margin blow out the auto-zoom). Falls back to the original wide range if the curves never cross.

**Single-leg delta/gamma routes (`/<instrument>/lp`, `/lc`, `/sp`, `/sc`):** four routes, one per leg type (long put, long call, short put, short call) — `_chart_png_single_leg(instrument, leg_key)` is the shared implementation, dispatched via `_LEG_ROUTES[leg_key]` (`direction`/`option_type` domain filter, which `OptionStrat` leg method accumulates trades, route label, and delta-saturation `fraction`/`side` — see below). Same domain convention as `chart_png_zones` (anchored `=ilike` instrument match, trades since 00:00 UTC), but restricted to that one direction+option_type combination and built over the **full configured price range** (`from_price`/`to_price`/`steps`), not the crossing-based zoomed window `/zones` uses — matching `chart_png_hours`/`chart_png_all`'s style instead, including their `r=0.05` risk-free rate for `delta.portfolio_delta`/`gamma.portfolio_gamma` (deliberately *not* `r=0.0`/zoomed-grid like `dankbit.zones.extrema`, a choice reverted after being tried the other way). `obj.plot()` renders the same delta+gamma dual-axis chart as those routes; the gamma peak/bottom + delta=0 crossing overlay is factored into `_annotate_gamma_delta_crossings(ax, STs, market_deltas, market_gammas)` (shared by all 4 routes, so they don't each carry their own copy of that block). A solid green `axvline` additionally marks the delta-saturation price from `options.delta_saturation_price()` — see `options.py` below — labeled with its price, drawn after the standard overlay.

**TradingView chart:** `/chart/BTC` and `/chart/ETH` serve a Lightweight Charts v4 page with live Deribit perpetual futures candles (proxied server-side) and price lines refreshed on `dankbit.refresh_interval` — except the yellow/teal zones boxes, which refresh on their own dedicated `dankbit.zones_box_refresh_interval` setting (default 3600s/1h, see Settings fields below):
- Delta=0 lines (3 sets) — each a single `createPriceLine`, `lineWidth: 1`, `lineStyle: Solid`. Color is by crossing type, not by set: **black** ("S/D" in the legend) for "supply" (delta negative below, positive above), **red** ("Tie-out" in the legend) for "demand" (delta positive below, negative above — the important case). Set identity is shown via the line's `title` only. Every `createPriceLine` call across the chart (delta=0 sets, gamma peak/bottom, ruler) sets `axisLabelColor: 'black'` so the axis label box is always black regardless of the line's own color — only the label background is forced black, not the line stroke — **except** Daily (24H)/Daily+1 (24H) (below), whose axis label is `axisLabelColor: 'orange'` instead, a deliberate one-off deviation from the rest of the chart. `addLineSeries` (Net Call/Put Gamma) has no equivalent per-series label-color override in Lightweight Charts v4, so those labels still match their line color (blue/orange):
  - **Weekly** — sourced from `/api/delta-zero/<instrument>`
  - **Monthly** — sourced from `/api/delta-zero/<monthly-instrument>`
  - **All** — all active expiries; sourced from `/api/delta-zero-all/<asset>`
- Daily (24H) / Daily+1 (24H) delta=0 lines — two distinct `createPriceLine` sets, color-by-crossing-type like Weekly/Monthly/All (black for "supply" — delta negative below, positive above; red for "demand" — delta positive below, negative above), but with an **orange** axis label (`axisLabelColor: 'orange'`, unlike every other line on this chart — see above) and **no footer-legend entry**, titles `Daily (24H)`/`Daily+1 (24H)`, sourced from `/api/delta-zero-tomorrow/<asset>` and `/api/delta-zero-day-after-tomorrow/<asset>` respectively — both thin wrappers around the shared `_delta_zero_for_calendar_day(asset, days_ahead)` helper (`days_ahead=1`/`2`) so the two can never disagree on how a calendar-day expiry/trade-window is computed. Each targets the specific expiry landing `days_ahead` calendar days out (UTC) — computed directly as `today + N days` formatted to Deribit's no-leading-zero day convention (e.g. `7JUL26`), not "nearest active expiry" (`MIN(expiration) >= NOW()`), since the latter can still resolve to *today's* not-yet-happened expiry. Restricted to trades from the trailing 24h (`deribit_ts >= NOW() - INTERVAL '24 hours'`). Gated by `show_daily_lines` (see Settings fields below).
- Zones Extrema lines (3) — `High/Resistance` (green, `#2e7d32`, solid), `Low/Support` (red, `#c62828`, solid — renamed from `Top Intersection`/`Bottom Intersection` to match Thales's own Resistance/Support terminology for this reference band), and `Gamma Band` (light violet, `#c9a0dc`, **dashed** — `LightweightCharts.LineStyle.Dashed`), `addLineSeries` on the **default/main price scale** (no `priceScaleId` override) since these are plain BTC/ETH price levels, not dollar amounts — they render directly alongside the candles. Sourced from `/api/zones-extrema/<asset>`: one point per **instrument** stored in `dankbit.zones.extrema` (see Models — one continuously-refined-then-frozen record per expiry, not a time-series of polls), positioned at that instrument's own expiration time rather than a poll timestamp (there isn't one anymore) — so these are term-structure-style lines connecting each expiry's final value to the next expiry's, not a history of one expiry's value changing over wall-clock time. `Gamma Band` is the average of that expiry's 4 gamma extrema (Buyer Call/Put Gamma (BCG/BPG), Seller Call/Put Gamma (SCG/SPG) — see Models, `dankbit.zones.extrema._compute_asset`'s `gamma_band`), the same 4 numbers the `/<instrument>/zones` PNG page's info overlay shows. The API response also carries `delta_band` — the average of that expiry's 4 delta-saturation prices (see `options.delta_saturation_price()`) — but it is deliberately **not drawn** on this chart (no `deltaBandSeries` in the template); it's computed and persisted for later use, not currently rendered. The endpoint derives each point's `t` via a single grouped lookup against `dankbit_trade` (`SUBSTRING(name FROM '^[^-]+-[^-]+')` to recover the instrument prefix, `MIN(expiration)`) rather than parsing the instrument string's day-suffix and assuming a settlement hour — raw SQL, so it still finds archived (past-expiry) instruments' trades despite the ORM's implicit `active=True` filter, since those rows must keep contributing a fixed historical point. Connected point-to-point by the line series. Shown on **every timeframe** (1h/4h/1d) — no longer restricted to 4h; `setTf()`/`init()` call `refreshZonesExtrema()` on every timeframe switch regardless. High/Resistance and Low/Support points additionally carry a `+`/`-` `setMarkers()` label (via `high_resistance_positive`/`low_support_positive` — see Models) showing whether the payoff *at* that intersection sits above or below the zero-payoff line, since the crossing's x-price alone doesn't say — `size: 0` hides the marker's shape glyph so only the text renders, colored to match its own line.
- Zones boxes (2, plus a 3rd Seller Max Profit/Buyer Max Loss box — see below; toggled together by the "Zones" checkbox, TradingView Chart Notes below — checked by default) — filled rectangles (`#fbc02d`, 25% opacity fill, matching top+bottom border lines) marking the top and bottom zero-crossing zones for the **nearest** active expiry only — there is no second (blue) box for a later expiry any more (removed; see Models, `get_box_n`/`TRACKED_EXPIRY_COUNT` for how later expiries are still tracked, just without a box). The 4 box-boundary fields themselves are **not** read from stored history — sourced live from `/api/zones-box/<asset>`, which calls `dankbit.zones.extrema.get_box()` to compute fresh on every request via `_compute_asset()` (see Models: nothing reads box-boundary history, only the latest value is ever drawn). As a side effect, `get_box()` *does* upsert that instrument's zones-extrema record (`index_price`/`high_resistance`/`low_support`/`gamma_band`/`delta_band`/`high_zone_min`/`high_zone_max`/`low_zone_min`/`low_zone_max`/`middle_zone_min`/`middle_zone_max`/`seller_max_profit`/`buyer_max_loss`, plus the 32 per-leg Greek+Abs fields — see Models, `_persist_extrema` — not the 4 zero-crossing box-boundary fields) on every call — this endpoint's own polling (at `dankbit.zones_box_refresh_interval`, a dedicated setting separate from `dankbit.refresh_interval`, default 3600s/1h — see Settings fields below) is the primary path feeding the Zones Extrema lines' per-expiry history, with a 4-hourly cron as a fallback for when nobody's viewing the chart (see Models, `_persist_extrema`). The boxes' own visual state is never affected by the cron either way — box-boundary fields are recomputed fresh on whatever request (browser or cron) triggers `_compute_asset()`, but only the browser's request result is ever rendered; the cron's result is only ever persisted, never displayed. Each box is built from two `addBaselineSeries` (`aboveBoxSeries`/`belowBoxSeries` — a flat top-price line filled down to a `baseValue` bottom price) plus a plain `addLineSeries` for the bottom edge (`aboveBoxBottomLine`/`belowBoxBottomLine`), since `addBaselineSeries` only strokes its top edge. Top/bottom of each box are `max`/`min` of that side's two zero-crossing prices (`short_zero_above_price`/`long_zero_above_price` for the top box, `short_zero_below_price`/`long_zero_below_price` for the bottom box) — despite the field names, these are each curve's own highest (`*_above_price`) or lowest (`*_below_price`) zero-crossing, not "the nearest crossing above/below current index price"; names kept for backward compatibility (API/JS consumers), see Models, `_compute_asset`. `setBoxData()` (in `dankbit_templates.xml`) draws the full rectangle when both a top and bottom price exist and differ; if only one curve contributed a crossing, or both coincide (a zero-height box), it instead draws a single line at whichever price is available (reusing the box's own bottom-edge line series) rather than dropping it silently; only clears both when neither curve crosses zero at all (a `0.0` value on **both** sides). Time span is `[now - 10 days, expiration]` — the right edge is that expiry's own real settlement time (Deribit's `expiration_timestamp`, e.g. 08:00 UTC / 10:00 CEST — read off `dankbit_trade.expiration` via `_compute_asset()`, not assumed/hardcoded), so the box now visibly runs all the way out to when the trades it's summarizing actually stop mattering, rather than stopping at "now". The left edge is a flat 10 days back from the browser's current wall-clock time (`Date.now()`, tz-shifted like every other timestamp on this chart), recomputed on every poll rather than fixed to a stored tick — previously anchored to the live lookup's own `computed_at` timestamp and snapped to 18:00 on that calendar day (`snapTo18()`, since removed along with this change). Shown on **every timeframe** (1h/4h/1d) — no longer restricted to 4h, same as the Zones Extrema lines.
- Seller Max Profit / Buyer Max Loss box (3rd box, teal) — same mechanism (`addBaselineSeries` + bottom-edge `addLineSeries`, `setBoxData()`'s single-line fallback when top/bottom coincide) and the exact same `fromTime`/`toTime` as the yellow box above, since both come from the same `/api/zones-box/<asset>` response for the nearest expiry — `smpBmlBoxSeries`/`smpBmlBoxBottomLine`. Top/bottom are `max`/`min` of `seller_max_profit` (price where the Shorts payoff curve peaks) and `buyer_max_loss` (price where the Longs curve bottoms out) — restored model fields, see Models, `dankbit.zones.extrema` (renamed from `short_max_price`/`long_min_price` to match Thales's own SMP/BML terminology). No separate endpoint: `seller_max_profit`/`buyer_max_loss` were already computed by `_compute_asset()` as a side effect of the same `get_box()` call the yellow box uses, just added to `/api/zones-box/<asset>`'s JSON payload.

## Odoo Addon (`my_addons/dankbit/`)

The addon depends only on `website`. Key components:

**Models:**
- `dankbit.trade` — Core model. Fields map directly to Deribit trade fields. `strike` and `option_type` are computed from `name` (instrument name like `BTC-29NOV24-98000-P`). `days_to_expiry` is UTC-safe. `get_hours_to_expiry()` returns continuous time used in Black-Scholes.
- `dankbit.zones.extrema` — **One record per instrument** (e.g. `"BTC-10JUL26"`, unique-constrained on `instrument`), not per snapshot-in-time — a row's position on the TradingView chart is still that instrument's own expiration time, looked up from `dankbit_trade.expiration` when the API serves it (see below and TradingView Chart Notes' Zones Extrema lines), never `computed_at` below. Fields: `computed_at` (Datetime, `default=fields.Datetime.now` — backend visibility only, e.g. "how stale is this row"; refreshed to the moment `_compute_asset()` ran on every `_persist_extrema()` upsert, not just set once at creation — a deliberate reversal of this model's earlier "no timestamp field" design, added at explicit user request), `asset`, `instrument`, `index_price`, `high_resistance`/`low_support` (renamed from `top_intersection`/`bottom_intersection` to match Thales's own Resistance/Support terminology for this reference band — the highest/lowest price where the Longs-vs-Shorts payoff curves cross each other — not where either crosses zero, and **not** relative to `index_price` — labels kept for backward compatibility, but with only one crossing that single point can land on either side of price by a trivial amount, which used to make the "other" field silently read `0.0` even though the plot clearly showed one real intersection; same `longs.payoffs - shorts.payoffs` sign-change `build_zone_curves()` finds internally for its own ±$2000 auto-zoom), `high_resistance_positive`/`low_support_positive` (Boolean — whether the *payoff value* at that intersection, not its price, sits above or below the zero-payoff line; interpolated via `np.interp` at the crossing price against `longs_obj.payoffs`, which equals `shorts_obj.payoffs` at a true crossing — drives the `+`/`-` marker on the TradingView chart, see TradingView Chart Notes), `gamma_band` (average of 4 gamma extrema — Buyer Call Gamma (BCG), Buyer Put Gamma (BPG), Seller Call Gamma (SCG), Seller Put Gamma (SPG) — the same numbers the `/<instrument>/zones` PNG page's info overlay shows, sourced from `options.per_leg_greeks()` against this same instrument's `trades`/`STs`), `delta_band` (average of 4 delta-saturation prices, one per leg, also from `options.per_leg_greeks()` — not derived from gamma, since portfolio delta and portfolio gamma are numerically related but conceptually distinct measures here), `high_zone_min`/`high_zone_max`, `low_zone_min`/`low_zone_max`, `middle_zone_min`/`middle_zone_max` (same definitions as `options.zone_summary()`'s `high_zone`/`low_zone`/`middle_zone` — see Zones chart above and `options.py` below — stored as a `_min`/`_max` Float pair per zone since a single Float can't hold a range; `high_zone`/`low_zone` default `0.0`/`0.0` when neither curve ever crosses zero, same convention `high_resistance`/`low_support` use, while `middle_zone` is always defined), `seller_max_profit`/`buyer_max_loss` (renamed from `short_max_price`/`long_min_price` to match Thales's own SMP/BML terminology, see `dankbit.forecast3.snapshot`'s `bml`/`smp` fields — price where the Shorts payoff curve peaks / where the Longs curve bottoms out, against this same `longs_obj`/`shorts_obj` — this model's original two fields, added in `8c59981`, repurposed away into `top_intersection`/`bottom_intersection` in `8da5a1a`, and since reintroduced as their own fields alongside those rather than replacing them again; same values `options.zone_summary()`'s `seller_max_profit`/`buyer_max_loss` show on the `/<instrument>/zones` PNG page's info overlay, not exposed via `/api/zones-extrema/<asset>` — model-only, feeds the `/api/zones-box/<asset>` teal box instead, see TradingView Chart Notes), and the 32 individual per-leg gamma/delta/theta/vega price + Abs fields (`bcg_price`/`bcg_abs`/`bpg_price`/`bpg_abs`/`scg_price`/`scg_abs`/`spg_price`/`spg_abs`, and the BCD/BPD/SCD/SPD, BCT/BPT/SCT/SPT, BCV/BPV/SCV/SPV equivalents — see `_PER_LEG_GREEK_FIELDS`) behind `gamma_band`/`delta_band`'s averages — the exact same numbers, same field names, and same scaling (`*_abs` is `abs(value)/scale` — 1e6 gamma, 10 delta, 1e4 theta, 100 vega — matching the `/<instrument>/zones` PNG page's own Abs. lines, not rounded further) as `dankbit.forecast3.snapshot`'s own `bcg_price`/`bcg_abs`/etc. fields, sourced from the same `forecast3_lib.per_leg_greeks()` this model now calls directly (previously called `options_lib.per_leg_greeks()` and only derived `gamma_band`/`delta_band` from it; theta/vega per-leg values were computed then dropped, then briefly removed as `theta`/`vega` average fields entirely, before being reintroduced in this fuller per-leg-plus-Abs shape at the user's request — this model, the `/<instrument>/zones` PNG page, and the Thales Forecast engine can now never quietly disagree on any of these 32 numbers for the same trades, all three ultimately calling the same `options.per_leg_greeks()`). `_compute_asset(asset, expiry_index=0, hours=None)` is the shared computation core (the `expiry_index`-th soonest active expiry — 0 is nearest, 1 is the one after that, via a raw `SELECT DISTINCT expiration` query so a single expiry's 100k+ trades can't crowd out the next one — trades since 00:00 UTC through now for that expiry by default, or the trailing `hours` hours through now when given — see `get_box`'s `hours` override below, `options.build_zone_curves()`, then `forecast3_lib.per_leg_greeks()` for the 32 per-leg fields (`gamma_band`/`delta_band` are simply the average of 4 of those), `high_zone_min`/`high_zone_max`/`low_zone_min`/`low_zone_max` built from the same `short_crossings`/`long_crossings` also used for the 4 raw zero-crossing box boundaries (rather than calling `options.zone_summary()` again and re-finding the same crossings), `middle_zone_min`/`middle_zone_max` from `seller_max_profit`/`buyer_max_loss`, then `high_resistance`/`low_support`/`seller_max_profit`/`buyer_max_loss`, that expiry's own `instrument` string and `expiration` datetime, plus 4 zero-crossing box boundaries — each curve's own highest/lowest zero-crossing, also not relative to `index_price` (same principle as `high_resistance`/`low_support`; see below) — returns `None` if there's nothing computable (missing index price/expiry at that index/since-midnight trades) rather than writing a row that would silently look like real data (e.g. the configured price-range floor). Persisted as a side effect of `get_box_n()` (below), called by the live zones-box polling (`expiry_index` 0, at `dankbit.zones_box_refresh_interval`) and zones-extrema-refresh polling (`expiry_index` 1-2, at `dankbit.refresh_interval`) that already happen on every TradingView chart page, and by a 4-hourly fallback cron, `compute_snapshot()`, since live-polling-only would leave a gap for any instrument nobody happens to be viewing (a real missing point in the per-expiry history, not just a staler one). The cron calls the exact same `get_box_n()` the live endpoints call (for both BTC and ETH, every tracked `expiry_index` — see `TRACKED_EXPIRY_COUNT` below), so it can never compute or persist anything a live page view wouldn't have, and never touches the TradingView horizontal price lines (delta=0, gamma peak/bottom) — only this model.
  - `_persist_extrema(data)` — upserts (search by `instrument`, `write()` if found else `create()`) the one record for `data['instrument']`, storing only the historical-line fields (`computed_at`/`index_price`/`high_resistance`/`low_support`/`high_resistance_positive`/`low_support_positive`/`gamma_band`/`delta_band`/`high_zone_min`/`high_zone_max`/`low_zone_min`/`low_zone_max`/`middle_zone_min`/`middle_zone_max`/`seller_max_profit`/`buyer_max_loss`, plus the 32 per-leg gamma/delta/theta/vega price+Abs fields listed in `_PER_LEG_GREEK_FIELDS`) — `computed_at` is refreshed on every upsert (unlike a plain creation timestamp), so it reflects the moment this specific `_compute_asset()` call ran, not when the row was first created; the 4 zero-crossing box-boundary fields are never persisted. `self.sudo()` internally: `get_box()`/`get_box_n()` are `auth="public"` routes, so this otherwise runs as the anonymous public user, which has no access rights at all on `dankbit.zones.extrema` (`ir.model.access.csv` only grants `base.group_user`) — without the explicit sudo, every real (cookie-less) request 403s even though it works fine from an already-authenticated browser session or `odoo shell`, which is why this went unnoticed when `_persist_extrema` was first introduced. Called from `get_box_n()` for every tracked `expiry_index`, so an instrument's row starts accumulating as soon as it's tracked at all — typically well before it becomes nearest. Either way, this is still enough to build a connected multi-expiry history: while an instrument is tracked, every poll refines its one row right up until it expires and rolls off the active list, at which point a *different* instrument takes over that slot and starts its own new row — the old row is simply never touched again, freezing at its last computed value, which is exactly the final point the chart needs for that expiry.
  - `TRACKED_EXPIRY_COUNT = 3` — how many active expiries (soonest-first, 0 = nearest) get a persisted zones-extrema row at all. Only `expiry_index` 0 (the nearest expiry) gets an actual drawn box on the TradingView chart (see Zones boxes above — the 2nd, "next expiry" box was removed); `expiry_index` 1 and 2 still get tracked (feeding the High/Resistance-Low/Support and Gamma Band term-structure lines, which render whatever rows exist for the asset regardless of whether a box was ever drawn) but only via `get_box_n()`, not a dedicated box endpoint.
  - `get_box_n(asset, expiry_index)` — generic live computation for any tracked `expiry_index`, computed fresh on every call and persisted via `_persist_extrema()`. `get_box(asset)` (below) is a thin wrapper over `get_box_n(asset, 0)`, kept as its own method since `/api/zones-box/<asset>` is the one that actually renders the yellow (and teal) box. `expiry_index` 1 through 2 are called via `/api/zones-extrema-refresh/<asset>/<expiry_index>` instead — no box is drawn from that endpoint's response.
  - `get_box(asset, hours=None)` — called live by `/api/zones-box/<asset>` on every request; returns `_compute_asset()`'s (nearest-expiry) result including the 4 zero-crossing box fields (**not persisted** — nothing reads box-boundary history, only the latest value is ever drawn, see TradingView chart's Zones boxes; `seller_max_profit`/`buyer_max_loss` are also in this same result but, unlike the box fields, *are* persisted by `_persist_extrema()`), but does call `_persist_extrema()` on that same result first. An explicit `hours` (from `/api/zones-box/<asset>`'s optional `?hours=` query param — see TradingView chart's zones box trade-window radio toggle) instead calls `_compute_asset()` directly and skips `get_box_n()`/`_persist_extrema()` entirely — display-only, never written to the shared per-instrument row.
  - `nearest_expiry(asset)` — standalone cheap lookup (no curve-building), returning the nearest active expiry as a full Deribit-style instrument string (e.g. `"BTC-9JUL26"`, matching the `weekly_expiry`/`monthly_expiry`/`INSTRUMENT`/`MONTHLY_INST` convention every other expiry identifier in this addon uses — not just the bare day-string); shares `_distinct_expirations()` with `_compute_asset()` rather than duplicating the DISTINCT-expiration SQL. Called by `/api/nearest-expiry/<asset>` for the TradingView footer's `Nearest:` text.
- `dankbit.forecast3.snapshot` — one record per `(asset, bucket_start)` (a rolling 4h time bucket, see `BUCKET_HOURS`), supplying the Thales Forecast's Gamma-Band Consensus engine (see TradingView Chart Notes' Thales Forecast candles) with a real historical time series, since Dankbit — unlike Thales's manually-dated CSV rows — has no other place a "previous" observation of these levels is stored. `compute_and_persist(asset)` calls `dankbit.zones.extrema._compute_asset(asset, expiry_index=0)` **once** and reads everything off that single result — `index_price`/`computed_at`, `top`/`low` (= that call's own `high_zone_max`/`low_zone_min`, byte-for-byte the same "each curve's own highest/lowest zero-crossing" value, since `_compute_asset()` now persists that pair itself — see Models, `dankbit.zones.extrema`), `bml`/`smp` (= `buyer_max_loss`/`seller_max_profit`), and all 32 per-leg gamma/delta/theta/vega price+Abs fields (`_PER_LEG_GREEK_FIELDS`, spread via a dict comprehension) — rather than reimplementing `options.zone_summary()`/re-querying `dankbit.trade`/re-calling `options.build_zone_curves()` and `forecast3_lib.per_leg_greeks()` a second time, as this method originally did before `dankbit.zones.extrema` itself started persisting the per-leg fields. This removes a second identical trades-query-plus-curve-build for the same asset/expiry/instant on every call, and means the Thales Forecast's band and per-leg Greeks can never quietly drift from what's actually drawn on screen elsewhere in the app — both now come from the exact same `_compute_asset()` result. The whole row is upserted into the current bucket, continuously refining it until the bucket rolls over — the same pattern `_persist_extrema()` uses per-instrument. `recent_history(asset, limit)` returns the most recent rows newest-first for that slope math — **no cron fallback**: history only accumulates as a side effect of `compute_and_persist()` being called live, and a gap just widens the Gamma-Band Consensus engine's slope window (see `gamma_band_consensus()`'s real-elapsed-hours normalization) rather than breaking anything, so the extra cron this model briefly had wasn't worth keeping.
- `res.config.settings` extension — Chart price ranges, refresh interval, Deribit cache TTL, weekly/monthly expiries for BTC and ETH.

- Key helpers:
  - `find_gamma_peaks(STs, gamma_curve, min_fraction)` — local maxima of the gamma curve
  - `find_gamma_bottoms(STs, gamma_curve, min_fraction)` — local minima of the gamma curve
  - `_gamma_peaks_bottoms_all(asset, hours=None)` — the all-active-expiries (no expiration cutoff), optionally trailing-`hours`-restricted peaks/bottoms computation behind `gamma_levels_all_json`'s "Universal"/"Universal 24h" lines
- `delta.py` / `gamma.py` — Pure Black-Scholes functions. Dollar gamma is `Γ × S²`. No side effects.
- `options.py` — `OptionStrat` class. Accumulates legs, then `plot()` returns a matplotlib `Figure`; `plot_zones(longs_curve, shorts_curve, index_price, title)` renders the Longs/Shorts zones chart from two separately-accumulated `OptionStrat` instances — `title` is accepted (callers pass `"Zones"`) but no longer appears in the rendered title, which is always just `"{name} | {now} UTC"`. Uses Agg backend; do not import `pyplot` here. `build_zone_curves(instrument_name, index_price, trades, from_price, to_price, steps)` is the shared Longs/Shorts curve-building + crossing-based ±$2000 zoom logic (module-level function, not a method) — used by both `chart_png_zones` and `dankbit.zones.extrema._compute_asset`, so the PNG chart, the persisted zones-extrema record, and the live zones-box lookup can never compute different values for the same trades. `find_zero_crossings(STs, curve)` is the shared zero-crossing finder (linear interpolation on each sign change) used by `build_zone_curves`'s own Longs-vs-Shorts crossing and by `_compute_asset`'s per-curve box-boundary crossings. `zone_summary(STs, longs_curve, shorts_curve)` computes `seller_max_profit`/`buyer_max_loss`, the `high_zone`/`low_zone`/`middle_zone` boundaries (`high_zone`/`low_zone` renamed from `top_box`/`bottom_box`), and `high_resistance`/`low_support` — from an already-built pair of curves, used by `chart_png_zones` to populate the zones-PNG page's top-left info overlay (see Zones chart above). Current price is deliberately not a parameter of this function at all: `high_zone`/`low_zone` are each curve's own highest (`high_zone`) or lowest (`low_zone`) zero-crossing — a zone forms from a single curve's crossing alone (returned as a degenerate `(price, price)` pair, rendered by `chart_png_zones` as one number rather than a range) if the other curve never crosses zero; `None` only when neither curve crosses zero at all. `middle_zone` is the min/max of `seller_max_profit`/`buyer_max_loss` instead — always defined (those two are argmax/argmin over the full curve, not zero-crossings), same degenerate-single-price rendering when they coincide. `high_resistance`/`low_support` are simply the highest/lowest Longs-vs-Shorts crossing, same reasoning (see Models, `dankbit.zones.extrema`). A separate, differently-shaped computation in `dankbit.zones.extrema._compute_asset` (needs the 4 raw box fields individually rather than a summarized pair, and doesn't need `seller_max_profit`/`buyer_max_loss` at all since it no longer persists them) duplicates the same box/intersection logic rather than sharing this function, but both apply the identical "ignore index price, use each curve's own extremes" rule. `delta_saturation_price(STs, trades, fraction, stop_at)` finds the interpolated price where `delta.portfolio_delta(STs, trades)` first reaches `fraction` of its own extreme value at the `stop_at` edge (`'min'`/`'max'`) of `STs` — the point where a leg's sigmoid-shaped delta curve stops curving and flattens into a straight line (deep enough ITM to "trade like synthetic stock"). Relative to the curve's own extreme, not an absolute delta value, since `portfolio_delta` sums `sign*amount*per-contract delta` across every matching trade — its scale reflects total traded size (can be in the hundreds), not a single option's `[-1, 1]` range, so a fixed absolute threshold would be crossed almost immediately near the start of the curve rather than at any deep-ITM point. Curves are monotonic per leg (see `delta.py`'s sign convention), so the `stop_at` edge value is genuinely the curve's extreme in this window and its sign is inherited automatically — no separate sign parameter needed. Falls back to the `STs` edge on `stop_at` if the extreme is `0` or the curve never reaches that fraction within the window. Shared by `dankbit.zones.extrema` (`delta_band`) and the `/<instrument>/lp,lc,sp,sc` single-leg routes' green marker line, so the two can never disagree on where this point is. `DELTA_SATURATION_FRACTION = 0.9` is the single source of truth for that fraction — `main.py`'s `ChartController.DELTA_SATURATION_FRACTION` is just `options.DELTA_SATURATION_FRACTION` re-exposed as a class attribute (for `_LEG_ROUTES`), rather than each file keeping its own separate `0.9` constant as `forecast3.py` and `dankbit.zones.extrema` both used to. `per_leg_greeks(STs, trades, r=0.0)` computes, for each of the 4 legs split out of `trades` (`long_call`/`long_put`/`short_call`/`short_put`), where that leg's own portfolio gamma/theta/vega curve peaks or bottoms (long positions: gamma/vega peak via `argmax`, theta troughs via `argmin` since long theta is negative decay; short positions: gamma/vega bottom via `argmin`, theta peaks via `argmax` since short theta is positive decay gain) and the price/value where its portfolio delta first saturates (via `delta_saturation_price`, calls at the high-`S` edge, puts at the low-`S` edge, independent of long/short) — returning `{leg_name: {"trades", "gamma_price", "gamma_value", "delta_price", "delta_value", "theta_price", "theta_value", "vega_price", "vega_value"}}`, raw/unscaled. This is the single source of truth for this computation, previously hand-duplicated three times (`chart_png_zones` inline in `main.py`, `dankbit.zones.extrema._compute_asset`'s `gamma_band`/`delta_band` computation, and `forecast3.per_leg_greeks`'s own version) — all three now call this one function and apply only their own display scaling on top (`chart_png_zones`'s Value lines divide gamma/delta/theta/vega by `1e6`/`10`/`1e4`/`100`; `forecast3.per_leg_greeks` does the same for its `*_abs` fields; `dankbit.zones.extrema` averages the 4 gamma/delta prices into `gamma_band`/`delta_band` with no scaling at all).

- `zones_box_window_hours` — how many trailing hours of trades the zones boxes use when the chart's 00:00-UTC-vs-trailing-hours radio toggle is set to the "hours ago" option instead of the default "00:00 UTC" (default 8)

- `weekly_expiry`, `monthly_expiry` — BTC expiry strings (e.g. `BTC-4JUL26`)
- `eth_weekly_expiry`, `eth_monthly_expiry` — ETH expiry strings (e.g. `ETH-4JUL26`)
- `show_daily_lines`, `show_weekly_lines`, `show_monthly_lines` — TradingView chart line visibility toggles (all default `True`). Each gates *every* line carrying that label: `show_weekly_lines`/`show_monthly_lines` control the Weekly/Monthly delta=0 crossing line; `show_daily_lines` controls both Daily-family lines: Daily (24H), Daily+1 (24H). The "All" delta=0 line set has no toggle — always shown. These 3 fields deliberately **don't** use the plain `config_parameter=` attribute like every other setting in this file: Odoo's generic config_parameter handling for `Boolean` fields treats a stored `False` the same as "delete the parameter" (`ir.config_parameter.set_param()` special-cases Python `False`/`None` as unset), so unchecking one and saving would silently revert to `default=True` on next read instead of persisting `False`. `ResConfigSettings.get_values()`/`set_values()` override this explicitly, storing/reading the literal strings `"True"`/`"False"` instead (a string doesn't trigger the delete-on-False special case). `chart_tv` reads the params the same way (`icp.get_param(..., default="True") == "True"`) and converts to the literal strings `"true"`/`"false"` when passing into the template (QWeb's `t-att-*` silently drops the attribute for a falsy Python value, so a real boolean can't round-trip through `data-show-daily`), and the template reads `cfg.showDaily`/`showWeekly`/`showMonthly` into `SHOW_DAILY_LINES`/`SHOW_WEEKLY_LINES`/`SHOW_MONTHLY_LINES` consts — settings are baked in at page render, so a change takes effect on next page load, not live.
- **Thales Forecast section** (72 fields, `forecast3_*`, all `config_parameter="dankbit.forecast3_*"`, all `groups="base.group_no_one"`) — live-configurable versions of every constant `forecast3.simulate_forecast3()` references directly in its own body (the "top-level tunables"), **plus** the 3 gamma/curve/theta center-weight fields (see below) **plus** 4 rendering-only candle-color fields (`forecast3_up_color`/`forecast3_down_color`/`forecast3_wick_up_color`/`forecast3_wick_down_color`, plain hex-string `Char` fields, default `#a5d6a7`/`#ef9a9a`/`#66bb6a`/`#e57373` — light green/red so the forecast candles stay easily distinguishable from the real candle series; unlike every other field in this section, these don't feed `simulate_forecast3()`'s `cfg` at all — they're read by `_build_tv_chart_context()` and passed to the template's `#cfg` div for the client-side `forecast3Series`, since they only affect how the candles are drawn, not the forecast math itself — see TradingView Chart Notes' Thales Forecast candles): base-impulse/wick factors (`forecast3_pull_factor`, `forecast3_slope_factor`, `forecast3_body_factor`, `forecast3_curve_extreme_body_weight`, `forecast3_wick_factor`, `forecast3_atr_factor`, `forecast3_curve_wick_weight`), Gamma-Band Consensus/Trend Lock (`forecast3_gb_*`), the Option Cluster Structure Engine (`forecast3_cluster_*`), Liquidity (`forecast3_liquidity_*`), Momentum/Gamma Neutral Zone (`forecast3_momentum_*`, `forecast3_near_gamma_*`), the low/high-volume and weekday/weekend regime multipliers (`forecast3_{high_vol,low_vol,weekday,weekend}_{pull,shock,atr,body}_factor`), the 4 session-of-day tables (`forecast3_session_{body,atr,shock,firstmove}_{asia,london,overlap,ny,postny}` — 20 fields, one per session per table), `forecast3_bucket_hours_fallback`, `forecast3_gamma_center_weight`/`forecast3_curve_center_weight`/`forecast3_theta_center_weight` (see the engine-cascade bullet above for why these 3 — normally nested-helper-private — are exposed anyway), and the path's own horizon (`forecast3_hours_ahead`/`forecast3_step_hours`/`forecast3_start_offset_hours`, default 72/4/4 — 18 candles). Constants private to every *other* nested helper function (`vega_regime`, `market_maker_gamma_contest`, `cluster_*`, `smart_synthetic_liquidity`'s internals, `delta_shock_module`, `gamma_shock_module`, etc.) are deliberately **not** exposed — a scoping decision made when this section was added, to avoid the much larger refactor of threading overrides through every nested pure function in `forecast3.py`. Each field carries a field-level `default=` matching the engine's own hardcoded constant (so a freshly-opened settings form shows real values, not `0.0`/blank) — `main.py`'s `ChartController._forecast3_cfg()` still separately hardcodes the same defaults as the fallback passed to `icp.get_param(key, default=X)`, matching the convention every other setting in this list uses, so the two stay in sync only by both being kept up to date by hand, not by sharing one source of truth. `_forecast3_cfg()` builds the override dict (keyed by the constant's own name, e.g. `{"FORECAST_PULL_FACTOR": 0.6, "SESSION_BODY_FACTOR": {...}}`) and the `{hours_ahead, step_hours, start_offset_hours}` kwargs, and `forecast3_json` passes both straight into `simulate_forecast3(..., cfg=cfg, **horizon)`. Inside `simulate_forecast3` (and `derive_levels`, which also takes `cfg` and is called from 5 places within `simulate_forecast3`), each overridable constant is shadowed by a same-named local variable at the top of the function (`FORECAST_PULL_FACTOR = _cfg(cfg, "FORECAST_PULL_FACTOR")`) — Python's scoping rules make this a plain local assignment that doesn't touch the module attribute, so every other caller (tests, `cfg=None`) is unaffected, and every existing usage further down the function body picks up the override for free since it already references that same bare name. `_cfg(cfg, name)` reads `cfg[name]` if the caller supplied one, else `globals()[name]` (the module's own hardcoded constant) — looked up by string key specifically so this helper's own `globals()` (the module's, not the caller's) resolves correctly regardless of what the caller's local scope contains.

**Scheduled crons (data/ir_cron.xml, all `active=False` by default — activate manually post-install):**
- Every minute: `get_last_trades()` — ensures no trades were missed (e.g. during brief downtime)
- Daily: `_delete_expired_trades()` — archives expired trades
- Every 4 hours: `compute_snapshot()` (on `dankbit.zones.extrema`) — fallback for BTC's and ETH's current tracked instrument rows (every `expiry_index` up to `TRACKED_EXPIRY_COUNT`), calling the same `get_box_n()` the live `/api/zones-box`/`/api/zones-extrema-refresh` polling calls (see Models, `_persist_extrema`) — a safety net for when nobody's actually viewing the TradingView chart, not the primary update path.

**Caching:** `trade.py` uses a module-level `_DERIBIT_CACHE` dict (key → `{ts, value}`) for Deribit index price and instrument lookups. TTL is configurable via settings (`deribit_cache_ttl`, default 300s).

**Backend menu:** `Dankbit` root menu (`trade_views.xml`) with two items — `Trades` (`dankbit.trade` list/form) and `Zones Extrema` (`zones_extrema_views.xml`, `dankbit.zones.extrema` list/form/search with BTC/ETH filters). A third item, `Liquidity` (`dankbit.liquidity.snapshot`, for manually-entered CoinGlass levels), was removed once the Thales Forecast's Smart Role-Aware Synthetic Liquidity engine made it redundant — see Thales Forecast candles.

## WebSocket Service (`dankbit_ws_service/`)

Single file: `dankbit_ws_batch.py`. Authenticates with `DERIBIT_KEY`/`DERIBIT_SECRET` env vars using `client_credentials` grant before fetching instruments. Reconnects every 3 seconds on any failure. Rate-limit (`over_limit`) errors trigger a 0.5s sleep and retry.

The DB connection uses `autocommit=True` — no explicit transaction management needed.

## TradingView Chart Notes

- Candles sourced from Deribit perpetual futures (`BTC-PERPETUAL`, `ETH-PERPETUAL`) via `/api/klines/<asset>` proxy (avoids CORS). Uses `get_tradingview_chart_data` endpoint. Deribit returns oldest-first parallel arrays (`ticks`, `open`, `high`, `low`, `close`); proxy zips and reverses to newest-first `{t,o,h,l,c}` format (t in ms). **Deribit has no native 4h resolution** (`resolution=240` errors with "unsupported resolution" — valid values are 1, 3, 5, 10, 15, 30, 60, 120, 180, 360, 720, `1D`); the `4h` interval fetches native 1h candles and aggregates every 4 into one OHLC bar server-side, bucketed by `t // (4*3600*1000)` (timestamp-based, not positional) so buckets stay calendar-aligned to 00:00 UTC regardless of the fetch window's boundaries.
- Candles refresh every 5 seconds; delta=0 lines refresh on `dankbit.refresh_interval`
- Chart timezone is auto-detected from the browser (`Intl.DateTimeFormat().resolvedOptions().timeZone`, falling back to UTC if detection throws), not hardcoded — `tzOffset` is that zone's current UTC offset in seconds (DST-aware, computed against "now"). Lightweight Charts has no native IANA-timezone support and always renders epoch seconds as UTC wall-clock time, so every timestamp fed into the chart (candles, all price-line/series time fields, the zones-box's `now - 10 days` left edge) is pre-shifted by `tzOffset` before being handed to the library, and `localization.timeFormatter` formats with `timeZone: 'UTC'` so the shifted epoch reads as the viewer's real local time.
- Timeframe buttons: 1h / 4h (default) / 1d — visible windows: 1h=30d, 4h=15d, 1d=365d; Deribit resolution: 1h→60, 1d→1D, 4h→aggregated from 1h (see above). The 4h window was deliberately narrowed from an initial 90d/30d — `loadCandles()` always fetches a flat 500 bars regardless of timeframe, and Lightweight Charts renders candle body width as a fixed proportion of `barSpacing`; cramming too many bars into one screen (as 90d did, ~3.2px/bar) made each candle's body-to-gap ratio look like a sparse, "half-empty" chart. 15d keeps `barSpacing` comfortably in the ~17px range. `applyVisibleRange()`'s right edge is `now + FUTURE_MARGIN_DAYS` (5 days, fixed regardless of timeframe) rather than a proportion of the visible window — a proportional margin (the original `days * 0.03`) left only ~10 hours of empty space past the current candle on the 4h view, making the chart look locked to "now" instead of leaving room to see future price-line/box levels laid out ahead of the latest candle.
- Ruler tool: click twice on chart to measure price-to-price percentage
- **"Lines" checkbox** (Delta Chart only — labeled "Lines" there; the identically-`id`'d `#lines-toggle` checkbox is labeled "All" on the Gamma Chart instead, see below; next to Ruler, **unchecked by default**) — hides every horizontal price-line indicator (Weekly/Monthly/All/Daily/Daily+1 delta=0 lines) via a client-side `SHOW_ALL_LINES` flag. Unlike the config-driven `SHOW_DAILY_LINES`/`SHOW_WEEKLY_LINES`/`SHOW_MONTHLY_LINES` toggles (which still fetch even when hidden, just skip drawing), leaving this unchecked stops each affected `refresh*()` function from fetching at all — to actually reduce server load, not just hide the client-side result; check it to turn every one of them on. `refreshMonthlyDeltaZero()` is the one exception, same as with `MINIMAL_CHART`: it always keeps polling regardless, since its `trade_count` feeds the footer. `onToggleLines()` clears every line immediately when switched off (each function's own `clearLines()` call is skipped along with its fetch, so the toggle handler clears them directly) and re-fetches everything immediately when switched back on, rather than waiting for the next poll either way.
- **"Zones" checkbox** (Delta Chart only — this indicator is already suppressed on `/my/<asset>` via `MINIMAL_CHART`, so this control has a `t-if="show_gamma_point == 'false'"` and isn't shown there at all; right after "Lines", **checked by default**) — controls the yellow top/bottom zero-crossing boxes (High Zone/Low Zone) and the teal Seller Max Profit/Buyer Max Loss box (Middle Zone), i.e. everything `refreshZonesBoxes()`/`clearZonesBoxes()` draw — these boxes were always shown before this checkbox was added, so it defaults checked to preserve that behavior. Same server-load-reducing pattern as every other checkbox here, via a client-side `SHOW_ZONES_BOXES` flag: unchecking it stops `refreshZonesBoxes()` from calling `/api/zones-box/<asset>` at all, which also means this browser's own polling stops refreshing the nearest expiry's (`expiry_index` 0) `dankbit.zones.extrema` row as a side effect — the same trade-off the "Zones Extrema" checkbox below already accepts for `expiry_index` 1/2 — while the 4-hourly `compute_snapshot()` cron keeps that row fresh on its own regardless. `onToggleZonesBoxes()` clears the boxes immediately when switched off and re-fetches immediately when switched back on.
- **"Thales Forecast" checkbox** (both Delta and Gamma Chart — the only checkbox on this page not gated by a `show_gamma_point` `t-if`, so it renders identically on `/chart/<asset>` and `/my/<asset>`; next to Ruler, **unchecked by default**) — toggles the Thales Forecast candles (see TradingView Chart Notes above) via a client-side `SHOW_FORECAST3` flag. Same server-load-reducing pattern as every other checkbox here: `refreshForecast3()` skips the `/api/forecast3/<asset>` fetch entirely while unchecked rather than just hiding the result; `onToggleForecast3()` clears the forecast series immediately when switched off and fetches immediately when switched back on. This addon previously also had "Forecast" and "Forecast 2" checkboxes (simulated GBM price paths, one plain and one steered by the Zones Extrema levels) — both removed; only this Thales-engine-derived path remains.
- **"All" / "Nearest Expiry" / "Weekly" / "Monthly" checkboxes** (Gamma Chart only, `/my/<asset>` — next to Ruler, `t-if="show_gamma_point == 'true'"`; see the gamma point description above) — four independently-toggleable scopes for the gamma point lines, not mutually exclusive: checking several at once layers all of their lines on the chart together. "All" is the same `#lines-toggle`/`SHOW_ALL_LINES` checkbox the Delta Chart calls "Lines" (see above), just relabeled and, on this page, only ever gating the gamma point lines (there's nothing else on this minimal page for it to gate). **Nearest Expiry** defaults checked (`GAMMA_SCOPE_NEAREST`); **All**/**Weekly**/**Monthly** default unchecked (`GAMMA_SCOPE_WEEKLY`/`GAMMA_SCOPE_MONTHLY`, and `SHOW_ALL_LINES` respectively) — so the chart shows the Nearest-scope lines on load rather than either an empty chart or the (heavier) all-expiries computation. Same server-load-reducing pattern as every other checkbox on this page: `onToggleGammaScope(scope)` flips that one scope's flag and calls `refreshGammaScopeLines()` for just that scope, leaving the other three untouched.
- **"Zones Extrema" checkbox** (Delta Chart only — this indicator is already suppressed on `/my/<asset>` via `MINIMAL_CHART`, so this control has a `t-if="show_gamma_point == 'false'"` and isn't shown there at all; next to "Zones", **unchecked by default**) — controls only the High/Resistance-Low/Support + Gamma Band lines (green/red/dashed violet), deliberately **not** the yellow/teal zones boxes, which have their own separate "Zones" checkbox instead (see above — `refreshZonesBoxes()` is gated by `SHOW_ZONES_BOXES`, not by this flag). Same server-load-reducing pattern as "Lines" above, via a client-side `SHOW_ZONES_EXTREMA` flag: stops `refreshZonesExtrema()` and `refreshZonesExtremaExtended()` (the `expiry_index` 1/2 background-refresh calls, which have no visual of their own) from calling the server at all while unchecked. This only stops this browser's own contribution to keeping `dankbit.zones.extrema`'s `expiry_index` 1/2 rows fresh — `expiry_index` 0 (the nearest expiry) is governed by the "Zones" checkbox above instead, and the 4-hourly `compute_snapshot()` cron is completely untouched either way, keeping every tracked expiry's row fresh on its own. `clearZonesExtremaLines()` (factored out of `refreshZonesExtrema()`'s existing inline clear block, alongside `clearZonesBoxes()` which stays used only within `refreshZonesBoxes()` itself now) clears the High/Resistance-Low/Support-Gamma Band lines immediately when switched off; `onToggleZonesExtrema()` re-fetches them immediately when switched back on.
- **Zones box trade-window radio toggle** (Delta Chart only, after "Thales Forecast" — two radios, "00:00 UTC" checked by default and "`<N>`h ago", `N` baked in from `dankbit.zones_box_window_hours` at page render via `<t t-esc="zones_box_window_hours"/>` in the label) — switches the trade window the yellow/teal zones boxes are computed from between the default since-00:00-UTC-through-now window and a trailing-`N`-hours one. Backed by a client-side `ZONES_BOX_WINDOW_MODE` (`'midnight'`/`'hours'`); when `'hours'`, `refreshZonesBoxes()` appends `?hours=<N>` to `/api/zones-box/<asset>` (`zones_box_json`/`get_box()`/`_compute_asset()` all gained an optional `hours` param for this — passing it overrides the domain's default `deribit_ts` lower bound with `now - hours` instead of today's UTC midnight), and — same principle as the "Zones Extrema" checkbox and the reverted start/end-datetime feature before it — that path is **never persisted** (`get_box()` calls `_compute_asset()` directly rather than `get_box_n()` when `hours` is given, skipping `_persist_extrema()` entirely): display-only, so switching this never corrupts the shared per-expiry history other viewers/expiries' term-structure lines depend on, and the 4-hourly `compute_snapshot()` cron is completely unaffected either way. `onZonesBoxWindowChange()` re-fetches immediately on change rather than waiting for the next `ZONES_BOX_REFRESH` poll.

Leaving this note here rather than silently deleting the section or guessing at a rewrite; if this feature is still wanted, it needs to be (re)implemented, not just re-documented.

## Odoo Gotchas

- **Manifest licence field:** Odoo only accepts a fixed set of strings. Use `"Other OSI approved licence"` (British spelling) for MIT — `"MIT"` alone will fail validation.
- **`auth="public"` routes writing to the ORM:** every model in this addon except `dankbit.trade` only has `ir.model.access.csv` rules for `base.group_user` (internal users) — no public/portal grant. Reading via raw SQL (`request.env.cr.execute(...)`) bypasses this entirely, but any `self.search()`/`self.create()`/`record.write()` call reached from a public route needs an explicit `.sudo()`, or it 403s for genuinely anonymous requests. This is easy to miss in testing: `odoo shell` runs as superuser (no check at all), and hitting the route from a browser tab that's already logged into the backend authenticates as that admin user instead of the public one — only a cookie-less request (e.g. plain `curl`) exercises the real public-user path. See `dankbit.zones.extrema._persist_extrema()` for the fix.
