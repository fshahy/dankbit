# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Running the Stack

```bash
# Start all services
docker compose up -d

# Rebuild after code changes
docker compose up -d --build

# Rebuild a single service (e.g. after editing dankbit_ws_batch.py)
docker compose up -d --build dankbit_ws

# Tail logs
docker compose logs -f
docker compose logs -f dankbit_ws

# Restart just the Odoo addon (no rebuild needed for Python changes to my_addons)
docker compose restart web
```

After first startup, install the Dankbit addon via Odoo UI: Apps → search "Dankbit" → Install.

**Required `.env` variables:**
```
DANKBIT_POSTGRES_DB=<db_name>   # any name; db1 is the development default
DERIBIT_KEY=<key>
DERIBIT_SECRET=<secret>
```
The `UID` and `GID` env vars are also needed so the `web` container writes files as the host user (dev default: `1000`/`1000`). `docker-compose.yml` sets `user: "${UID}:${GID}"` on the `web` service and `chown`s the Odoo data dir to that UID/GID before starting — the values must match whichever host user owns the bind-mounted directories, so Odoo's writes (filestore, logs) land with an owner the host can actually manage. Check with `ls -ln <bind-mounted dir>` if unsure. Setting it to `0`/`0` (root) is correct only when the production host directory itself is root-owned — it also means Odoo runs as root inside the container, a broader security surface than an unprivileged UID.

> **Production note:** `config/odoo.conf` contains `dbfilter = ^db1$` — update this to match the actual database name.
> **Production note:** production's `.env` uses `UID=0`/`GID=0` (root), unlike the dev default of `1000`/`1000` — see above for when that's the correct choice.

## Architecture

Two Docker services talk to a shared PostgreSQL database:

```
Deribit WS API ──► dankbit_ws (Python asyncio) ──► PostgreSQL
                                                        │
                   Deribit REST API ◄── Odoo cron       │
                                                        ▼
                                               Odoo 18 (web:8069)
                                                        │
                               ┌───────────────────────┤
                               │                       │
                        PNG charts                TradingView
                  /BTC-<EXPIRY>, /i/BTC-<EXPIRY>  /chart/BTC
```

**Trade data enters two ways:**
1. **WebSocket (primary):** `dankbit_ws_service/dankbit_ws_batch.py` — connects to Deribit, authenticates, fetches all BTC+ETH option instruments, subscribes to `trades.<instrument>.raw` channels in chunks of 400, inserts rows directly into PostgreSQL with `ON CONFLICT IGNORE` on `deribit_trade_identifier`.
2. **REST backfill:** `Trade.get_last_trades()` — runs every minute via cron; ensures no trades were missed by paging through the last few days of Deribit REST API history.

**PNG chart rendering:** Per-expiry routes like `/BTC-7MAY26` aggregate all trades for that expiry, compute a Black-Scholes portfolio delta and dollar-gamma (GEX = Γ × S²) curve, and return a PNG via matplotlib's Agg backend (`Figure + FigureCanvas`, never `pyplot` — server-safe). Overlaid lines: gamma peaks and bottoms (dashed black), delta=0 crossings (solid — green for "supply" [delta negative below, positive above], red for "demand" [delta positive below, negative above]).

**Zones chart (`/<instrument>/zones`):** a distinct plot mode — separates that instrument's trades since 00:00 UTC into buy-side and sell-side (trade domain uses `("name", "=ilike", f"{instrument}-%")`, an anchored/left-prefix match — not a bare `ilike` substring match — so a query for one expiry can never pull in another instrument's trades), builds two independent `OptionStrat` payoff curves via `options.build_zone_curves()` (`long_call`/`long_put` for buys, `short_call`/`short_put` for sells — same payoff math as the combined-portfolio routes, just accumulated into two separate objects, plus the crossing-based auto-zoom below — same function `dankbit.zones.extrema` calls, see Models), and plots them together via `OptionStrat.plot_zones()`: green "Longs" curve, red "Shorts" curve, blue vertical line at index price — `plot_zones()` itself draws no text overlay. The `Short Max`/`Long Min`/`Top Box`/`Bottom Box` info (previously drawn inside the PNG via `ax.text()`) is now rendered as page HTML instead, alongside further per-expiry stats: `chart_png_zones` calls `options.zone_summary(STs, longs_curve, shorts_curve)` (module-level function, same extrema/box-boundary definitions as `dankbit.zones.extrema` — see Models) for `Short Max`/`Long Min`/`Top Box`/`Bottom Box`/`Top Intersection`/`Bottom Intersection`. Current price is not a factor in any of these: `top_box`/`bottom_box` are each curve's own highest/lowest zero-crossing (the Shorts curve's and Longs curve's max feed `top_box`, their min feeds `bottom_box`) rather than "the nearest crossing above/below index price" — a box forms as soon as *either* curve has at least one crossing (a lone curve's crossing yields a degenerate same-price pair, rendered as a single price rather than a dashed range); only `None`/`"n/a"` if neither curve ever crosses zero. Then separately computes 4 gamma extrema restricted to the single nearest expiry among that route's own `trades` (mirroring `dankbit.zones.extrema`'s "next expiry only" restriction) via `gamma.portfolio_gamma()`, preceded in `zone_info_lines` by a violet-colored `Gamma` section-header line: `Long Call Gamma Peak`/`Long Put Gamma Peak` (price where the long-call/put gamma curve peaks, `argmax`) and `Short Call Gamma Bottom`/`Short Put Gamma Bottom` (price where the short-call/put gamma curve bottoms out, `argmin` — short positions carry negative gamma, so the relevant extremum is a trough, not a peak) — plus, in their own block right below, the gamma value (not dollar-formatted, unlike the price lines) at each of those same 4 extrema (`{curve}[argmax/argmin index]`, the y-value at the point already located for the price line, not a separate computation) — shown as an absolute value divided by 1,000,000 and rounded (the sign is already conveyed by which side of the section it's on: peaks are long/positive, bottoms are short/negative), since these raw dollar-gamma numbers run into the hundreds of millions and the user copies them by hand into another tool — labeled `Long Call Gamma Peak Value`/`Long Put Gamma Peak Value`/`Short Call Gamma Bottom Value`/`Short Put Gamma Bottom Value`. Also computes each leg's delta-saturation price via `options.delta_saturation_price()` (calls saturate ITM at high S, puts at low S, independent of long/short, `fraction=DELTA_SATURATION_FRACTION`=0.9 — see below), preceded by a green-colored `Delta` header, labeled `Long Call Delta`/`Long Put Delta`/`Short Call Delta`/`Short Put Delta`, plus — in their own block right below — the portfolio-delta value at that same price, labeled `Long Call Delta Value`/`Long Put Delta Value`/`Short Call Delta Value`/`Short Put Delta Value` — shown as an absolute value divided by 10 and rounded to a whole number, same reasoning as the Gamma Peak/Bottom Value lines above (dividing by 1,000,000 there, 10 here, since delta values are already two to three orders of magnitude smaller). That value is interpolated off the underlying `delta.portfolio_delta()` curve at the saturation price rather than assumed to equal `fraction * curve edge` exactly, since `delta_saturation_price()` falls back to the `STs` edge (not a true 90%-threshold crossing) when the curve never reaches that threshold within this window — interpolating keeps the reported value correct in that fallback case too. Also computes, per leg, the price where portfolio theta ($/day — see `theta.py`, a new module mirroring `delta.py`/`gamma.py`'s `bs_*`/`portfolio_*` pattern; `bs_theta()` returns the annualized Black-Scholes theta ÷ 365, the convention traders quote, not the raw per-year value) is most extreme over the same price grid gamma/delta use — sign-flipped from gamma's peak/bottom split: long positions carry negative theta (decay cost), so their extremum is the trough (`argmin`, worst decay); short positions carry positive theta (decay gain), so theirs is the peak (`argmax`, best decay) — against the same next-expiry-only `long_calls`/`long_puts`/`short_calls`/`short_puts` recordsets gamma/delta reuse, with the same `r=0.0` convention as this page's other Greeks. Preceded by an orange-colored `Theta` header, labeled `Long Call Theta`/`Long Put Theta`/`Short Call Theta`/`Short Put Theta` — plus, in their own block right below, the theta value ($/day) at each of those same 4 extrema (`{curve}[argmin/argmax index]`, same pattern as the gamma/delta value blocks), labeled `Long Call Theta Value`/`Long Put Theta Value`/`Short Call Theta Value`/`Short Put Theta Value` — shown as an absolute value divided by 10,000 and rounded, same reasoning as the Gamma/Delta Value lines (dividing by 1,000,000 for gamma, 10 for delta, 10,000 here, since raw $/day theta values sit in between those two scales). Also computes, per leg, the price where portfolio vega (`vega.py`, a module mirroring `gamma.py`'s `bs_*`/`portfolio_*` pattern rather than `theta.py`'s — vega has no call/put distinction in Black-Scholes, so `bs_vega()` takes no `option_type`, same as `bs_gamma()`; returns raw vega / 100, the $-per-1-IV-point convention traders quote, mirroring `bs_theta()`'s /365) is most extreme over the same price grid — same peak/bottom split as gamma, not theta's flipped one: vega is always positive for a long option and always negative for a short one, so long legs peak (`argmax`) and short legs bottom out (`argmin`) — against the same next-expiry-only `long_calls`/`long_puts`/`short_calls`/`short_puts` recordsets and `r=0.0` convention as this page's other Greeks. Preceded by a blue-colored `Vega` header, labeled `Long Call Vega`/`Long Put Vega`/`Short Call Vega`/`Short Put Vega` — plus, in their own block right below, the vega value ($ per 1 IV point) at each of those same 4 extrema, labeled `Long Call Vega Value`/`Long Put Vega Value`/`Short Call Vega Value`/`Short Put Vega Value` — shown as an absolute value divided by 100 and rounded, same reasoning as the other Value lines (raw portfolio-vega values sit in the low thousands, an order of magnitude below theta's ten-thousands and well above delta's tens, so /100 keeps the displayed number in the same easy-to-copy 2-3 digit range). Unlike every block above Theta, Theta and Vega are kept in a *separate* `right_info_lines` list rather than appended to `zone_info_lines` (passed into the shared template under the `theta_info_lines` context var name, kept for backward compatibility even though it now carries both Greeks), and rendered in its own top-right `.zone-info-right` overlay (mirrors `.zone-info`'s styling, just `right: 12px` instead of `left: 12px` for the box position — text inside stays left-aligned, same as `.zone-info`) — so it sits to the right of the PNG instead of stacking underneath Gamma/Delta on the left. All of this is assembled into `zone_info_lines`, where every line is a `{text, color}` dict rather than a plain string (color `None` for the default black styling; only section headers set one), rendered by the template via `t-att-style` on that per-line `<div>` (grouped into 7 blocks — price extrema; boxes; intersections; gamma peak/bottom prices; gamma peak/bottom values; delta-saturation prices; delta-saturation values — separated by blank spacer lines, a `" "` non-breaking space since a truly empty/whitespace-only `<div>` can collapse to zero height) and passed to the shared `dankbit_page` template's `zone_info_lines` context var, rendered as a fixed top-left overlay div (`.zone-info`, `user-select: text` so it's copyable, only shown when that var is present, so every other PNG route using `dankbit_page` is unaffected). A separate bottom-left annotation (added by `chart_png_zones` itself, still drawn inside the image via `ax.text()`, not moved) shows the long/short trade counts since 00:00 UTC. Restores a feature this repo had (and removed) in its early history (`d416bcd` "add zones" → removed in `6c293d6`), reimplemented against the current architecture rather than reusing the old pyplot-based code. **Auto-zoomed x-axis:** the configured `from_price`/`to_price` settings are only used for an initial wide pass to locate where the Longs/Shorts curves cross (linear interpolation on the sign change of `longs.payoffs - shorts.payoffs`, same technique as the delta=0 crossing finders); the curves are then rebuilt over `[leftmost_crossing - margin_below, rightmost_crossing + margin_above]` at the same `steps` resolution, so the rendered chart is tightly framed around the relevant region instead of the full (much wider) configured range. Margin is asset-dependent: BTC uses ±$2000; ETH uses ±$100 (ETH's much smaller price scale made the ±$2000 BTC margin blow out the auto-zoom). Falls back to the original wide range if the curves never cross.

**Single-leg delta/gamma routes (`/<instrument>/lp`, `/lc`, `/sp`, `/sc`):** four routes, one per leg type (long put, long call, short put, short call) — `_chart_png_single_leg(instrument, leg_key)` is the shared implementation, dispatched via `_LEG_ROUTES[leg_key]` (`direction`/`option_type` domain filter, which `OptionStrat` leg method accumulates trades, route label, and delta-saturation `fraction`/`side` — see below). Same domain convention as `chart_png_zones` (anchored `=ilike` instrument match, trades since 00:00 UTC), but restricted to that one direction+option_type combination and built over the **full configured price range** (`from_price`/`to_price`/`steps`), not the crossing-based zoomed window `/zones` uses — matching `chart_png_hours`/`chart_png_all`'s style instead, including their `r=0.05` risk-free rate for `delta.portfolio_delta`/`gamma.portfolio_gamma` (deliberately *not* `r=0.0`/zoomed-grid like `dankbit.zones.extrema`, a choice reverted after being tried the other way). `obj.plot()` renders the same delta+gamma dual-axis chart as those routes; the gamma peak/bottom + delta=0 crossing overlay is factored into `_annotate_gamma_delta_crossings(ax, STs, market_deltas, market_gammas)` (shared by all 4 routes, so they don't each carry their own copy of that block). A solid green `axvline` additionally marks the delta-saturation price from `options.delta_saturation_price()` — see `options.py` below — labeled with its price, drawn after the standard overlay.

**TradingView chart:** `/chart/BTC` and `/chart/ETH` serve a Lightweight Charts v4 page with live Deribit perpetual futures candles (proxied server-side) and price lines refreshed on `dankbit.refresh_interval` — except the yellow/teal zones boxes, which refresh on their own dedicated `dankbit.zones_box_refresh_interval` setting (default 3600s/1h, see Settings fields below):
- Delta=0 lines (3 sets) — each a single `createPriceLine`, `lineWidth: 1`, `lineStyle: Solid`. Color is by crossing type, not by set: **black** ("S/D" in the legend) for "supply" (delta negative below, positive above), **red** ("Tie-out" in the legend) for "demand" (delta positive below, negative above — the important case). Set identity is shown via the line's `title` only. Every `createPriceLine` call across the chart (delta=0 sets, gamma peak/bottom, ruler) sets `axisLabelColor: 'black'` so the axis label box is always black regardless of the line's own color — only the label background is forced black, not the line stroke — **except** Daily (24H)/Daily+1 (24H) (below), whose axis label is `axisLabelColor: 'orange'` instead, a deliberate one-off deviation from the rest of the chart. `addLineSeries` (Net Call/Put Gamma) has no equivalent per-series label-color override in Lightweight Charts v4, so those labels still match their line color (blue/orange):
  - **Weekly** — sourced from `/api/delta-zero/<instrument>`
  - **Monthly** — sourced from `/api/delta-zero/<monthly-instrument>`
  - **All** — all active expiries; sourced from `/api/delta-zero-all/<asset>`
- Daily (24H) / Daily+1 (24H) delta=0 lines — two distinct `createPriceLine` sets, color-by-crossing-type like Weekly/Monthly/All (black for "supply" — delta negative below, positive above; red for "demand" — delta positive below, negative above), but with an **orange** axis label (`axisLabelColor: 'orange'`, unlike every other line on this chart — see above) and **no footer-legend entry**, titles `Daily (24H)`/`Daily+1 (24H)`, sourced from `/api/delta-zero-tomorrow/<asset>` and `/api/delta-zero-day-after-tomorrow/<asset>` respectively — both thin wrappers around the shared `_delta_zero_for_calendar_day(asset, days_ahead)` helper (`days_ahead=1`/`2`) so the two can never disagree on how a calendar-day expiry/trade-window is computed. Each targets the specific expiry landing `days_ahead` calendar days out (UTC) — computed directly as `today + N days` formatted to Deribit's no-leading-zero day convention (e.g. `7JUL26`), not "nearest active expiry" (`MIN(expiration) >= NOW()`), since the latter can still resolve to *today's* not-yet-happened expiry. Restricted to trades from the trailing 24h (`deribit_ts >= NOW() - INTERVAL '24 hours'`). Gated by `show_daily_lines` (see Settings fields below).
- Zones Extrema lines (3) — `Top Intersection` (green, `#2e7d32`, solid), `Bottom Intersection` (red, `#c62828`, solid), and `Gamma Band` (light violet, `#c9a0dc`, **dashed** — `LightweightCharts.LineStyle.Dashed`), `addLineSeries` on the **default/main price scale** (no `priceScaleId` override) since these are plain BTC/ETH price levels, not dollar amounts — they render directly alongside the candles. Sourced from `/api/zones-extrema/<asset>`: one point per **instrument** stored in `dankbit.zones.extrema` (see Models — one continuously-refined-then-frozen record per expiry, not a time-series of polls), positioned at that instrument's own expiration time rather than a poll timestamp (there isn't one anymore) — so these are term-structure-style lines connecting each expiry's final value to the next expiry's, not a history of one expiry's value changing over wall-clock time. `Gamma Band` is the average of that expiry's 4 gamma extrema (Long Call/Put Gamma Peak, Short Call/Put Gamma Bottom — see Models, `dankbit.zones.extrema._compute_asset`'s `gamma_band`), the same 4 numbers the `/<instrument>/zones` PNG page's info overlay shows. The API response also carries `delta_band` — the average of that expiry's 4 delta-saturation prices (see `options.delta_saturation_price()`) — but it is deliberately **not drawn** on this chart (no `deltaBandSeries` in the template); it's computed and persisted for later use, not currently rendered. The endpoint derives each point's `t` via a single grouped lookup against `dankbit_trade` (`SUBSTRING(name FROM '^[^-]+-[^-]+')` to recover the instrument prefix, `MIN(expiration)`) rather than parsing the instrument string's day-suffix and assuming a settlement hour — raw SQL, so it still finds archived (past-expiry) instruments' trades despite the ORM's implicit `active=True` filter, since those rows must keep contributing a fixed historical point. Connected point-to-point by the line series. Shown on **every timeframe** (1h/4h/1d) — no longer restricted to 4h; `setTf()`/`init()` call `refreshZonesExtrema()` on every timeframe switch regardless. Top/Bottom Intersection points additionally carry a `+`/`-` `setMarkers()` label (via `top_intersection_positive`/`bottom_intersection_positive` — see Models) showing whether the payoff *at* that intersection sits above or below the zero-payoff line, since the crossing's x-price alone doesn't say — `size: 0` hides the marker's shape glyph so only the text renders, colored to match its own line.
- Zones boxes (2, plus a 3rd Short Max/Long Min box — see below) — filled rectangles (`#fbc02d`, 25% opacity fill, matching top+bottom border lines) marking the top and bottom zero-crossing zones for the **nearest** active expiry only — there is no second (blue) box for a later expiry any more (removed; see Models, `get_box_n`/`TRACKED_EXPIRY_COUNT` for how later expiries are still tracked, just without a box). The 4 box-boundary fields themselves are **not** read from stored history — sourced live from `/api/zones-box/<asset>`, which calls `dankbit.zones.extrema.get_box()` to compute fresh on every request via `_compute_asset()` (see Models: nothing reads box-boundary history, only the latest value is ever drawn). As a side effect, `get_box()` *does* upsert that instrument's zones-extrema record (`index_price`/`top_intersection`/`bottom_intersection`/`gamma_band`/`delta_band`/`short_max_price`/`long_min_price`, not the 4 zero-crossing box-boundary fields) on every call — this endpoint's own polling (at `dankbit.zones_box_refresh_interval`, a dedicated setting separate from `dankbit.refresh_interval`, default 3600s/1h — see Settings fields below) is the primary path feeding the Zones Extrema lines' per-expiry history, with an hourly cron as a fallback for when nobody's viewing the chart (see Models, `_persist_extrema`). The boxes' own visual state is never affected by the cron either way — box-boundary fields are recomputed fresh on whatever request (browser or cron) triggers `_compute_asset()`, but only the browser's request result is ever rendered; the cron's result is only ever persisted, never displayed. Each box is built from two `addBaselineSeries` (`aboveBoxSeries`/`belowBoxSeries` — a flat top-price line filled down to a `baseValue` bottom price) plus a plain `addLineSeries` for the bottom edge (`aboveBoxBottomLine`/`belowBoxBottomLine`), since `addBaselineSeries` only strokes its top edge. Top/bottom of each box are `max`/`min` of that side's two zero-crossing prices (`short_zero_above_price`/`long_zero_above_price` for the top box, `short_zero_below_price`/`long_zero_below_price` for the bottom box) — despite the field names, these are each curve's own highest (`*_above_price`) or lowest (`*_below_price`) zero-crossing, not "the nearest crossing above/below current index price"; names kept for backward compatibility (API/JS consumers), see Models, `_compute_asset`. `setBoxData()` (in `dankbit_templates.xml`) draws the full rectangle when both a top and bottom price exist and differ; if only one curve contributed a crossing, or both coincide (a zero-height box), it instead draws a single line at whichever price is available (reusing the box's own bottom-edge line series) rather than dropping it silently; only clears both when neither curve crosses zero at all (a `0.0` value on **both** sides). Time span is `[now - 10 days, expiration]` — the right edge is that expiry's own real settlement time (Deribit's `expiration_timestamp`, e.g. 08:00 UTC / 10:00 CEST — read off `dankbit_trade.expiration` via `_compute_asset()`, not assumed/hardcoded), so the box now visibly runs all the way out to when the trades it's summarizing actually stop mattering, rather than stopping at "now". The left edge is a flat 10 days back from the browser's current wall-clock time (`Date.now()`, tz-shifted like every other timestamp on this chart), recomputed on every poll rather than fixed to a stored tick — previously anchored to the live lookup's own `computed_at` timestamp and snapped to 18:00 on that calendar day (`snapTo18()`, since removed along with this change). Shown on **every timeframe** (1h/4h/1d) — no longer restricted to 4h, same as the Zones Extrema lines.
- Short Max / Long Min box (3rd box, teal) — same mechanism (`addBaselineSeries` + bottom-edge `addLineSeries`, `setBoxData()`'s single-line fallback when top/bottom coincide) and the exact same `fromTime`/`toTime` as the yellow box above, since both come from the same `/api/zones-box/<asset>` response for the nearest expiry — `smlmBoxSeries`/`smlmBoxBottomLine`. Top/bottom are `max`/`min` of `short_max_price` (price where the Shorts payoff curve peaks) and `long_min_price` (price where the Longs curve bottoms out) — restored model fields, see Models, `dankbit.zones.extrema`. No separate endpoint: `short_max_price`/`long_min_price` were already computed by `_compute_asset()` as a side effect of the same `get_box()` call the yellow box uses, just added to `/api/zones-box/<asset>`'s JSON payload.
- Forecast candles — **shown on both Delta and Gamma Chart, restricted to the 4h timeframe only** (`updateForecastVisibility()`, called from `setTf()` and at page init, shows/hides `#forecast-toggle-wrap` — starts hidden inline — based on `INTERVAL === '4h'`; `refreshForecast()` itself also gates on that condition so a stray poll can't redraw it off-timeframe); the "Forecast" checkbox next to Ruler is otherwise unconditionally rendered outside the `show_gamma_point` t-if split (so it's the one indicator shared by both charts), unchecked by default. One simulated GBM price path extending 2 days past the last real candle at 4h resolution (12 candles — matches real 4h-spaced candles so the two series line up; see `forecast.simulate_path`'s `step_hours=4` default), sourced from `/api/forecast/<asset>` (`forecast_json` in `main.py`) — not a directional call, just one randomly sampled realization consistent with the market's own implied volatility (`sigma` = the trailing-24h amount-weighted average IV across the asset's trades, `SUM(iv*amount)/NULLIF(SUM(amount),0)`, same aggregation `gamma_levels_json` uses, independent of any specific expiry). **Steered toward the nearest trailing-24h, all-expiries gamma peak/bottom** (added after this feature's initial no-drift version — `forecast_json` calls the shared `_gamma_peaks_bottoms_all(asset, hours=24)` helper, the same computation `gamma_levels_all_json`'s "Universal 24h" line uses, and picks the single nearest level to `index_price` among peaks above it and bottoms below it — i.e. whichever resistance-above or support-below level is closest, not an average of all of them): reuses `forecast.simulate_path_with_levels` (see Forecast 2 below) with that level as the reversion center and **no** barriers (`top_intersection`/`bottom_intersection` both `None`), falling back to the plain unguided `simulate_path` (bare `-0.5*sigma^2*dt` drift, same `r=0.0` convention this addon's other Greeks use) when no such level exists, e.g. no trailing-24h trades to build a gamma curve from. The chosen level (or `null`) is echoed back as `gamma_level` in the JSON payload. Each 4h candle is built from 48 finer intra-candle GBM sub-steps server-side (~5-minute resolution; `open`/`close` = that candle's first/last sub-step, `high`/`low` = the sub-steps' own max/min), so bars have a genuine intra-candle range rather than a flat body — two earlier versions got this wrong: a line-series band read as a sparse dot cluster (the forecast's points were denser than real candles, compressing at typical zoom), and a doji-candle version (`open === close`) drew no visible body at all, just a wick. Rendered as a second `addCandlestickSeries` (`forecastSeries`), pastel green/red (`#a5d6a7`/`#ef9a9a`, wicks `#66bb6a`/`#e57373`) — same up/down convention as the real candles, just muted so the two series stay visually distinguishable. `seed` (in `forecast_json`) is derived from the current UTC hour, not true randomness, so the path stays stable between polls within the same hour rather than jittering on every `REFRESH` tick, only moving on to a new realization once new IV data rolls in. The server's own `t` field (now-relative) is **not** used for placement on the chart — `refreshForecast()` instead recomputes each candle's display time client-side, anchored to the same grid `futureAxisSeries` uses for its own future ticks (`realCandles[last].time`, stepped by `intervalSec`); an earlier version trusted the server's `t` directly, which almost never landed on that shared time-axis grid, so Lightweight Charts ended up alternating logical-index slots between the two series in that stretch — rendering as every other forecast candle missing. The anchor itself walks forward from `realCandles[last].time` a whole number of `intervalSec` steps at a time (staying on the grid) until it's strictly past "now" (`Date.now()`, tz-shifted), rather than assuming `realCandles[last]` is always the current still-forming candle one step behind "now" — guaranteeing the first forecast candle is always the next 4h candle after now regardless of which candle Deribit/`updateLatestCandle()` actually last returned. `points` (and the drawn candles) are empty whenever there's no index price or no trailing-24h trades to estimate a volatility from. `refreshForecast()` skips the fetch entirely while its checkbox is off, the timeframe isn't 4h, or no real candles are loaded yet, same server-load-reducing pattern as every other optional indicator here.
- Forecast 2 candles — a second simulated path, drawn alongside the plain Forecast candles (own "Forecast 2" checkbox next to "Forecast", same 4h-only visibility gating via `updateForecastVisibility()`, unchecked by default), that — unlike the plain Forecast, which ignores the Zones Extrema lines entirely — is deliberately steered by them: pulled toward `Gamma Band` (mean-reversion center, in log-price space) and softly reflected off `Top Intersection`/`Bottom Intersection` whenever a sub-step lands beyond them, rather than a pure unguided random walk. `forecast.simulate_path_with_levels()` is the same GBM engine as `simulate_path()`, with the plain variance-drag-only drift replaced by `reversion_speed * (log(gamma_band) - log(S)) * dt - 0.5*sigma^2*dt` (annualized `reversion_speed=300`, i.e. `MEAN_REVERSION_SPEED` — an OU half-life of `ln(2)/300` years, ~20 hours, so a deviation from `gamma_band` is mostly reverted by the end of the 2-day horizon without swamping the short-term noise term) and a post-step reflection — `S = level - (S - level) * barrier_softness`/`level + (level - S) * barrier_softness` (`barrier_softness=0.5`, i.e. `BARRIER_SOFTNESS`) — whenever `S` lands past `top_intersection`/`bottom_intersection`, a partial bounce-back rather than a hard clamp. Sourced from `/api/forecast2/<asset>` (`forecast2_json` in `main.py`), which shares `forecast_json`'s index price, trailing-24h IV, and hour-seed (so the two paths use identical random draws and differ only by the added mean-reversion/barrier terms — directly comparable side by side) and pulls the levels themselves from `dankbit.zones.extrema.get_levels(asset)`: the *persisted* `top_intersection`/`bottom_intersection`/`gamma_band` for the asset's nearest tracked instrument — the same numbers the Zones Extrema lines currently draw — rather than recomputing the curves fresh on every forecast poll (`get_box()` already keeps that row fresh at its own cadence). `top_intersection`/`bottom_intersection` of `0.0` (this model's "no crossing" convention, see Models) are passed through as `None`, skipping that side's barrier rather than reflecting against a bogus zero price. Rendered as a third `addCandlestickSeries` (`forecast2Series`), blue/purple (`#90caf9`/`#ce93d8`, wicks `#42a5f5`/`#ab47bc`) so it stays visually distinct from both the real candles and the plain (pastel green/red) Forecast series. `points` is empty under the same conditions as the plain Forecast, plus whenever there's no persisted `gamma_band` yet for the asset's nearest instrument (e.g. right after a fresh install, before any zones-box poll has run). `refreshForecast2()`/`onToggleForecast2()`/`clearForecast2Lines()` mirror the plain Forecast's own functions exactly, including the same client-side time-anchoring logic (display time is recomputed off `realCandles[last].time`, not the server's `t` — see the plain Forecast entry above for why).
- Forecast 3 candles ("Thales Forecast" checkbox — renamed from "Forecast 3" since the underlying feature/variable names (`forecast3.py`, `SHOW_FORECAST3`, `/api/forecast3/<asset>`, etc.) are kept as-is; same 4h-only gating/unchecked-default as the other two) — a full port of Thales's "Thales Bands" Pine indicator's forecast-candle engine (`controllers/forecast3.py`) onto Dankbit's own live-computed per-leg Greeks, in place of Thales's manually-typed-in daily CSV rows (a companion HTML data-entry tool, "Thales Offline Data Builder", is what a human uses to feed that Pine script by hand — Dankbit needs no such tool since it already computes the equivalent numbers live). **Fully deterministic** — unlike Forecast/Forecast 2, there is no GBM/random component anywhere in this engine (matching the source script, which has none either); every candle is a direct function of the current Greeks, real recent price action, and time-of-day/day-of-week, not a sampled realization. Sourced from `/api/forecast3/<asset>` (`forecast3_json` in `main.py`), rendered as a 4th `addCandlestickSeries` (`forecast3Series`), amber/brown (`#ffcc80`/`#bcaaa4`, wicks `#ff9800`/`#8d6e63`) to stay visually distinct from the real candles and the other two forecasts. `refreshForecast3()`/`onToggleForecast3()`/`clearForecast3Lines()` mirror the other two forecasts' functions exactly, same client-side time-anchoring off `realCandles[last].time`.
  - **Data mapping.** Thales's manually-entered Buyer/Seller Call/Put Gamma/Delta/Theta/Vega quadrant + BML/SMP (curve extremes) + Resistance/Support fields map onto values Dankbit's Greeks already compute for the nearest active expiry's since-00:00-UTC trades: per-leg gamma peak/bottom prices (`gamma.portfolio_gamma()`, same as the `/<instrument>/zones` page), per-leg delta-saturation prices (`options.delta_saturation_price()`), per-leg theta/vega extrema prices (`theta.py`/`vega.py`, same argmin/argmax split those already use), `long_min_price`/`short_max_price` (BML/SMP), and the zones box's own top/bottom edges (`options.zone_summary()`'s `top_box`/`bottom_box`, the same numbers the yellow zones box draws) standing in for Thales's manually-drawn Resistance/Support reference band. Each quadrant/leg value's "Abs" strength counterpart (used throughout the engine as a confidence/weighting signal) is the same dollar-gamma/delta/theta/vega *Value* number (abs, divided by 1e6/10/1e4/100 respectively) the `/<instrument>/zones` page's info overlay already shows. Thales's CoinGlass liquidity-map fields (lower/upper liquidity price+volume) have **no automated Dankbit equivalent** — there's no CoinGlass API integration — so they're entered manually instead, via the new `dankbit.liquidity.snapshot` model (see Models, and the `Liquidity` backend menu item), the same way Thales's own companion HTML tool feeds its Pine script by hand, just twice a day per the user's own 12h liquidity-window cadence rather than Thales's 08:00/16:00 UTC convention. `forecast3.liquidity_map_engine()` **is** ported (unlike the rest of Thales's script, nothing here was skipped): whichever side (lower/upper) has the stronger, closer liquidity pulls price toward it once dominant enough over the other side (`LIQUIDITY_DOMINANCE_THRESHOLD`), and a real candle sweeping through a level and closing back away from it (`upper_swept_rejected`/`lower_swept_rejected`) fires an extra rejection impulse and a body-confidence floor/wick compression, same as every other engine above. Entirely inert (zero impulse, no confidence floor) whenever nothing has been entered yet for that asset — same na-safe degradation the source script has for a blank liquidity column.
  - **No future-known levels, unlike Thales.** Thales's own rows are manually dated, entered in advance up to a known expiry, so its forecast loop interpolates between a "current" and a "future" row that are both already-known data points. Dankbit has no foreknowledge of tomorrow's Greek levels, so there is only ever one "current" set of levels (top/low/gamma/curve/theta averages) — held constant across every forecast candle, the same way Forecast/Forecast 2's `sigma_annual`/`gamma_band` inputs are constant across their own path. The only thing that varies over real elapsed time is the history feeding the Gamma-Band Consensus engine's slope math (see below).
  - **`dankbit.forecast3.snapshot`** (new model, see Models) is what supplies that real time-series: one row per `(asset, 4h time bucket)`, continuously refined while its bucket is current then frozen once time rolls past it — the same "continuously-refined-then-frozen" pattern `dankbit.zones.extrema` uses per-instrument, just keyed by a rolling time bucket instead of an expiry. `compute_and_persist()` is called live on every `/api/forecast3/<asset>` request (refining the current bucket) — **no cron fallback** (unlike Zones Extrema's dual live-poll-plus-hourly-cron pattern): history only accumulates while the "Thales Forecast" checkbox is open somewhere, and a gap just widens the Gamma-Band Consensus engine's slope window rather than breaking anything, so the extra cron wasn't worth keeping.
  - **Engine cascade** (`forecast3.py`, Thales's own default constants ported as fixed module constants rather than ~150 exposed settings fields): a weighted gamma/curve/theta center (`GAMMA_CENTER_WEIGHT`/`CURVE_CENTER_WEIGHT`/`THETA_CENTER_WEIGHT` = 0.55/0.30/0.15, reweighted under Gamma-Curve Divergence when the gamma and curve averages disagree by more than `DIVERGENCE_THRESHOLD` band-widths); session-aware (Asia/London/Overlap/NY/Post-NY) and weekend/regime multiplier tables (UTC-only — Thales's own `sessionTimezone` also defaults to UTC); the Gamma-Band Consensus backbone (`gamma_band_consensus()`, a 3-point top/low/gamma slope comparison against the last 2 persisted snapshot rows, real-hours-normalized) plus its Reclaim Bias; a Delta Shock Module and Gamma Shock Module (band breaks confirmed by price also sitting on the correct side of the gamma reference for `GAMMA_CONFIRM_BARS` real closes); a Vega Regime Engine (buyer/seller vega dominance expanding or compressing the wick); a Market-Maker Gamma Contest Engine (buyer-call breakout force vs seller-call pin force, and the put-side mirror, each scaled by price's proximity to that leg's own gamma level); an Option Cluster Structure Engine (how tightly top/low/gamma/BML/SMP agree with each other, and whether they're all moving the same direction since the previous snapshot); a Gamma Neutral/Hysteresis Zone (chop damping near the gamma reference); and a Momentum/Liquidity-Sweep Override, all computed against real recent 4h candles fetched server-side from Deribit (`ChartController._fetch_candles()`, factored out of `/api/klines/<asset>` so both share one aggregation implementation) for a genuine ATR14 and body/wick momentum detection — Thales's own bar-by-bar `barstate.isconfirmed` live-candle gating collapses to "the last N already-closed real candles" here, since Dankbit generates the whole path in one batch rather than re-evaluating live each bar. The Quadrant Behavior Engine and full wick-to-body acceptance-score cascade from the source script are intentionally simplified (the former is off by Thales's own default anyway; the latter's ~150-line asymmetry/acceptance scoring is replaced by the lighter cluster-based confidence floor above) rather than transliterated line-for-line.
- No footer legend — removed entirely (`#dz-legend` div and its CSS); all lines/series remain, just without a color-key legend.
- Footer status shows: `<weekly-expiry>  ·  <monthly-expiry>  ·  N trades  ·  Nearest: <expiry>`; trade count = all active trades up to and including monthly expiry (from monthly endpoint). The `Nearest:` segment is the same nearest-active-expiry the yellow zones boxes use (`dankbit.zones.extrema.nearest_expiry()`, a cheap standalone `SELECT DISTINCT expiration ... LIMIT 1` lookup, no curve-building) — shown regardless of timeframe (`refreshNearestExpiry()`, a lightweight standalone lookup unlike the boxes' own full computation — though both are now shown on every timeframe), via a shared `updateStatusLine()` that both it and `refreshMonthlyDeltaZero()` call so neither overwrites the other's half of the status text.

**`/my/<asset>` (BTC/ETH):** a deliberately minimal page — candles plus exactly one indicator, the **gamma point** — sharing the same template (`dankbit_tv_chart_until`) and context-building (`_build_tv_chart_context()`) as `/chart/<asset>` rather than a duplicate template, but with every other indicator that page draws (Weekly/Monthly/All/Daily delta=0 lines, Zones Extrema lines, zones boxes) suppressed. `/chart/<asset>` itself is completely unaffected by any of this — `_build_tv_chart_context()` always sets `show_gamma_point: "false"` there, only `/my/<asset>`'s route overrides it to `"true"` (passed through as the literal string, not a Python bool, for the same QWeb `t-att-*`-drops-falsy-values reason `show_daily_lines` etc. do). The same flag also picks the `<title>`: `Delta Chart` when `"false"` (i.e. `/chart/<asset>`), `Gamma Chart` when `"true"` (i.e. `/my/<asset>`) — reusing `show_gamma_point` rather than adding a separate page-title context var. In the template, `const MINIMAL_CHART = SHOW_GAMMA_POINT;` (the two concepts coincide today — `/my/<asset>` is both the only minimal page and the only one with the gamma point — reused rather than adding a second backend flag) gates every other `refresh*` function: most (`refreshDeltaZero`, `refreshAllDeltaZero`, `refreshTomorrowDeltaZero`, `refreshDayAfterTomorrowDeltaZero`, `refreshZonesExtrema`, `refreshZonesBoxes`, `refreshZonesExtremaExtended`) `return` before even fetching when `MINIMAL_CHART`, so no wasted network/server calls either. `refreshMonthlyDeltaZero` is the one exception — it still fetches `/api/delta-zero/<monthly-instrument>` and updates `lastTradeCount` on `MINIMAL_CHART`, just skips drawing its price lines, since that fetch's `trade_count` feeds the footer's "N trades", and the footer (`<weekly-expiry> · <monthly-expiry> · N trades · Nearest: <expiry>`) is unaffected by any of this — same standing rule as always, it's never removed.

The indicator this page draws, the **gamma point**, is drawn as 4 independently-toggleable *scope* checkboxes next to the Ruler — **Nearest** / **Weekly** / **Monthly** / **Universal** (none mutually exclusive: checking several layers all of their lines on the chart together), all unchecked by default. Each scope draws **two** line sets, toggled together by the one checkbox — a black "every trade through expiry" set and a paired 24h-restricted set (orange for Nearest 24h, violet for Weekly 24h/Monthly 24h/Universal 24h). Each line set is **every local gamma peak/bottom** found in that scope's combined portfolio-gamma curve (`find_gamma_peaks`/`find_gamma_bottoms`, 15%-of-max threshold — can be 0, 1, or several lines, not a single averaged point) via the shared `drawGammaLines(data, lineStore, lineWidth, prefix, color, axisLabelColor)` helper, `lineWidth: 1`, solid. Titled `"<prefix> ∧ Long Call"`/`"<prefix> ∧ Long Put"` for peaks and `"<prefix> ∨ Short Put"`/`"<prefix> ∨ Short Call"` for bottoms — which of the two labels applies is decided by `delta_positive` (whether the combined portfolio delta is positive or negative at that exact gamma extremum price), not a separate per-leg computation. Scope determines which trades feed that combined curve: **Nearest** is the single nearest active expiry (`/api/nearest-expiry/<asset>` looked up first, then fed into `gamma_levels_json` as the cutoff instrument — since nothing sits between now and the nearest expiry, that cutoff naturally resolves to just the one expiry); **Weekly**/**Monthly** are every expiry from now through the configured `dankbit.weekly_expiry`/`monthly_expiry` (or `eth_*` for ETH) — same `expiration <= cutoff` convention `/BTC/weekly` and `/BTC/monthly` already use via `chart_png_until`; **Universal** is every currently active expiry with **no** cutoff at all. Sourced from `/api/gamma-levels/<instrument>` (`gamma_levels_json` — the `instrument` string's day-suffix is the cutoff, not a single-instrument filter; optional `?hours=` restricts to the trailing N hours of trades on top of that, feeding the 24h variants) for Nearest/Weekly/Monthly, and from `/api/gamma-levels-all/<asset>` (`gamma_levels_all_json` — same computation and same optional `?hours=` override, just with no expiration cutoff at all) for Universal/Universal 24h. `refreshGammaPointLines()` (a no-op when `!SHOW_GAMMA_POINT`) calls one `refreshGammaPoint<Scope>[24h]()` function per line set from `init()` and its own `REFRESH`-interval poll; each skips its own fetch entirely (not just hiding the result) while its checkbox is off, same server-load-reducing pattern as "Lines"/"Zones Extrema" on the Delta Chart. This replaced an earlier single-averaged-point-per-scope-per-window design (`/api/gamma-point/<asset>/<hours>`, `GAMMA_POINT_HOURS = [4,8,12,24]`, dominant-leg titling, an "All" scope reusing the `#lines-toggle` checkbox, and a `🕸` HTML overlay badge) — none of that code exists anymore; do not describe or rely on it.

## Odoo Addon (`my_addons/dankbit/`)

The addon depends only on `website`. Key components:

**Models:**
- `dankbit.trade` — Core model. Fields map directly to Deribit trade fields. `strike` and `option_type` are computed from `name` (instrument name like `BTC-29NOV24-98000-P`). `days_to_expiry` is UTC-safe. `get_hours_to_expiry()` returns continuous time used in Black-Scholes.
- `dankbit.zones.extrema` — **One record per instrument** (e.g. `"BTC-10JUL26"`, unique-constrained on `instrument`), not per snapshot-in-time — deliberately has no timestamp field of its own (a row's position on the TradingView chart is that instrument's own expiration time, looked up from `dankbit_trade.expiration` when the API serves it — see below and TradingView Chart Notes' Zones Extrema lines). Fields: `asset`, `instrument`, `index_price`, `top_intersection`/`bottom_intersection` (the highest/lowest price where the Longs-vs-Shorts payoff curves cross each other — not where either crosses zero, and **not** relative to `index_price` — labels kept for backward compatibility, but with only one crossing that single point can land on either side of price by a trivial amount, which used to make the "other" field silently read `0.0` even though the plot clearly showed one real intersection; same `longs.payoffs - shorts.payoffs` sign-change `build_zone_curves()` finds internally for its own ±$2000 auto-zoom), `top_intersection_positive`/`bottom_intersection_positive` (Boolean — whether the *payoff value* at that intersection, not its price, sits above or below the zero-payoff line; interpolated via `np.interp` at the crossing price against `longs_obj.payoffs`, which equals `shorts_obj.payoffs` at a true crossing — drives the `+`/`-` marker on the TradingView chart, see TradingView Chart Notes), `gamma_band` (average of 4 gamma extrema — Long Call Gamma Peak, Long Put Gamma Peak, Short Call Gamma Bottom, Short Put Gamma Bottom — the same numbers the `/<instrument>/zones` PNG page's info overlay shows, computed here against this same instrument's `trades`/`STs` via `gamma.portfolio_gamma()`, `argmax` for the two Long peaks and `argmin` for the two Short bottoms since short positions carry negative gamma), `delta_band` (average of 4 delta-saturation prices, one per leg — see `options.delta_saturation_price()` under `options.py` — computed via `delta.portfolio_delta()` against the same `trades`/`STs`, not derived from gamma, since portfolio delta and portfolio gamma are numerically related but conceptually distinct measures here), `short_max_price`/`long_min_price` (price where the Shorts payoff curve peaks / where the Longs curve bottoms out, against this same `longs_obj`/`shorts_obj` — this model's original two fields, added in `8c59981`, repurposed away into `top_intersection`/`bottom_intersection` in `8da5a1a`, and since reintroduced as their own fields alongside those rather than replacing them again; same values `options.zone_summary()`'s `short_max_price`/`long_min_price` show on the `/<instrument>/zones` PNG page's info overlay, not exposed via `/api/zones-extrema/<asset>` — model-only, feeds the `/api/zones-box/<asset>` teal box instead, see TradingView Chart Notes). `_compute_asset(asset, expiry_index=0, hours=None)` is the shared computation core (the `expiry_index`-th soonest active expiry — 0 is nearest, 1 is the one after that, via a raw `SELECT DISTINCT expiration` query so a single expiry's 100k+ trades can't crowd out the next one — trades since 00:00 UTC through now for that expiry by default, or the trailing `hours` hours through now when given — see `get_box`'s `hours` override below, `options.build_zone_curves()`, then `top_intersection`/`bottom_intersection`/`gamma_band`/`delta_band`/`short_max_price`/`long_min_price`, that expiry's own `instrument` string and `expiration` datetime, plus 4 zero-crossing box boundaries — each curve's own highest/lowest zero-crossing, also not relative to `index_price` (same principle as `top_intersection`/`bottom_intersection`; see below) — returns `None` if there's nothing computable (missing index price/expiry at that index/since-midnight trades) rather than writing a row that would silently look like real data (e.g. the configured price-range floor). Persisted as a side effect of `get_box_n()` (below), called by the live zones-box polling (`expiry_index` 0, at `dankbit.zones_box_refresh_interval`) and zones-extrema-refresh polling (`expiry_index` 1-2, at `dankbit.refresh_interval`) that already happen on every TradingView chart page, and by an hourly fallback cron, `compute_snapshot()`, since live-polling-only would leave a gap for any instrument nobody happens to be viewing (a real missing point in the per-expiry history, not just a staler one). The cron calls the exact same `get_box_n()` the live endpoints call (for both BTC and ETH, every tracked `expiry_index` — see `TRACKED_EXPIRY_COUNT` below), so it can never compute or persist anything a live page view wouldn't have, and never touches the TradingView horizontal price lines (delta=0, gamma peak/bottom) — only this model.
  - `_persist_extrema(data)` — upserts (search by `instrument`, `write()` if found else `create()`) the one record for `data['instrument']`, storing only the historical-line fields (`index_price`/`top_intersection`/`bottom_intersection`/`top_intersection_positive`/`bottom_intersection_positive`/`gamma_band`/`delta_band`/`short_max_price`/`long_min_price`); the 4 zero-crossing box-boundary fields are never persisted. `self.sudo()` internally: `get_box()`/`get_box_n()` are `auth="public"` routes, so this otherwise runs as the anonymous public user, which has no access rights at all on `dankbit.zones.extrema` (`ir.model.access.csv` only grants `base.group_user`) — without the explicit sudo, every real (cookie-less) request 403s even though it works fine from an already-authenticated browser session or `odoo shell`, which is why this went unnoticed when `_persist_extrema` was first introduced. Called from `get_box_n()` for every tracked `expiry_index`, so an instrument's row starts accumulating as soon as it's tracked at all — typically well before it becomes nearest. Either way, this is still enough to build a connected multi-expiry history: while an instrument is tracked, every poll refines its one row right up until it expires and rolls off the active list, at which point a *different* instrument takes over that slot and starts its own new row — the old row is simply never touched again, freezing at its last computed value, which is exactly the final point the chart needs for that expiry.
  - `TRACKED_EXPIRY_COUNT = 3` — how many active expiries (soonest-first, 0 = nearest) get a persisted zones-extrema row at all. Only `expiry_index` 0 (the nearest expiry) gets an actual drawn box on the TradingView chart (see Zones boxes above — the 2nd, "next expiry" box was removed); `expiry_index` 1 and 2 still get tracked (feeding the Top/Bottom Intersection and Gamma Band term-structure lines, which render whatever rows exist for the asset regardless of whether a box was ever drawn) but only via `get_box_n()`, not a dedicated box endpoint.
  - `get_box_n(asset, expiry_index)` — generic live computation for any tracked `expiry_index`, computed fresh on every call and persisted via `_persist_extrema()`. `get_box(asset)` (below) is a thin wrapper over `get_box_n(asset, 0)`, kept as its own method since `/api/zones-box/<asset>` is the one that actually renders the yellow (and teal) box. `expiry_index` 1 through 2 are called via `/api/zones-extrema-refresh/<asset>/<expiry_index>` instead — no box is drawn from that endpoint's response.
  - `get_box(asset, hours=None)` — called live by `/api/zones-box/<asset>` on every request; returns `_compute_asset()`'s (nearest-expiry) result including the 4 zero-crossing box fields (**not persisted** — nothing reads box-boundary history, only the latest value is ever drawn, see TradingView chart's Zones boxes; `short_max_price`/`long_min_price` are also in this same result but, unlike the box fields, *are* persisted by `_persist_extrema()`), but does call `_persist_extrema()` on that same result first. An explicit `hours` (from `/api/zones-box/<asset>`'s optional `?hours=` query param — see TradingView chart's zones box trade-window radio toggle) instead calls `_compute_asset()` directly and skips `get_box_n()`/`_persist_extrema()` entirely — display-only, never written to the shared per-instrument row.
  - `nearest_expiry(asset)` — standalone cheap lookup (no curve-building), returning the nearest active expiry as a full Deribit-style instrument string (e.g. `"BTC-9JUL26"`, matching the `weekly_expiry`/`monthly_expiry`/`INSTRUMENT`/`MONTHLY_INST` convention every other expiry identifier in this addon uses — not just the bare day-string); shares `_distinct_expirations()` with `_compute_asset()` rather than duplicating the DISTINCT-expiration SQL. Called by `/api/nearest-expiry/<asset>` for the TradingView footer's `Nearest:` text.
  - `get_levels(asset)` — `top_intersection`/`bottom_intersection`/`gamma_band` from the persisted row for `asset`'s nearest tracked instrument (via `nearest_expiry()`), read-only: does not compute or persist anything itself. `self.sudo()` since this is reached from an `auth="public"` route with no public grant on this model. Returns `None` if there's no active expiry or nothing persisted for it yet. Called by `forecast2_json` to steer the "Forecast 2" path (see TradingView Chart Notes' Forecast 2 candles) — deliberately reuses whatever `get_box()`/`get_box_n()` last persisted rather than recomputing the curves fresh on every forecast poll, so Forecast 2 always matches the Zones Extrema lines currently on screen.
- `dankbit.forecast3.snapshot` — one record per `(asset, bucket_start)` (a rolling 4h time bucket, see `BUCKET_HOURS`), supplying Forecast 3's Gamma-Band Consensus engine (see TradingView Chart Notes' Forecast 3 candles) with a real historical time series, since Dankbit — unlike Thales's manually-dated CSV rows — has no other place a "previous" observation of these levels is stored. `compute_and_persist(asset)` computes the nearest-active-expiry, since-00:00-UTC per-leg gamma/delta/theta/vega extrema (`forecast3.per_leg_greeks()`) plus the zones-box top/bottom band edges (`options.zone_summary()`'s `top_box`/`bottom_box`, reimplemented standalone here rather than shared with `dankbit.zones.extrema`/`get_box()` — same reasoning as Trial Points: a new/experimental feature shouldn't share a live code path with the existing persisted history other viewers' charts depend on), carries over whatever `dankbit.liquidity.snapshot` row (see below) was freshest as of that moment (`lower_liq_price`/`lower_liq_m`/`upper_liq_price`/`upper_liq_m`, `0.0` on all four if nothing had been entered yet — this model's own "absent" convention), and upserts the whole row into the current bucket, continuously refining it until the bucket rolls over — the same pattern `_persist_extrema()` uses per-instrument. `recent_history(asset, limit)` returns the most recent rows newest-first for that slope math — **no cron fallback**: history only accumulates as a side effect of `compute_and_persist()` being called live, and a gap just widens the Gamma-Band Consensus engine's slope window (see `gamma_band_consensus()`'s real-elapsed-hours normalization) rather than breaking anything, so the extra cron this model briefly had wasn't worth keeping.
- `dankbit.liquidity.snapshot` — manually-entered CoinGlass liquidity levels (see TradingView Chart Notes' Forecast 3 candles and its Liquidity Map Engine discussion), since there's no CoinGlass API integration. Fields: `asset` (BTC/ETH selection), `as_of` (Datetime — the moment this reading is valid for, defaulting to now but editable, so a late or backdated entry still lands in the right place in history), `lower_liq_price`/`lower_liq_m`, `upper_liq_price`/`upper_liq_m`, `note`. `get_latest(asset, as_of=None)` returns the freshest row at-or-before `as_of` (default: now) — cadence-agnostic on purpose, so it works whether entries land every 12h (the user's own cadence), 8h (Thales's own convention), or irregularly; returns an empty recordset (not an error) when nothing has been entered yet. Entered via the `Liquidity` backend menu item (see Backend menu) — internal users only (`base.group_user`), no public route ever writes to this model.
- `res.config.settings` extension — Chart price ranges, refresh interval, Deribit cache TTL, weekly/monthly expiries for BTC and ETH.

Accumulates legs, then `plot()` returns a matplotlib `Figure`; `plot_zones(longs_curve, shorts_curve, index_price, title)` renders the Longs/Shorts zones chart from two separately-accumulated `OptionStrat` instances — `title` is accepted (callers pass `"Zones"`) but no longer appears in the rendered title, which is always just `"{name} | {now} UTC"`. Current price is deliberately not a parameter of this function at all: `top_box`/`bottom_box` are each curve's own highest (`top_box`) or lowest (`bottom_box`) zero-crossing — a box forms from a single curve's crossing alone (returned as a degenerate `(price, price)` pair, rendered by `chart_png_zones` as one number rather than a range) if the other curve never crosses zero; `None` only when neither curve crosses zero at all — and `top_intersection`/`bottom_intersection` are simply the highest/lowest Longs-vs-Shorts crossing, same reasoning (see Models, `dankbit.zones.extrema`). A separate, differently-shaped computation in `dankbit.zones.extrema._compute_asset` (needs the 4 raw box fields individually rather than a summarized pair, and doesn't need `short_max_price`/`long_min_price` at all since it no longer persists them) duplicates the same box/intersection logic rather than sharing this function, but both apply the identical "ignore index price, use each curve's own extremes" rule. `delta_saturation_price(STs, trades, fraction, stop_at)` finds the interpolated price where `delta.portfolio_delta(STs, trades)` first reaches `fraction` of its own extreme value at the `stop_at` edge (`'min'`/`'max'`) of `STs` — the point where a leg's sigmoid-shaped delta curve stops curving and flattens into a straight line (deep enough ITM to "trade like synthetic stock"). Relative to the curve's own extreme, not an absolute delta value, since `portfolio_delta` sums `sign*amount*per-contract delta` across every matching trade — its scale reflects total traded size (can be in the hundreds), not a single option's `[-1, 1]` range, so a fixed absolute threshold would be crossed almost immediately near the start of the curve rather than at any deep-ITM point. Curves are monotonic per leg (see `delta.py`'s sign convention), so the `stop_at` edge value is genuinely the curve's extreme in this window and its sign is inherited automatically — no separate sign parameter needed. Falls back to the `STs` edge on `stop_at` if the extreme is `0` or the curve never reaches that fraction within the window. Shared by `dankbit.zones.extrema` (`delta_band`) and the `/<instrument>/lp,lc,sp,sc` single-leg routes' green marker line, so the two can never disagree on where this point is.

## WebSocket Service (`dankbit_ws_service/`)

Single file: `dankbit_ws_batch.py`. Authenticates with `DERIBIT_KEY`/`DERIBIT_SECRET` env vars using `client_credentials` grant before fetching instruments. Reconnects every 3 seconds on any failure. Rate-limit (`over_limit`) errors trigger a 0.5s sleep and retry.

The DB connection uses `autocommit=True` — no explicit transaction management needed.

## TradingView Chart Notes

- Candles sourced from Deribit perpetual futures (`BTC-PERPETUAL`, `ETH-PERPETUAL`) via `/api/klines/<asset>` proxy (avoids CORS). Uses `get_tradingview_chart_data` endpoint. Deribit returns oldest-first parallel arrays (`ticks`, `open`, `high`, `low`, `close`); proxy zips and reverses to newest-first `{t,o,h,l,c}` format (t in ms). **Deribit has no native 4h resolution** (`resolution=240` errors with "unsupported resolution" — valid values are 1, 3, 5, 10, 15, 30, 60, 120, 180, 360, 720, `1D`); the `4h` interval fetches native 1h candles and aggregates every 4 into one OHLC bar server-side, bucketed by `t // (4*3600*1000)` (timestamp-based, not positional) so buckets stay calendar-aligned to 00:00 UTC regardless of the fetch window's boundaries.
- Candles refresh every 5 seconds; delta=0 lines refresh on `dankbit.refresh_interval`
- Chart timezone is auto-detected from the browser (`Intl.DateTimeFormat().resolvedOptions().timeZone`, falling back to UTC if detection throws), not hardcoded — `tzOffset` is that zone's current UTC offset in seconds (DST-aware, computed against "now"). Lightweight Charts has no native IANA-timezone support and always renders epoch seconds as UTC wall-clock time, so every timestamp fed into the chart (candles, all price-line/series time fields, the zones-box's `now - 10 days` left edge) is pre-shifted by `tzOffset` before being handed to the library, and `localization.timeFormatter` formats with `timeZone: 'UTC'` so the shifted epoch reads as the viewer's real local time.
- Timeframe buttons: 1h / 4h (default) / 1d — visible windows: 1h=30d, 4h=15d, 1d=365d; Deribit resolution: 1h→60, 1d→1D, 4h→aggregated from 1h (see above). The 4h window was deliberately narrowed from an initial 90d/30d — `loadCandles()` always fetches a flat 500 bars regardless of timeframe, and Lightweight Charts renders candle body width as a fixed proportion of `barSpacing`; cramming too many bars into one screen (as 90d did, ~3.2px/bar) made each candle's body-to-gap ratio look like a sparse, "half-empty" chart. 15d keeps `barSpacing` comfortably in the ~17px range. `applyVisibleRange()`'s right edge is `now + FUTURE_MARGIN_DAYS` (5 days, fixed regardless of timeframe) rather than a proportion of the visible window — a proportional margin (the original `days * 0.03`) left only ~10 hours of empty space past the current candle on the 4h view, making the chart look locked to "now" instead of leaving room to see future price-line/box levels laid out ahead of the latest candle.
- Ruler tool: click twice on chart to measure price-to-price percentage
- **"Lines" checkbox** (Delta Chart only — labeled "Lines" there; the identically-`id`'d `#lines-toggle` checkbox is labeled "All" on the Gamma Chart instead, see below; next to Ruler, **unchecked by default**) — hides every horizontal price-line indicator (Weekly/Monthly/All/Daily/Daily+1 delta=0 lines) via a client-side `SHOW_ALL_LINES` flag. Unlike the config-driven `SHOW_DAILY_LINES`/`SHOW_WEEKLY_LINES`/`SHOW_MONTHLY_LINES` toggles (which still fetch even when hidden, just skip drawing), leaving this unchecked stops each affected `refresh*()` function from fetching at all — to actually reduce server load, not just hide the client-side result; check it to turn every one of them on. `refreshMonthlyDeltaZero()` is the one exception, same as with `MINIMAL_CHART`: it always keeps polling regardless, since its `trade_count` feeds the footer. `onToggleLines()` clears every line immediately when switched off (each function's own `clearLines()` call is skipped along with its fetch, so the toggle handler clears them directly) and re-fetches everything immediately when switched back on, rather than waiting for the next poll either way.
- **"Forecast" checkbox** (both Delta and Gamma Chart — the only checkbox on this page not gated by a `show_gamma_point` `t-if`, so it renders identically on `/chart/<asset>` and `/my/<asset>`; next to Ruler, **unchecked by default**) — toggles the Forecast candles (see TradingView Chart Notes above) via a client-side `SHOW_FORECAST` flag. - **Zones box trade-window radio toggle** (Delta Chart only, next to "Zones Extrema" — two radios, "00:00 UTC" checked by default and "`<N>`h ago", `N` baked in from `dankbit.zones_box_window_hours` at page render via `<t t-esc="zones_box_window_hours"/>` in the label) — switches the trade window the yellow/teal zones boxes are computed from between the default since-00:00-UTC-through-now window and a trailing-`N`-hours one. Backed by a client-side `ZONES_BOX_WINDOW_MODE` (`'midnight'`/`'hours'`); when `'hours'`, `refreshZonesBoxes()` appends `?hours=<N>` to `/api/zones-box/<asset>` (`zones_box_json`/`get_box()`/`_compute_asset()` all gained an optional `hours` param for this — passing it overrides the domain's default `deribit_ts` lower bound with `now - hours` instead of today's UTC midnight), and — same principle as the "Zones Extrema" checkbox and the reverted start/end-datetime feature before it — that path is **never persisted** (`get_box()` calls `_compute_asset()` directly rather than `get_box_n()` when `hours` is given, skipping `_persist_extrema()` entirely): display-only, so switching this never corrupts the shared per-expiry history other viewers/expiries' term-structure lines depend on, and the hourly `compute_snapshot()` cron is completely unaffected either way. `onZonesBoxWindowChange()` re-fetches immediately on change rather than waiting for the next `ZONES_BOX_REFRESH` poll.
- **Trial points** (Delta Chart only, experimental — "Trial Points" checkbox next to the zones-box window radios, **unchecked by default**) — three single-point markers, **not** time-series lines: a red dot for `bottom_intersection`, a green dot for `top_intersection`, and a violet dot for `gamma_band` (same colors as their Zones Extrema line counterparts, for visual association), all plotted at the current (latest) candle's own time — `realCandles[realCandles.length - 1].time`, not raw `Date.now()`, so the points sit exactly on that candle rather than floating mid-bucket on coarser timeframes (4h/1d) where "now" rarely lands on a bucket boundary. `trialBottomPoint`/`trialTopPoint`/`trialGammaPoint` are `addLineSeries` with `lineVisible: false, pointMarkersVisible: true, pointMarkersRadius: 6` — a single-point `addLineSeries` otherwise draws nothing, since there's no second point to connect a line to. Sourced from `/api/trial-points/<asset>` (`trial_points_json` in `main.py`), computed from trades in the trailing `dankbit.zones_box_window_hours` hours (the same setting the zones-box trade-window radio toggle uses) for the nearest active expiry — same underlying math as `dankbit.zones.extrema._compute_asset()` (`options.build_zone_curves()`/`find_zero_crossings()` for the intersections, `gamma.portfolio_gamma()` with the same `r=0.0` convention for the gamma extrema), but **deliberately reimplemented standalone in `main.py`** rather than calling that model at all — no shared code path, no persistence, so this trial feature can never affect the existing Zones Extrema lines/boxes, their cron, or their persisted history. Same server-load-reducing pattern as "Lines"/"Zones Extrema": leaving the checkbox unchecked stops `refreshTrialPoints()` from calling the server at all, not just hiding the result; `onToggleTrialPoints()` clears the 3 points immediately when switched off and fetches immediately when switched on. `refreshTrialPoints()` polls on the ordinary `REFRESH` interval (not `ZONES_BOX_REFRESH`) when on, same as the Zones Extrema lines/boxes: shown on every timeframe, no `INTERVAL` gating at all. Right after the checkbox, a plain **"UTC Now:" clock** (`#utc-now-label`) — no fetch involved, just `new Date().toISOString().slice(11, 19)` (the "UTC Now:" label itself already conveys the timezone, so the value has no trailing "UTC" suffix), ticking every second via its own `setInterval(updateUtcNowLabel, 1000)` independent of any data poll.
- Crosshair price label (`crosshair.horzLine.labelBackgroundColor`) is set to blue — the library default (`#131722`, near-black) was indistinguishable from the black `axisLabelColor` used on every indicator price line
- Do **not** change `title:` values in `createPriceLine()` calls — the user maintains these manually

## Odoo Gotchas

- **Manifest licence field:** Odoo only accepts a fixed set of strings. Use `"Other OSI approved licence"` (British spelling) for MIT — `"MIT"` alone will fail validation.
- **`auth="public"` routes writing to the ORM:** every model in this addon except `dankbit.trade` only has `ir.model.access.csv` rules for `base.group_user` (internal users) — no public/portal grant. Reading via raw SQL (`request.env.cr.execute(...)`) bypasses this entirely, but any `self.search()`/`self.create()`/`record.write()` call reached from a public route needs an explicit `.sudo()`, or it 403s for genuinely anonymous requests. This is easy to miss in testing: `odoo shell` runs as superuser (no check at all), and hitting the route from a browser tab that's already logged into the backend authenticates as that admin user instead of the public one — only a cookie-less request (e.g. plain `curl`) exercises the real public-user path. See `dankbit.zones.extrema._persist_extrema()` for the fix.

## Odoo Module Reload

Python changes to `my_addons/` take effect after `docker compose restart web`. XML/view changes and new model fields require upgrading the module:

```bash
docker compose exec web odoo -d <db_name> -u dankbit --stop-after-init
docker compose restart web
```
