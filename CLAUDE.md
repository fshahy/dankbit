# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Running the Stack

```bash
# Start all services
docker compose up -d

# Rebuild after code changes
docker compose up -d --build

# Rebuild a single service (e.g. after editing dankbit_ws_batch.py)
docker compose up -d --build dankbit_ws

# Tail logs
docker compose logs -f
docker compose logs -f dankbit_ws

# Restart just the Odoo addon (no rebuild needed for Python changes to my_addons)
docker compose restart web
```

After first startup, install the Dankbit addon via Odoo UI: Apps → search "Dankbit" → Install.

**Required `.env` variables:**
```
DANKBIT_POSTGRES_DB=<db_name>   # any name; db1 is the development default
DERIBIT_KEY=<key>
DERIBIT_SECRET=<secret>
```
The `UID` and `GID` env vars are also needed so the `web` container writes files as the host user (dev default: `1000`/`1000`). `docker-compose.yml` sets `user: "${UID}:${GID}"` on the `web` service and `chown`s the Odoo data dir to that UID/GID before starting — the values must match whichever host user owns the bind-mounted directories, so Odoo's writes (filestore, logs) land with an owner the host can actually manage. Check with `ls -ln <bind-mounted dir>` if unsure. Setting it to `0`/`0` (root) is correct only when the production host directory itself is root-owned — it also means Odoo runs as root inside the container, a broader security surface than an unprivileged UID.

> **Production note:** `config/odoo.conf` contains `dbfilter = ^db1$` — update this to match the actual database name.
> **Production note:** production's `.env` uses `UID=0`/`GID=0` (root), unlike the dev default of `1000`/`1000` — see above for when that's the correct choice.

## Architecture

Two Docker services talk to a shared PostgreSQL database:

```
Deribit WS API ──► dankbit_ws (Python asyncio) ──► PostgreSQL
                                                        │
                   Deribit REST API ◄── Odoo cron       │
                                                        ▼
                                               Odoo 18 (web:8069)
                                                        │
                               ┌───────────────────────┤
                               │                       │
                        PNG charts                TradingView
                  /BTC-<EXPIRY>, /i/BTC-<EXPIRY>  /chart/BTC
```

**Trade data enters two ways:**
1. **WebSocket (primary):** `dankbit_ws_service/dankbit_ws_batch.py` — connects to Deribit, authenticates, fetches all BTC+ETH option instruments, subscribes to `trades.<instrument>.raw` channels in chunks of 400, inserts rows directly into PostgreSQL with `ON CONFLICT IGNORE` on `deribit_trade_identifier`.
2. **REST backfill:** `Trade.get_last_trades()` — runs every minute via cron; ensures no trades were missed by paging through the last few days of Deribit REST API history.

**PNG chart rendering:** Per-expiry routes like `/BTC-7MAY26` aggregate all trades for that expiry, compute a Black-Scholes portfolio delta and dollar-gamma (GEX = Γ × S²) curve, and return a PNG via matplotlib's Agg backend (`Figure + FigureCanvas`, never `pyplot` — server-safe). Overlaid lines: gamma peaks and bottoms (dashed black), delta=0 crossings (solid — green for "supply" [delta negative below, positive above], red for "demand" [delta positive below, negative above]).

**Zones chart (`/<instrument>/zones`):** a distinct plot mode — separates that instrument's trades since 00:00 UTC into buy-side and sell-side (trade domain uses `("name", "=ilike", f"{instrument}-%")`, an anchored/left-prefix match — not a bare `ilike` substring match — so a query for one expiry can never pull in another instrument's trades), builds two independent `OptionStrat` payoff curves via `options.build_zone_curves()` (`long_call`/`long_put` for buys, `short_call`/`short_put` for sells — same payoff math as the combined-portfolio routes, just accumulated into two separate objects, plus the crossing-based auto-zoom below — same function `dankbit.bands` calls, see Models), and plots them together via `OptionStrat.plot_zones()`: green "Longs" curve, red "Shorts" curve, blue vertical line at index price — `plot_zones()` itself draws no text overlay. The `Seller Max Profit (SMP)`/`Buyer Max Loss (BML)`/`High Zone`/`Low Zone`/`Middle Zone` info is rendered as page HTML, alongside further per-expiry stats: `chart_png_zones` calls `options.zone_summary(STs, longs_curve, shorts_curve)` (module-level function, same extrema/box-boundary definitions as `dankbit.bands` — see Models) for `seller_max_profit`/`buyer_max_loss`/`high_zone`/`low_zone`/`middle_zone`/`high_resistance`/`low_support`, labeled on the page as `Seller Max Profit (SMP)`/`Buyer Max Loss (BML)`/`High Zone`/`Low Zone`/`Middle Zone`/`High/Resistance`/`Low/Support`, same `(SMP)`/`(BML)` convention the Gamma/Delta/Theta/Vega headers below use. Current price is not a factor in any of these: `high_zone`/`low_zone` are each curve's own highest/lowest zero-crossing (the Shorts curve's and Longs curve's max feed `high_zone`, their min feeds `low_zone`) rather than "the nearest crossing above/below index price" — a zone forms as soon as *either* curve has at least one crossing (a lone curve's crossing yields a degenerate same-price pair, rendered as a single price rather than a dashed range); only `None`/`"n/a"` if neither curve ever crosses zero. `middle_zone` is a third zone bounded by `seller_max_profit`/`buyer_max_loss` (min/max of the two) rather than any zero-crossing — always defined, since those two values are argmax/argmin over the full curve and don't depend on a crossing existing, unlike `high_zone`/`low_zone`. Then restricts that route's own `trades` to the single nearest expiry (mirroring `dankbit.bands`'s "next expiry only" restriction) and delegates the actual per-leg gamma/delta/theta/vega extrema computation to `options.per_leg_greeks()` — the single source of truth for this also used by `dankbit.bands`'s `gamma_band`/`delta_band` and by `forecast3.per_leg_greeks()`, so this page, the persisted bands record, and the Thales Forecast engine can never quietly disagree on these numbers for the same trades. Gamma extrema are preceded in `zone_info_lines` by a violet-colored `Gamma` section-header line: `Buyer Call Gamma (BCG)`/`Buyer Put Gamma (BPG)` (price where the long-call/put gamma curve peaks, `argmax`) and `Seller Call Gamma (SCG)`/`Seller Put Gamma (SPG)` (price where the short-call/put gamma curve bottoms out, `argmin` — short positions carry negative gamma, so the relevant extremum is a trough, not a peak — same BCG/BPG/SCG/SPG abbreviations `dankbit.forecast3.snapshot`'s `bcg_price`/`bpg_price`/`scg_price`/`spg_price` fields use) — plus, in their own block right below, the gamma value (not dollar-formatted, unlike the price lines) at each of those same 4 extrema (`{curve}[argmax/argmin index]`, the y-value at the point already located for the price line, not a separate computation) — shown as an absolute value divided by 1,000,000 and rounded (the sign is already conveyed by which side of the section it's on: peaks are long/positive, bottoms are short/negative), since these raw dollar-gamma numbers run into the hundreds of millions and the user copies them by hand into another tool — labeled `BCG Abs.`/`BPG Abs.`/`SCG Abs.`/`SPG Abs.`. Also computes each leg's delta-saturation price via `options.delta_saturation_price()` (calls saturate ITM at high S, puts at low S, independent of long/short, `fraction=DELTA_SATURATION_FRACTION`=0.9 — see below), preceded by a green-colored `Delta` header, labeled `Buyer Call Delta (BCD)`/`Buyer Put Delta (BPD)`/`Seller Call Delta (SCD)`/`Seller Put Delta (SPD)` (same abbreviations `dankbit.forecast3.snapshot`'s `bcd_price`/`bpd_price`/`scd_price`/`spd_price` fields use), plus — in their own block right below — the portfolio-delta value at that same price, labeled `BCD Abs.`/`BPD Abs.`/`SCD Abs.`/`SPD Abs.` — shown as an absolute value divided by 10 and rounded to a whole number, same reasoning as the Gamma Value lines above (dividing by 1,000,000 there, 10 here, since delta values are already two to three orders of magnitude smaller). That value is interpolated off the underlying `delta.portfolio_delta()` curve at the saturation price rather than assumed to equal `fraction * curve edge` exactly, since `delta_saturation_price()` falls back to the `STs` edge (not a true 90%-threshold crossing) when the curve never reaches that threshold within this window — interpolating keeps the reported value correct in that fallback case too. Also computes, per leg, the price where portfolio theta ($/day — see `theta.py`, a new module mirroring `delta.py`/`gamma.py`'s `bs_*`/`portfolio_*` pattern; `bs_theta()` returns the annualized Black-Scholes theta ÷ 365, the convention traders quote, not the raw per-year value) is most extreme over the same price grid gamma/delta use — sign-flipped from gamma's peak/bottom split: long positions carry negative theta (decay cost), so their extremum is the trough (`argmin`, worst decay); short positions carry positive theta (decay gain), so theirs is the peak (`argmax`, best decay) — against the same next-expiry-only `long_calls`/`long_puts`/`short_calls`/`short_puts` recordsets gamma/delta reuse, with the same `r=0.0` convention as this page's other Greeks. Preceded by an orange-colored `Theta` header, labeled `Buyer Call Theta (BCT)`/`Buyer Put Theta (BPT)`/`Seller Call Theta (SCT)`/`Seller Put Theta (SPT)` (same abbreviations `dankbit.forecast3.snapshot`'s `bct_price`/`bpt_price`/`sct_price`/`spt_price` fields use) — plus, in their own block right below, the theta value ($/day) at each of those same 4 extrema (`{curve}[argmin/argmax index]`, same pattern as the gamma/delta value blocks), labeled `BCT Abs.`/`BPT Abs.`/`SCT Abs.`/`SPT Abs.` — shown as an absolute value divided by 10,000 and rounded, same reasoning as the Gamma/Delta Value lines (dividing by 1,000,000 for gamma, 10 for delta, 10,000 here, since raw $/day theta values sit in between those two scales). Also computes, per leg, the price where portfolio vega (`vega.py`, a module mirroring `gamma.py`'s `bs_*`/`portfolio_*` pattern rather than `theta.py`'s — vega has no call/put distinction in Black-Scholes, so `bs_vega()` takes no `option_type`, same as `bs_gamma()`; returns raw vega / 100, the $-per-1-IV-point convention traders quote, mirroring `bs_theta()`'s /365) is most extreme over the same price grid — same peak/bottom split as gamma, not theta's flipped one: vega is always positive for a long option and always negative for a short one, so long legs peak (`argmax`) and short legs bottom out (`argmin`) — against the same next-expiry-only `long_calls`/`long_puts`/`short_calls`/`short_puts` recordsets and `r=0.0` convention as this page's other Greeks. Preceded by a blue-colored `Vega` header, labeled `Buyer Call Vega (BCV)`/`Buyer Put Vega (BPV)`/`Seller Call Vega (SCV)`/`Seller Put Vega (SPV)` (same abbreviations `dankbit.forecast3.snapshot`'s `bcv_price`/`bpv_price`/`scv_price`/`spv_price` fields use) — plus, in their own block right below, the vega value ($ per 1 IV point) at each of those same 4 extrema, labeled `BCV Abs.`/`BPV Abs.`/`SCV Abs.`/`SPV Abs.` — shown as an absolute value divided by 100 and rounded, same reasoning as the other Value lines (raw portfolio-vega values sit in the low thousands, an order of magnitude below theta's ten-thousands and well above delta's tens, so /100 keeps the displayed number in the same easy-to-copy 2-3 digit range). Unlike every block above Theta, Theta and Vega are kept in a *separate* `right_info_lines` list rather than appended to `zone_info_lines` (passed into the shared template under the `theta_info_lines` context var name, which carries both Greeks), and rendered in its own top-right `.zone-info-right` overlay (mirrors `.zone-info`'s styling, just `right: 12px` instead of `left: 12px` for the box position — text inside stays left-aligned, same as `.zone-info`) — so it sits to the right of the PNG instead of stacking underneath Gamma/Delta on the left. All of this is assembled into `zone_info_lines`, where every line is a `{text, color}` dict rather than a plain string (color `None` for the default black styling; only section headers set one), rendered by the template via `t-att-style` on that per-line `<div>` (grouped into 7 blocks — price extrema; boxes; intersections; gamma peak/bottom prices; gamma peak/bottom values; delta-saturation prices; delta-saturation values — separated by blank spacer lines, a `" "` non-breaking space since a truly empty/whitespace-only `<div>` can collapse to zero height) and passed to the shared `dankbit_page` template's `zone_info_lines` context var, rendered as a fixed top-left overlay div (`.zone-info`, `user-select: text` so it's copyable, only shown when that var is present, so every other PNG route using `dankbit_page` is unaffected). A separate bottom-left annotation (added by `chart_png_zones` itself, drawn inside the image via `ax.text()`) shows the long/short trade counts since 00:00 UTC. **Auto-zoomed x-axis:** the configured `from_price`/`to_price` settings are only used for an initial wide pass to locate where the Longs/Shorts curves cross (linear interpolation on the sign change of `longs.payoffs - shorts.payoffs`, same technique as the delta=0 crossing finders); the curves are then rebuilt over `[leftmost_crossing - margin_below, rightmost_crossing + margin_above]` at the same `steps` resolution, so the rendered chart is tightly framed around the relevant region instead of the full (much wider) configured range. Margin is asset-dependent: BTC uses ±$2000; ETH uses ±$100 (ETH's much smaller price scale made the ±$2000 BTC margin blow out the auto-zoom). Falls back to the original wide range if the curves never cross.

**Single-leg delta/gamma routes (`/<instrument>/lp`, `/lc`, `/sp`, `/sc`):** four routes, one per leg type (long put, long call, short put, short call) — `_chart_png_single_leg(instrument, leg_key)` is the shared implementation, dispatched via `_LEG_ROUTES[leg_key]` (`direction`/`option_type` domain filter, which `OptionStrat` leg method accumulates trades, route label, and delta-saturation `fraction`/`side` — see below). Same domain convention as `chart_png_zones` (anchored `=ilike` instrument match, trades since 00:00 UTC), but restricted to that one direction+option_type combination and built over the **full configured price range** (`from_price`/`to_price`/`steps`), not the crossing-based zoomed window `/zones` uses — matching `chart_png_hours`/`chart_png_all`'s style instead, including their `r=0.05` risk-free rate for `delta.portfolio_delta`/`gamma.portfolio_gamma` (deliberately *not* `r=0.0`/zoomed-grid like `dankbit.bands`). `obj.plot()` renders the same delta+gamma dual-axis chart as those routes; the gamma peak/bottom + delta=0 crossing overlay is factored into `_annotate_gamma_delta_crossings(ax, STs, market_deltas, market_gammas)` (shared by all 4 routes, so they don't each carry their own copy of that block). A solid green `axvline` additionally marks the delta-saturation price from `options.delta_saturation_price()` — see `options.py` below — labeled with its price, drawn after the standard overlay.

**TradingView chart:** `/chart/BTC` and `/chart/ETH` serve a Lightweight Charts v4 page with live Deribit perpetual futures candles (proxied server-side) and price lines refreshed on `dankbit.refresh_interval` — except the yellow/teal zones boxes, which refresh on their own dedicated `dankbit.zones_box_refresh_interval` setting (default 3600s/1h, see Settings fields below):
- Delta=0 lines (3 sets) — each a single `createPriceLine`, `lineWidth: 1`, `lineStyle: Solid`. Color is by crossing type, not by set: **black** ("S/D" in the legend) for "supply" (delta negative below, positive above), **red** ("Tie-out" in the legend) for "demand" (delta positive below, negative above — the important case). Set identity is shown via the line's `title` only. Every `createPriceLine` call across the chart (delta=0 sets, gamma peak/bottom, ruler) sets `axisLabelColor: 'black'` so the axis label box is always black regardless of the line's own color — only the label background is forced black, not the line stroke — **except** Daily (24H)/Daily+1 (24H) (below), whose axis label is `axisLabelColor: 'orange'` instead, a deliberate one-off deviation from the rest of the chart. `addLineSeries` (Net Call/Put Gamma) has no equivalent per-series label-color override in Lightweight Charts v4, so those labels still match their line color (blue/orange):
  - **Weekly** — sourced from `/api/delta-zero/<instrument>`
  - **Monthly** — sourced from `/api/delta-zero/<monthly-instrument>`
  - **All** — all active expiries; sourced from `/api/delta-zero-all/<asset>`
- Daily (24H) / Daily+1 (24H) delta=0 lines — two distinct `createPriceLine` sets, color-by-crossing-type like Weekly/Monthly/All (black for "supply" — delta negative below, positive above; red for "demand" — delta positive below, negative above), but with an **orange** axis label (`axisLabelColor: 'orange'`, unlike every other line on this chart — see above) and **no footer-legend entry**, titles `Daily (24H)`/`Daily+1 (24H)`, sourced from `/api/delta-zero-tomorrow/<asset>` and `/api/delta-zero-day-after-tomorrow/<asset>` respectively — both thin wrappers around the shared `_delta_zero_for_calendar_day(asset, days_ahead)` helper (`days_ahead=1`/`2`) so the two can never disagree on how a calendar-day expiry/trade-window is computed. Each targets the specific expiry landing `days_ahead` calendar days out (UTC) — computed directly as `today + N days` formatted to Deribit's no-leading-zero day convention (e.g. `7JUL26`), not "nearest active expiry" (`MIN(expiration) >= NOW()`), since the latter can still resolve to *today's* not-yet-happened expiry. Restricted to trades from the trailing 24h (`deribit_ts >= NOW() - INTERVAL '24 hours'`). Gated by `show_daily_lines` (see Settings fields below).
- Bands lines (3) — `High/Resistance` (green, `#2e7d32`, solid), `Low/Support` (red, `#c62828`, solid), and `Gamma Band` (light violet, `#c9a0dc`, **dashed** — `LightweightCharts.LineStyle.Dashed`), `addLineSeries` on the **default/main price scale** (no `priceScaleId` override) since these are plain BTC/ETH price levels, not dollar amounts — they render directly alongside the candles. Sourced from `/api/bands/<asset>`: one point per **instrument** stored in `dankbit.bands` (see Models — one continuously-refined-then-frozen record per expiry, not a time-series of polls), positioned at that instrument's own expiration time rather than a poll timestamp — so these are term-structure-style lines connecting each expiry's final value to the next expiry's, not a history of one expiry's value changing over wall-clock time. `Gamma Band` is the average of that expiry's 4 gamma extrema (Buyer Call/Put Gamma (BCG/BPG), Seller Call/Put Gamma (SCG/SPG) — see Models, `dankbit.bands._compute_asset`'s `gamma_band`), the same 4 numbers the `/<instrument>/zones` PNG page's info overlay shows. The API response also carries `delta_band` — the average of that expiry's 4 delta-saturation prices (see `options.delta_saturation_price()`) — but it is deliberately **not drawn** on this chart (no `deltaBandSeries` in the template); it's computed and persisted for later use, not currently rendered. The endpoint derives each point's `t` via a single grouped lookup against `dankbit_trade` (`SUBSTRING(name FROM '^[^-]+-[^-]+')` to recover the instrument prefix, `MIN(expiration)`) rather than parsing the instrument string's day-suffix and assuming a settlement hour — raw SQL, so it still finds archived (past-expiry) instruments' trades despite the ORM's implicit `active=True` filter, since those rows must keep contributing a fixed historical point. Connected point-to-point by the line series. Shown on **every timeframe** (1h/4h/1d); `setTf()`/`init()` call `refreshBands()` on every timeframe switch. High/Resistance and Low/Support points additionally carry a `+`/`-` `setMarkers()` label (via `high_resistance_positive`/`low_support_positive` — see Models) showing whether the payoff *at* that intersection sits above or below the zero-payoff line, since the crossing's x-price alone doesn't say — `size: 0` hides the marker's shape glyph so only the text renders, colored to match its own line.
- Zones boxes (2, plus a 3rd Seller Max Profit/Buyer Max Loss box — see below; toggled together by the "Zones" checkbox, TradingView Chart Notes below — checked by default) — filled rectangles (`#fbc02d`, 25% opacity fill, matching top+bottom border lines) marking the top and bottom zero-crossing zones for the **nearest** active expiry only (see Models, `get_box_n`/`TRACKED_EXPIRY_COUNT` for how later expiries are still tracked, just without a box). The 4 box-boundary fields themselves are **not** read from stored history — sourced live from `/api/zones-box/<asset>`, which calls `dankbit.bands.get_box()` to compute fresh on every request via `_compute_asset()` (see Models: nothing reads box-boundary history, only the latest value is ever drawn). `get_box()` computes fresh via `_compute_asset()` on every call but never persists anything (see Models, `get_box`) — the Bands lines' per-expiry history, for every tracked expiry including this nearest one, is fed *solely* by the 4-hourly `compute_snapshot()` cron (see Models, `_persist_extrema`); no browser action, on this endpoint or any other, ever writes to `dankbit.bands`. The boxes' own visual state is never affected by the cron either way — box-boundary fields are recomputed fresh on whatever request (browser or cron) triggers `_compute_asset()`, but only the browser's request result is ever rendered; the cron's result is only ever persisted, never displayed. Each box is built from two `addBaselineSeries` (`aboveBoxSeries`/`belowBoxSeries` — a flat top-price line filled down to a `baseValue` bottom price) plus a plain `addLineSeries` for the bottom edge (`aboveBoxBottomLine`/`belowBoxBottomLine`), since `addBaselineSeries` only strokes its top edge. Top/bottom of each box are `max`/`min` of that side's two zero-crossing prices (`short_zero_above_price`/`long_zero_above_price` for the top box, `short_zero_below_price`/`long_zero_below_price` for the bottom box) — despite the field names, these are each curve's own highest (`*_above_price`) or lowest (`*_below_price`) zero-crossing, not "the nearest crossing above/below current index price" (see Models, `_compute_asset`). `setBoxData()` (in `dankbit_templates.xml`) draws the full rectangle when both a top and bottom price exist and differ; if only one curve contributed a crossing, or both coincide (a zero-height box), it instead draws a single line at whichever price is available (reusing the box's own bottom-edge line series) rather than dropping it silently; only clears both when neither curve crosses zero at all (a `0.0` value on **both** sides). Time span is `[now - 10 days, expiration]` — the right edge is that expiry's own real settlement time (Deribit's `expiration_timestamp`, e.g. 08:00 UTC / 10:00 CEST — read off `dankbit_trade.expiration` via `_compute_asset()`, not assumed/hardcoded), so the box now visibly runs all the way out to when the trades it's summarizing actually stop mattering, rather than stopping at "now". The left edge is a flat 10 days back from the browser's current wall-clock time (`Date.now()`, tz-shifted like every other timestamp on this chart), recomputed on every poll rather than fixed to a stored tick. Shown on **every timeframe** (1h/4h/1d), same as the Bands lines.
- Seller Max Profit / Buyer Max Loss box (3rd box, teal) — same mechanism (`addBaselineSeries` + bottom-edge `addLineSeries`, `setBoxData()`'s single-line fallback when top/bottom coincide) and the exact same `fromTime`/`toTime` as the yellow box above, since both come from the same `/api/zones-box/<asset>` response for the nearest expiry — `smpBmlBoxSeries`/`smpBmlBoxBottomLine`. Top/bottom are `max`/`min` of `seller_max_profit` (price where the Shorts payoff curve peaks) and `buyer_max_loss` (price where the Longs curve bottoms out) — see Models, `dankbit.bands`. No separate endpoint: `seller_max_profit`/`buyer_max_loss` were already computed by `_compute_asset()` as a side effect of the same `get_box()` call the yellow box uses, just added to `/api/zones-box/<asset>`'s JSON payload.
- Thales Forecast candles ("Thales Forecast" checkbox) — **Delta Chart only** (the Gamma Chart's `show_gamma_point == 'true'` branch has no such checkbox at all — removed from there; `SHOW_FORECAST3` simply never gets set true on that page, so `forecast3Series` — still created there, since both charts share the same template/JS — never receives data), unchecked by default, after "Bands"; **restricted to the 4h timeframe** (`updateForecastVisibility()`, called from `setTf()` and at page init, shows/hides `#forecast3-toggle-wrap` — starts hidden inline — based on `INTERVAL === '4h'`; `refreshForecast3()` itself also gates on that condition so a stray poll can't redraw it off-timeframe). `updateForecastVisibility()` is still called from both charts' shared `setTf()`/`init()`, so it null-checks `#forecast3-toggle-wrap` before touching it rather than assuming the element exists, since the Gamma Chart no longer renders it. A full port of Thales's "Thales Bands" Pine indicator's forecast-candle engine (`controllers/forecast3.py`) onto Dankbit's own live-computed per-leg Greeks, in place of Thales's manually-typed-in daily CSV rows (a companion HTML data-entry tool, "Thales Offline Data Builder", is what a human uses to feed that Pine script by hand — Dankbit needs no such tool since it already computes the equivalent numbers live). **Fully deterministic** — there is no GBM/random component anywhere in this engine (matching the source script, which has none either); every candle is a direct function of the current Greeks, real recent price action, and time-of-day/day-of-week, not a sampled realization. Runs out 18 4h-spaced candles (72h — `simulate_forecast3`'s `hours_ahead=72` default), widened from Thales's own `forecastCandleCount` default of 6 candles/24h, since nothing in the engine assumes a specific cutoff — the per-step decay terms (`0.75 ** step`/`0.82 ** step`/etc.) just keep tapering further out regardless of horizon length. Sourced from `/api/forecast3/<asset>` (`forecast3_json` in `main.py`), rendered as an `addCandlestickSeries` (`forecast3Series`), light red/green by default (`#a5d6a7`/`#ef9a9a`, wicks `#66bb6a`/`#e57373`) to stay easily distinguishable from the real candles' own teal/red (`#26a69a`/`#ef5350`) at a glance. User-configurable via 4 Settings → Dankbit → Thales Forecast fields (`forecast3_up_color`/`forecast3_down_color`/`forecast3_wick_up_color`/`forecast3_wick_down_color`, plain hex-string `Char` fields, no color-picker widget) — rendering-only, so they're read by `_build_tv_chart_context()` (not `_forecast3_cfg()`, which only feeds `simulate_forecast3()`'s own math) and passed through the same `#cfg` div → JS `dataset` pattern every other per-page chart setting already uses (`data-forecast3-up-color` etc. → `cfg.forecast3UpColor` etc.), same as `data-zones-box-refresh` → `cfg.zonesBoxRefresh`. `refreshForecast3()`/`onToggleForecast3()`/`clearForecast3Lines()` recompute each candle's display time client-side, anchored to the same grid `futureAxisSeries` uses for its own future ticks (`realCandles[last].time`, stepped by `intervalSec`) rather than trusting the server's own now-relative `t` field, which almost never lands on that shared time-axis grid.
  - **Data mapping.** Thales's manually-entered Buyer/Seller Call/Put Gamma/Delta/Theta/Vega quadrant + BML/SMP (curve extremes) + Resistance/Support fields map onto values Dankbit's Greeks already compute for the nearest active expiry's since-00:00-UTC trades: per-leg gamma peak/bottom prices (`gamma.portfolio_gamma()`, same as the `/<instrument>/zones` page), per-leg delta-saturation prices (`options.delta_saturation_price()`), per-leg theta/vega extrema prices (`theta.py`/`vega.py`, same argmin/argmax split those already use), `buyer_max_loss`/`seller_max_profit` (BML/SMP), and the zones box's own top/bottom edges (`options.zone_summary()`'s `high_zone`/`low_zone`, the same numbers the yellow zones box draws) standing in for Thales's manually-drawn Resistance/Support reference band. Each quadrant/leg value's "Abs" strength counterpart (used throughout the engine as a confidence/weighting signal) is the same dollar-gamma/delta/theta/vega *Value* number (abs, divided by 1e6/10/1e4/100 respectively) the `/<instrument>/zones` page's info overlay already shows. Thales's original CoinGlass liquidity-map fields (lower/upper liquidity price+volume) have no automated Dankbit equivalent — there's no CoinGlass API integration. Pine v6.2.1's **Smart Role-Aware Synthetic Liquidity** covers this instead — levels computed straight from the same per-leg gamma/delta/theta/vega Greeks this module already has: `forecast3.smart_synthetic_liquidity()` treats each seller call/put leg as a "pin" role (a market-maker's short option defending a level, i.e. resistance/support) and each buyer call/put leg as a "sweep" role (a long option holder's pressure toward their strike), each weighted by that leg's own Abs strength; a seller call/put's delta contribution flips from pin to sweep once a real close has convincingly broken past that leg's own delta level (`SMART_LIQ_SELLER_DELTA_FLIP_BUFFER`), and the raw blended level is nudged toward the nearest zones-box edge when close to it, or damped when far past it (`_smart_band_adjusted_level`/`_smart_band_adjusted_strength`, mirroring the source script's band-merge/outside-band logic). `simulate_forecast3` feeds this straight into `forecast3.liquidity_map_engine()` (unchanged): whichever side (lower/upper) has the stronger, closer liquidity pulls price toward it once dominant enough over the other side (`LIQUIDITY_DOMINANCE_THRESHOLD`), and a real candle sweeping through a level and closing back away from it (`upper_swept_rejected`/`lower_swept_rejected`) fires an extra rejection impulse and a body-confidence floor/wick compression, same as every other engine above. Entirely inert (zero impulse, no confidence floor) whenever the synthetic engine produces no level for that side.
  - **No future-known levels, unlike Thales.** Thales's own rows are manually dated, entered in advance up to a known expiry, so its forecast loop interpolates between a "current" and a "future" row that are both already-known data points. Dankbit has no foreknowledge of tomorrow's Greek levels, so there is only ever one "current" set of levels (top/low/gamma/curve/theta averages) — held constant across every forecast candle. The only thing that varies over real elapsed time is the history feeding the Gamma-Band Consensus engine's slope math (see below).
  - **`dankbit.forecast3.snapshot`** (new model, see Models) is what supplies that real time-series: one row per `(asset, 4h time bucket)`, continuously refined while its bucket is current then frozen once time rolls past it — the same "continuously-refined-then-frozen" pattern `dankbit.bands` uses per-instrument, just keyed by a rolling time bucket instead of an expiry. `compute_and_persist()` is called live on every `/api/forecast3/<asset>` request (refining the current bucket) — **no cron fallback** (unlike Bands' dual live-poll-plus-4-hourly-cron pattern): history only accumulates while the "Thales Forecast" checkbox is open somewhere, and a gap just widens the Gamma-Band Consensus engine's slope window rather than breaking anything, so the extra cron wasn't worth keeping.
  - **Engine cascade** (`forecast3.py`, Thales's own default constants ported as fixed module constants rather than ~150 exposed settings fields): a weighted gamma/curve/theta center (`GAMMA_CENTER_WEIGHT`/`CURVE_CENTER_WEIGHT`/`THETA_CENTER_WEIGHT`, default 0.70/0.20/0.10; the one nested-helper (`derive_levels()`) constant trio exposed to res.config.settings' "Thales Forecast" section despite the "top-level only" scoping rule below, via `derive_levels`'s own `cfg` parameter — reweighted under Gamma-Curve Divergence when the gamma and curve averages disagree by more than `DIVERGENCE_THRESHOLD` band-widths, a separate risk-management behavior left at Thales's own 0.42/0.43/0.15 defaults) of `buyer_gamma`/`seller_gamma`/`gamma_avg` (`derive_levels()`) — the **plain unweighted** average `(BCG+BPG)/2` and `(SCG+SPG)/2`, matching the source script's real `buyerGammaVal`/`sellerGammaVal`/`avgGammaVal`, not its strength-weighted "Gamma Pressure Center" (`gammaPressureCenterVal`, behind the "Calculate Gamma Pressure Center From S Strengths" toggle) — tracing every use of that weighted value in the v6.2.1 script shows it only ever feeds a diagnostic label, never the real forecast center blend or the Gamma-Band Consensus slope math, both of which read the plain-average arrays instead; session-aware (Asia/London/Overlap/NY/Post-NY) and weekend/regime multiplier tables (UTC-only — Thales's own `sessionTimezone` also defaults to UTC); the Gamma-Band Consensus backbone (`gamma_band_consensus()`, a 3-point top/low/gamma slope comparison against the last 2 persisted snapshot rows, real-hours-normalized) plus its Reclaim Bias and, from the newer v6.2.1 Pine version, its **Trend Lock** — once an all-3-aligned consensus is strong (`GAMMA_BAND_TREND_LOCK_STRENGTH`) and a real close still sits on its "correct" side of the gamma reference, a single countertrend real candle is treated as a retest rather than a reversal: its body impulse is damped (`GAMMA_BAND_COUNTER_BODY_DAMPING`) and the forecast's opposite-direction impulse is capped (`GAMMA_BAND_COUNTER_MAX_OPP_IMPULSE`), until a real candle actually escapes back through the gamma reference with enough body-to-ATR force (`GAMMA_BAND_COUNTER_ESCAPE_ATR`) to lift the lock — computed once per call (`gb_counter_trend_locked` in `simulate_forecast3`), same as consensus/momentum, since it depends only on the current snapshot and the single most recent real candle, not on any per-step state; a Delta Shock Module and Gamma Shock Module (band breaks confirmed by price also sitting on the correct side of the gamma reference for `GAMMA_CONFIRM_BARS` real closes); a Vega Regime Engine (buyer/seller vega dominance expanding or compressing the wick); a Market-Maker Gamma Contest Engine (buyer-call breakout force vs seller-call pin force, and the put-side mirror, each scaled by price's proximity to that leg's own gamma level — the newer Pine version's per-leg delta-confirmation/momentum-factor/contest-detection enrichment of this engine is deferred; this port keeps the simpler proximity-only weighting it already had, now also exposing the raw local force totals for the Wick-to-Body Acceptance Engine below); an Option Cluster Structure Engine (how tightly top/low/gamma/BML/SMP agree with each other, and whether they're all moving the same direction since the previous snapshot); a Gamma Neutral/Hysteresis Zone (chop damping near the gamma reference); a Momentum/Liquidity-Sweep Override; and, also new in v6.2.1, the **Wick-to-Body Acceptance Engine** (`wick_to_body_acceptance()`) — converts part of an overly asymmetric wick into the candle body when one side's Market-Maker force clearly dominates and enough independent signals (force dominance, real-candle momentum, liquidity dominance) agree, modeling a market that "accepts" a directional move rather than merely wicking through it; simplified relative to the source script to use only the LOCAL (price-proximity-gated) MM forces rather than the newer version's richer "global pressure" blend, since that blend needs the deferred deeper MM enrichment above. All of this is computed against real recent 4h candles fetched server-side from Deribit (`ChartController._fetch_candles()`, factored out of `/api/klines/<asset>` so both share one aggregation implementation) for a genuine ATR14 and body/wick momentum detection — Thales's own bar-by-bar `barstate.isconfirmed` live-candle gating collapses to "the last N already-closed real candles" here, since Dankbit generates the whole path in one batch rather than re-evaluating live each bar. The Quadrant Behavior Engine (off by Thales's own default anyway), the newer version's full Regime-Adaptive Coefficient blending (weekday/weekend/low-volume/high-volume weight interpolation across ~7 factors — this port still only blends the pull/shock multipliers, not the gamma/curve/theta center weights), its granular per-phase (Saturday/Sunday-day/Sunday-evening) weekend move caps, its theta breakout impulse, its gamma-abs pull/pin/shock multipliers, and its Confirmed Scenario Gate (live/unclosed-candle damping) are intentionally deferred rather than transliterated line-for-line — see each function's own docstring in `forecast3.py`.
- No color-key footer legend; each line/series still carries its own identifying `title`.
- Footer status shows: `<weekly-expiry>  ·  <monthly-expiry>  ·  N trades  ·  Nearest: <expiry>`; trade count = all active trades up to and including monthly expiry (from monthly endpoint). The `Nearest:` segment is the same nearest-active-expiry the yellow zones boxes use (`dankbit.bands.nearest_expiry()`, a cheap standalone `SELECT DISTINCT expiration ... LIMIT 1` lookup, no curve-building) — shown regardless of timeframe (`refreshNearestExpiry()`, a lightweight standalone lookup unlike the boxes' own full computation — though both are now shown on every timeframe), via a shared `updateStatusLine()` that both it and `refreshMonthlyDeltaZero()` call so neither overwrites the other's half of the status text.

**`/my/<asset>` (BTC/ETH):** a deliberately minimal page — candles plus exactly one indicator, **Gamma Tops** — sharing the same template (`dankbit_tv_chart_until`) and context-building (`_build_tv_chart_context()`) as `/chart/<asset>` rather than a duplicate template, but with every other indicator that page draws (Weekly/Monthly/All/Daily delta=0 lines, Bands lines, zones boxes) suppressed. Like every route in this addon, `auth="user"` — a logged-out request is redirected to `/web/login?redirect=/my/<asset>` instead of rendering; the JSON APIs this page's own client-side JS calls (`gamma-by-strike`, `gamma-by-strike-until`, `bands`, `klines`, etc.) are `auth="user"` too, so they rely on the same session cookie sent automatically on those same-origin `fetch()` calls once the user is logged in to view the page at all — see Odoo Gotchas. `/chart/<asset>` itself is completely unaffected by any of this — `_build_tv_chart_context()` always sets `show_gamma_point: "false"` there, only `/my/<asset>`'s route overrides it to `"true"` (passed through as the literal string, not a Python bool, for the same QWeb `t-att-*`-drops-falsy-values reason `show_daily_lines` etc. do). The same flag also picks the `<title>`: `Delta Chart` when `"false"` (i.e. `/chart/<asset>`), `Gamma Chart` when `"true"` (i.e. `/my/<asset>`) — reusing `show_gamma_point` rather than adding a separate page-title context var. In the template, `const MINIMAL_CHART = SHOW_GAMMA_POINT;` (the two concepts coincide today — `/my/<asset>` is both the only minimal page and the only one with Gamma Tops — reused rather than adding a second backend flag) gates every other `refresh*` function: most (`refreshDeltaZero`, `refreshAllDeltaZero`, `refreshTomorrowDeltaZero`, `refreshDayAfterTomorrowDeltaZero`, `refreshBands`, `refreshZonesBoxes`) `return` before even fetching when `MINIMAL_CHART`, so no wasted network/server calls either. `refreshMonthlyDeltaZero` is the one exception — it still fetches `/api/delta-zero/<monthly-instrument>` and updates `lastTradeCount` on `MINIMAL_CHART`, just skips drawing its price lines, since that fetch's `trade_count` feeds the footer's "N trades", and the footer (`<weekly-expiry> · <monthly-expiry> · N trades · Nearest: <expiry>`) is unaffected by any of this — same standing rule as always, it's never removed.

**Gamma Tops (`/my/<asset>` only):** a single checkbox — replacing the earlier separate Strike Gamma/Strike Gamma Nearest/Strike Gamma Weekly/Strike Gamma Monthly checkboxes, one per scope, each drawing every strike that had traded — that instead draws exactly one horizontal price line per scope (Nearest/Weekly/Monthly/All active expiries): that scope's own single strike with the highest *positive* combined portfolio dollar-gamma, not a peak/bottom search over a synthetic price grid, and not every strike, just the one with the most gamma. The only indicator on this page — an earlier 4-checkbox "gamma point" scope indicator (Nearest/Weekly/Monthly/Universal, marking every local gamma peak/bottom rather than a single top strike) was removed entirely rather than kept alongside this one. `ChartController._gamma_by_strike(asset, expiry_cutoff=None)` is still the shared underlying computation (unchanged): collapses each instrument's buy/sell trade flow into one net signed position (buy amount minus sell amount — not raw cumulative volume, since a position that's since been closed back out nets to zero, same as it should), clamps that net to the instrument's real open interest via `dankbit.trade.get_open_interest_by_currency(asset)` whenever that instrument appears in the response (left unclamped, not zeroed, if it's missing — treated as "unknown", not "zero", so a transient gap in that one Deribit response can't zero out a real position), then evaluates `gamma.portfolio_gamma()` directly at each distinct strike price from those clamped positions. Net trade flow summed with no time bound can in principle still exceed real open interest (e.g. from data gaps); the OI cap bounds that without discarding the real buy/sell directional signal a textbook OI-times-assumed-dealer-sign GEX calculation would have to assume instead. "All" is sourced with no expiry cutoff (`/api/gamma-by-strike/<asset>`, `gamma_by_strike_json`); "Nearest"/"Weekly"/"Monthly" are restricted to active expiries through the single nearest active expiry / the configured weekly expiry / the configured monthly expiry respectively, all three via one shared `/api/gamma-by-strike-until/<instrument>` route (`gamma_by_strike_until_json` — Nearest resolves its own cutoff instrument client-side via `/api/nearest-expiry/<asset>` first, then passes that into the same route; Weekly/Monthly pass `INSTRUMENT`/`MONTHLY_INST` directly). Checking "Gamma Tops" fires all 4 fetches and, from each of the 4 resulting datasets, keeps only that dataset's single highest-gamma strike (`gammaTop()` in `dankbit_templates.xml` — skips a scope entirely if it has no strikes at all, or if its highest gamma is `<= 0`, i.e. nothing positive to mark). All 4 lines are solid black, `lineWidth: 2`, distinguished from each other only by title text (`'Γ Top ' + label + ' +' + round(gamma/1e6)`, `label` one of `Nearest`/`Weekly`/`Monthly`/`All`) rather than a per-scope color.

## Odoo Addon (`my_addons/dankbit/`)

The addon depends only on `website`. Key components:

**Models:**
- `dankbit.trade` — Core model. Fields map directly to Deribit trade fields. `strike` and `option_type` are computed from `name` (instrument name like `BTC-29NOV24-98000-P`). `days_to_expiry` is UTC-safe. `get_hours_to_expiry()` returns continuous time used in Black-Scholes. `get_open_interest_by_currency(asset)` returns `{instrument_name: open_interest}` for every option instrument on `asset`, from Deribit's public `get_book_summary_by_currency` — one cached bulk call per currency (same `_DERIBIT_CACHE`/`deribit_cache_ttl` pattern as `get_index_price`, see Caching below), not a per-instrument lookup. Used by `ChartController._gamma_by_strike()` to cap the Gamma Chart's "Gamma Tops" checkbox's net per-instrument position at what's actually outstanding right now — see Architecture's `/my/<asset>` section.
- `dankbit.bands` — **One record per instrument** (e.g. `"BTC-10JUL26"`, unique-constrained on `instrument`), not per snapshot-in-time — a row's position on the TradingView chart is still that instrument's own expiration time, looked up from `dankbit_trade.expiration` when the API serves it (see below and TradingView Chart Notes' Bands lines), never `computed_at` below. Fields: `computed_at` (Datetime, `default=fields.Datetime.now` — backend visibility only, e.g. "how stale is this row"; refreshed to the moment `_compute_asset()` ran on every `_persist_extrema()` upsert, not just set once at creation), `asset`, `instrument`, `index_price`, `high_resistance`/`low_support` (the highest/lowest price where the Longs-vs-Shorts payoff curves cross each other — not where either crosses zero, and **not** relative to `index_price`; same `longs.payoffs - shorts.payoffs` sign-change `build_zone_curves()` finds internally for its own ±$2000 auto-zoom), `high_resistance_positive`/`low_support_positive` (Boolean — whether the *payoff value* at that intersection, not its price, sits above or below the zero-payoff line; interpolated via `np.interp` at the crossing price against `longs_obj.payoffs`, which equals `shorts_obj.payoffs` at a true crossing — drives the `+`/`-` marker on the TradingView chart, see TradingView Chart Notes), `gamma_band` (average of 4 gamma extrema — Buyer Call Gamma (BCG), Buyer Put Gamma (BPG), Seller Call Gamma (SCG), Seller Put Gamma (SPG) — the same numbers the `/<instrument>/zones` PNG page's info overlay shows, sourced from `options.per_leg_greeks()` against this same instrument's `trades`/`STs`), `delta_band` (average of 4 delta-saturation prices, one per leg, also from `options.per_leg_greeks()` — not derived from gamma, since portfolio delta and portfolio gamma are numerically related but conceptually distinct measures here), `high_zone_min`/`high_zone_max`, `low_zone_min`/`low_zone_max`, `middle_zone_min`/`middle_zone_max` (same definitions as `options.zone_summary()`'s `high_zone`/`low_zone`/`middle_zone` — see Zones chart above and `options.py` below — stored as a `_min`/`_max` Float pair per zone since a single Float can't hold a range; `high_zone`/`low_zone` default `0.0`/`0.0` when neither curve ever crosses zero, same convention `high_resistance`/`low_support` use, while `middle_zone` is always defined), `seller_max_profit`/`buyer_max_loss` (see `dankbit.forecast3.snapshot`'s `bml`/`smp` fields — price where the Shorts payoff curve peaks / where the Longs curve bottoms out, against this same `longs_obj`/`shorts_obj`; same values `options.zone_summary()`'s `seller_max_profit`/`buyer_max_loss` show on the `/<instrument>/zones` PNG page's info overlay, not exposed via `/api/bands/<asset>` — model-only, feeds the `/api/zones-box/<asset>` teal box instead, see TradingView Chart Notes), and the 32 individual per-leg gamma/delta/theta/vega price + Abs fields (`bcg_price`/`bcg_abs`/`bpg_price`/`bpg_abs`/`scg_price`/`scg_abs`/`spg_price`/`spg_abs`, and the BCD/BPD/SCD/SPD, BCT/BPT/SCT/SPT, BCV/BPV/SCV/SPV equivalents — see `_PER_LEG_GREEK_FIELDS`) behind `gamma_band`/`delta_band`'s averages — the exact same numbers, same field names, and same scaling (`*_abs` is `abs(value)/scale` — 1e6 gamma, 10 delta, 1e4 theta, 100 vega — matching the `/<instrument>/zones` PNG page's own Abs. lines, not rounded further) as `dankbit.forecast3.snapshot`'s own `bcg_price`/`bcg_abs`/etc. fields, sourced from the same `forecast3_lib.per_leg_greeks()` this model calls directly — this model, the `/<instrument>/zones` PNG page, and the Thales Forecast engine can never quietly disagree on any of these 32 numbers for the same trades, all three ultimately calling the same `options.per_leg_greeks()`. `_compute_asset(asset, expiry_index=0, hours=None)` is the shared computation core (the `expiry_index`-th soonest active expiry — 0 is nearest, 1 is the one after that, via a raw `SELECT DISTINCT expiration` query so a single expiry's 100k+ trades can't crowd out the next one — trades since 00:00 UTC through now for that expiry by default, or the trailing `hours` hours through now when given — see `get_box`'s `hours` override below, `options.build_zone_curves()`, then `forecast3_lib.per_leg_greeks()` for the 32 per-leg fields (`gamma_band`/`delta_band` are simply the average of 4 of those), `high_zone_min`/`high_zone_max`/`low_zone_min`/`low_zone_max` built from the same `short_crossings`/`long_crossings` also used for the 4 raw zero-crossing box boundaries (rather than calling `options.zone_summary()` again and re-finding the same crossings), `middle_zone_min`/`middle_zone_max` from `seller_max_profit`/`buyer_max_loss`, then `high_resistance`/`low_support`/`seller_max_profit`/`buyer_max_loss`, that expiry's own `instrument` string and `expiration` datetime, plus 4 zero-crossing box boundaries — each curve's own highest/lowest zero-crossing, also not relative to `index_price` (same principle as `high_resistance`/`low_support`; see below) — returns `None` if there's nothing computable (missing index price/expiry at that index/since-midnight trades) rather than writing a row that would silently look like real data (e.g. the configured price-range floor). Persisted as a side effect of `get_box_n()` (below), whose **only** caller, for every tracked `expiry_index` including 0, is the 4-hourly cron, `compute_snapshot()` — there is no browser-triggered live path for any of them, deliberately: `/api/zones-box/<asset>` (the "Zones" checkbox) computes the nearest expiry's box boundaries fresh on every request via `get_box()` -> `_compute_asset()` directly, without ever routing through `get_box_n()`, and the "Bands" checkbox's `refreshBands()` only ever reads already-persisted rows (`/api/bands/<asset>`), never computing new ones. So opening the chart, toggling either checkbox, or switching timeframe can never cause new bands to be computed — only `compute_snapshot()` does, for both BTC and ETH, every tracked `expiry_index` (see `TRACKED_EXPIRY_COUNT` below) — and it never touches the TradingView horizontal price lines (delta=0, gamma peak/bottom) — only this model.
  - `_persist_extrema(data)` — upserts (search by `instrument`, `write()` if found else `create()`) the one record for `data['instrument']`, storing only the historical-line fields (`computed_at`/`index_price`/`high_resistance`/`low_support`/`high_resistance_positive`/`low_support_positive`/`gamma_band`/`delta_band`/`high_zone_min`/`high_zone_max`/`low_zone_min`/`low_zone_max`/`middle_zone_min`/`middle_zone_max`/`seller_max_profit`/`buyer_max_loss`, plus the 32 per-leg gamma/delta/theta/vega price+Abs fields listed in `_PER_LEG_GREEK_FIELDS`) — `computed_at` is refreshed on every upsert (unlike a plain creation timestamp), so it reflects the moment this specific `_compute_asset()` call ran, not when the row was first created; the 4 zero-crossing box-boundary fields are never persisted. `self.sudo()` internally: `get_box_n()` (the only caller of this method) is itself only ever called by `compute_snapshot()`'s cron, not from any HTTP route — `get_box()` (which *is* reached from the `/api/zones-box/<asset>` route) deliberately calls `_compute_asset()` directly instead, bypassing `get_box_n()`/this method entirely, so no browser request can ever write here. `sudo()` is kept anyway as a defensive backstop (the cron's own `user_id` isn't guaranteed to have write access on `dankbit.bands`, which only grants `base.group_user` — see `ir.model.access.csv`) and mirrors how the rest of this codebase already elevates for writes that shouldn't depend on the caller's own permissions. Called from `get_box_n()` for every tracked `expiry_index`, so an instrument's row starts accumulating as soon as it's tracked at all — typically well before it becomes nearest. Either way, this is still enough to build a connected multi-expiry history: while an instrument is tracked, every poll refines its one row right up until it expires and rolls off the active list, at which point a *different* instrument takes over that slot and starts its own new row — the old row is simply never touched again, freezing at its last computed value, which is exactly the final point the chart needs for that expiry.
  - `TRACKED_EXPIRY_COUNT = 3` — how many active expiries (soonest-first, 0 = nearest) get a persisted bands row at all. Only `expiry_index` 0 (the nearest expiry) gets an actual drawn box on the TradingView chart (see Zones boxes above); `expiry_index` 1 and 2 still get tracked (feeding the High/Resistance-Low/Support and Gamma Band term-structure lines, which render whatever rows exist for the asset regardless of whether a box was ever drawn) but, like `expiry_index` 0, only via `get_box_n()`, and only ever called by the 4-hourly `compute_snapshot()` cron — there is no HTTP route or browser-triggered path that computes *any* tracked index, including the nearest one.
  - `get_box_n(asset, expiry_index)` — computation for any tracked `expiry_index`, computed fresh on every call and persisted via `_persist_extrema()`. Its **only** caller, for every `expiry_index` including 0, is `compute_snapshot()`'s 4-hourly cron (see below) — there is no HTTP route or browser-triggered path that reaches this method at all. `/api/zones-box/<asset>` (the one that actually renders the yellow/teal box, and runs live on every page view) goes through `get_box()` instead, which calls `_compute_asset()` directly rather than this method, precisely so a live page view never persists anything.
  - `get_box(asset, hours=None)` — called live by `/api/zones-box/<asset>` on every request; always computes via `_compute_asset()` directly (never via `get_box_n()`), so it **never persists**, for either the default since-00:00-UTC-through-now window or an explicit `hours` override (from `/api/zones-box/<asset>`'s optional `?hours=` query param — see TradingView chart's zones box trade-window radio toggle). The nearest expiry's `dankbit.bands` row (`expiry_index` 0) is therefore refreshed *only* by `compute_snapshot()`'s cron, exactly like `expiry_index` 1/2 — no browser action, for any `expiry_index`, ever writes to this model.
  - `nearest_expiry(asset)` — standalone cheap lookup (no curve-building), returning the nearest active expiry as a full Deribit-style instrument string (e.g. `"BTC-9JUL26"`, matching the `weekly_expiry`/`monthly_expiry`/`INSTRUMENT`/`MONTHLY_INST` convention every other expiry identifier in this addon uses — not just the bare day-string); shares `_distinct_expirations()` with `_compute_asset()` rather than duplicating the DISTINCT-expiration SQL. Called by `/api/nearest-expiry/<asset>` for the TradingView footer's `Nearest:` text.
- `dankbit.forecast3.snapshot` — one record per `(asset, bucket_start)` (a rolling 4h time bucket, see `BUCKET_HOURS`), supplying the Thales Forecast's Gamma-Band Consensus engine (see TradingView Chart Notes' Thales Forecast candles) with a real historical time series, since Dankbit — unlike Thales's manually-dated CSV rows — has no other place a "previous" observation of these levels is stored. `compute_and_persist(asset)` calls `dankbit.bands._compute_asset(asset, expiry_index=0)` **once** and reads everything off that single result — `index_price`/`computed_at`, `top`/`low` (= that call's own `high_zone_max`/`low_zone_min`, byte-for-byte the same "each curve's own highest/lowest zero-crossing" value, since `_compute_asset()` now persists that pair itself — see Models, `dankbit.bands`), `bml`/`smp` (= `buyer_max_loss`/`seller_max_profit`), and all 32 per-leg gamma/delta/theta/vega price+Abs fields (`_PER_LEG_GREEK_FIELDS`, spread via a dict comprehension) — rather than reimplementing `options.zone_summary()`/re-querying `dankbit.trade`/re-calling `options.build_zone_curves()` and `forecast3_lib.per_leg_greeks()` a second time. This avoids a second identical trades-query-plus-curve-build for the same asset/expiry/instant on every call, and means the Thales Forecast's band and per-leg Greeks can never quietly drift from what's actually drawn on screen elsewhere in the app — both now come from the exact same `_compute_asset()` result. The whole row is upserted into the current bucket, continuously refining it until the bucket rolls over — the same pattern `_persist_extrema()` uses per-instrument. `recent_history(asset, limit)` returns the most recent rows newest-first for that slope math — **no cron fallback**: history only accumulates as a side effect of `compute_and_persist()` being called live, and a gap just widens the Gamma-Band Consensus engine's slope window (see `gamma_band_consensus()`'s real-elapsed-hours normalization) rather than breaking anything.

- `options.py` — `OptionStrat` class. Accumulates legs, then `plot()` returns a matplotlib `Figure`; `plot_zones(longs_curve, shorts_curve, index_price, title)` renders the Longs/Shorts zones chart from two separately-accumulated `OptionStrat` instances — `title` is accepted (callers pass `"Zones"`) but does not appear in the rendered title, which is always just `"{name} | {now} UTC"`. Uses Agg backend; do not import `pyplot` here. `build_zone_curves(instrument_name, index_price, trades, from_price, to_price, steps)` is the shared Longs/Shorts curve-building + crossing-based ±$2000 zoom logic (module-level function, not a method) — used by both `chart_png_zones` and `dankbit.bands._compute_asset`, so the PNG chart, the persisted bands record, and the live zones-box lookup can never compute different values for the same trades. `find_zero_crossings(STs, curve)` is the shared zero-crossing finder (linear interpolation on each sign change) used by `build_zone_curves`'s own Longs-vs-Shorts crossing and by `_compute_asset`'s per-curve box-boundary crossings. `zone_summary(STs, longs_curve, shorts_curve)` computes `seller_max_profit`/`buyer_max_loss`, the `high_zone`/`low_zone`/`middle_zone` boundaries, and `high_resistance`/`low_support` — from an already-built pair of curves, used by `chart_png_zones` to populate the zones-PNG page's top-left info overlay (see Zones chart above). Current price is deliberately not a parameter of this function at all: `high_zone`/`low_zone` are each curve's own highest (`high_zone`) or lowest (`low_zone`) zero-crossing — a zone forms from a single curve's crossing alone (returned as a degenerate `(price, price)` pair, rendered by `chart_png_zones` as one number rather than a range) if the other curve never crosses zero; `None` only when neither curve crosses zero at all. `middle_zone` is the min/max of `seller_max_profit`/`buyer_max_loss` instead — always defined (those two are argmax/argmin over the full curve, not zero-crossings), same degenerate-single-price rendering when they coincide. `high_resistance`/`low_support` are simply the highest/lowest Longs-vs-Shorts crossing, same reasoning (see Models, `dankbit.bands`). A separate, differently-shaped computation in `dankbit.bands._compute_asset` (needs the 4 raw box fields individually rather than a summarized pair, and doesn't need `seller_max_profit`/`buyer_max_loss` at all since it doesn't persist them) duplicates the same box/intersection logic rather than sharing this function, but both apply the identical "ignore index price, use each curve's own extremes" rule. `delta_saturation_price(STs, trades, fraction, stop_at)` finds the interpolated price where `delta.portfolio_delta(STs, trades)` first reaches `fraction` of its own extreme value at the `stop_at` edge (`'min'`/`'max'`) of `STs` — the point where a leg's sigmoid-shaped delta curve stops curving and flattens into a straight line (deep enough ITM to "trade like synthetic stock"). Relative to the curve's own extreme, not an absolute delta value, since `portfolio_delta` sums `sign*amount*per-contract delta` across every matching trade — its scale reflects total traded size (can be in the hundreds), not a single option's `[-1, 1]` range, so a fixed absolute threshold would be crossed almost immediately near the start of the curve rather than at any deep-ITM point. Curves are monotonic per leg (see `delta.py`'s sign convention), so the `stop_at` edge value is genuinely the curve's extreme in this window and its sign is inherited automatically — no separate sign parameter needed. Falls back to the `STs` edge on `stop_at` if the extreme is `0` or the curve never reaches that fraction within the window. Shared by `dankbit.bands` (`delta_band`) and the `/<instrument>/lp,lc,sp,sc` single-leg routes' green marker line, so the two can never disagree on where this point is. `DELTA_SATURATION_FRACTION = 0.9` is the single source of truth for that fraction — `main.py`'s `ChartController.DELTA_SATURATION_FRACTION` is just `options.DELTA_SATURATION_FRACTION` re-exposed as a class attribute (for `_LEG_ROUTES`). `per_leg_greeks(STs, trades, r=0.0)` computes, for each of the 4 legs split out of `trades` (`long_call`/`long_put`/`short_call`/`short_put`), where that leg's own portfolio gamma/theta/vega curve peaks or bottoms (long positions: gamma/vega peak via `argmax`, theta troughs via `argmin` since long theta is negative decay; short positions: gamma/vega bottom via `argmin`, theta peaks via `argmax` since short theta is positive decay gain) and the price/value where its portfolio delta first saturates (via `delta_saturation_price`, calls at the high-`S` edge, puts at the low-`S` edge, independent of long/short) — returning `{leg_name: {"trades", "gamma_price", "gamma_value", "delta_price", "delta_value", "theta_price", "theta_value", "vega_price", "vega_value"}}`, raw/unscaled. This is the single source of truth for this computation — `chart_png_zones`, `dankbit.bands._compute_asset`'s `gamma_band`/`delta_band` computation, and `forecast3.per_leg_greeks` all call this one function and apply only their own display scaling on top (`chart_png_zones`'s Value lines divide gamma/delta/theta/vega by `1e6`/`10`/`1e4`/`100`; `forecast3.per_leg_greeks` does the same for its `*_abs` fields; `dankbit.bands` averages the 4 gamma/delta prices into `gamma_band`/`delta_band` with no scaling at all).

## WebSocket Service (`dankbit_ws_service/`)

## TradingView Chart Notes

- Candles sourced from Deribit perpetual futures (`BTC-PERPETUAL`, `ETH-PERPETUAL`) via `/api/klines/<asset>` proxy (avoids CORS). Uses `get_tradingview_chart_data` endpoint. Deribit returns oldest-first parallel arrays (`ticks`, `open`, `high`, `low`, `close`); proxy zips and reverses to newest-first `{t,o,h,l,c}` format (t in ms). **Deribit has no native 4h resolution** (`resolution=240` errors with "unsupported resolution" — valid values are 1, 3, 5, 10, 15, 30, 60, 120, 180, 360, 720, `1D`); the `4h` interval fetches native 1h candles and aggregates every 4 into one OHLC bar server-side, bucketed by `t // (4*3600*1000)` (timestamp-based, not positional) so buckets stay calendar-aligned to 00:00 UTC regardless of the fetch window's boundaries.
- Candles refresh every 5 seconds; delta=0 lines refresh on `dankbit.refresh_interval`
- Chart timezone is auto-detected from the browser (`Intl.DateTimeFormat().resolvedOptions().timeZone`, falling back to UTC if detection throws), not hardcoded — `tzOffset` is that zone's current UTC offset in seconds (DST-aware, computed against "now"). Lightweight Charts has no native IANA-timezone support and always renders epoch seconds as UTC wall-clock time, so every timestamp fed into the chart (candles, all price-line/series time fields, the zones-box's `now - 10 days` left edge) is pre-shifted by `tzOffset` before being handed to the library, and `localization.timeFormatter` formats with `timeZone: 'UTC'` so the shifted epoch reads as the viewer's real local time.
- Timeframe buttons: 1h / 4h (default) / 1d — visible windows: 1h=30d, 4h=15d, 1d=365d; Deribit resolution: 1h→60, 1d→1D, 4h→aggregated from 1h (see above). `loadCandles()` always fetches a flat 500 bars regardless of timeframe, and Lightweight Charts renders candle body width as a fixed proportion of `barSpacing`; 15d keeps `barSpacing` comfortably in the ~17px range so each candle's body-to-gap ratio doesn't look sparse/"half-empty". `applyVisibleRange()`'s right edge is `now + FUTURE_MARGIN_DAYS` (5 days, fixed regardless of timeframe), leaving room to see future price-line/box levels laid out ahead of the latest candle.
- Ruler tool: click twice on chart to measure price-to-price percentage
- **"Lines" checkbox** (Delta Chart only — the Gamma Chart has no equivalent checkbox; next to Ruler, **unchecked by default**) — hides every horizontal price-line indicator (Weekly/Monthly/All/Daily/Daily+1 delta=0 lines) via a client-side `SHOW_ALL_LINES` flag. Unlike the config-driven `SHOW_DAILY_LINES`/`SHOW_WEEKLY_LINES`/`SHOW_MONTHLY_LINES` toggles (which still fetch even when hidden, just skip drawing), leaving this unchecked stops each affected `refresh*()` function from fetching at all — to actually reduce server load, not just hide the client-side result; check it to turn every one of them on. `refreshMonthlyDeltaZero()` is the one exception, same as with `MINIMAL_CHART`: it always keeps polling regardless, since its `trade_count` feeds the footer. `onToggleLines()` clears every line immediately when switched off (each function's own `clearLines()` call is skipped along with its fetch, so the toggle handler clears them directly) and re-fetches everything immediately when switched back on, rather than waiting for the next poll either way.
- **"Zones" checkbox** (Delta Chart only — this indicator is already suppressed on `/my/<asset>` via `MINIMAL_CHART`, so this control has a `t-if="show_gamma_point == 'false'"` and isn't shown there at all; right after "Lines", **checked by default**) — controls the yellow top/bottom zero-crossing boxes (High Zone/Low Zone) and the teal Seller Max Profit/Buyer Max Loss box (Middle Zone), i.e. everything `refreshZonesBoxes()`/`clearZonesBoxes()` draw. Same server-load-reducing pattern as every other checkbox here, via a client-side `SHOW_ZONES_BOXES` flag: unchecking it stops `refreshZonesBoxes()` from calling `/api/zones-box/<asset>` at all, purely to reduce server load — it has no effect on `dankbit.bands` either way, since that endpoint never persists anything regardless of whether it's called (see Models, `get_box`). The 4-hourly `compute_snapshot()` cron is the sole path that keeps the nearest expiry's (`expiry_index` 0) `dankbit.bands` row fresh. `onToggleZonesBoxes()` clears the boxes immediately when switched off and re-fetches immediately when switched back on.
- **"Thales Forecast" checkbox** (Delta Chart only — gated by `t-if="show_gamma_point == 'false'"`, so it does **not** render on `/my/<asset>`; next to Ruler, **unchecked by default**) — toggles the Thales Forecast candles (see TradingView Chart Notes above) via a client-side `SHOW_FORECAST3` flag. Same server-load-reducing pattern as every other checkbox here: `refreshForecast3()` skips the `/api/forecast3/<asset>` fetch entirely while unchecked rather than just hiding the result; `onToggleForecast3()` clears the forecast series immediately when switched off and fetches immediately when switched back on. `updateForecastVisibility()` (called from `setTf()`/`init()` on both charts) null-checks `#forecast3-toggle-wrap` before touching it, since the Gamma Chart no longer has that element at all.
- **"Gamma Tops" checkbox** (Gamma Chart only, `/my/<asset>` — next to Ruler, `t-if="show_gamma_point == 'true'"`; the only checkbox on this page, unchecked by default) — replaces the earlier separate "Strike Gamma"/"Strike Gamma Nearest"/"Strike Gamma Weekly"/"Strike Gamma Monthly" checkboxes (one line per strike that had traded, in each of 4 scopes) *and* the earlier separate "Nearest"/"Weekly"/"Monthly"/"Universal" gamma-point scope checkboxes (every local gamma peak/bottom, via `find_gamma_peaks`/`find_gamma_bottoms` over a synthetic price grid) with a single checkbox that instead draws exactly one line per scope: `ChartController._gamma_by_strike()` nets each instrument's buy/sell trade flow into a single signed position and clamps it to real open interest (`dankbit.trade.get_open_interest_by_currency()`), then evaluates combined portfolio dollar-gamma at every distinct real strike price — and the client (`gammaTop()` in `dankbit_templates.xml`) keeps only the single highest-*positive*-gamma strike from each scope's dataset (skipping a scope with no strikes at all, or whose highest gamma is `<= 0`), instead of drawing every strike. "All" is sourced with no expiry cutoff (`/api/gamma-by-strike/<asset>`); "Nearest"/"Weekly"/"Monthly" share one generic `/api/gamma-by-strike-until/<instrument>` route — Nearest first resolves that asset's own nearest active expiry client-side via `/api/nearest-expiry/<asset>` and passes that in as the cutoff instrument, while Weekly/Monthly pass `INSTRUMENT`/`MONTHLY_INST` directly. Checking the box fires all 4 fetches at once. Each resulting line (`drawGammaTopLine()`) is solid black, `lineWidth: 2`, titled `'Γ Top ' + label + ' +' + round(gamma/1e6)` (`label` one of `Nearest`/`Weekly`/`Monthly`/`All`), distinguished from each other only by that title text rather than a per-scope color. `SHOW_GAMMA_TOPS` gates `refreshGammaTops()`'s fetches, same server-load-reducing pattern as every other checkbox on this page.
- **"Bands" checkbox** (Delta Chart only — this indicator is already suppressed on `/my/<asset>` via `MINIMAL_CHART`, so this control has a `t-if="show_gamma_point == 'false'"` and isn't shown there at all; next to "Zones", **unchecked by default**) — controls only the High/Resistance-Low/Support + Gamma Band lines (green/red/dashed violet), deliberately **not** the yellow/teal zones boxes, which have their own separate "Zones" checkbox instead (see above — `refreshZonesBoxes()` is gated by `SHOW_ZONES_BOXES`, not by this flag). Purely a display toggle: `refreshBands()` (gated by a client-side `SHOW_BANDS` flag) only ever reads already-persisted `dankbit.bands` rows via `/api/bands/<asset>` — it never triggers a new computation, for any `expiry_index`, whether checked, unchecked, or polled repeatedly. Every tracked expiry's row (see Models, `TRACKED_EXPIRY_COUNT`), including the nearest one, is computed and refreshed *solely* by the 4-hourly `compute_snapshot()` cron — the separate "Zones" checkbox's `/api/zones-box` polling computes the nearest expiry's box boundaries fresh for its own rendering but never persists them (see Models, `get_box`) — so neither checkbox, nor any other browser action, can affect when new bands get computed. `clearBandsLines()` clears the High/Resistance-Low/Support-Gamma Band lines immediately when switched off; `onToggleBands()` re-fetches (reads) them immediately when switched back on.
- **Zones box trade-window radio toggle** (Delta Chart only, after "Thales Forecast" — two radios, "00:00 UTC" checked by default and "`<N>`h ago", `N` baked in from `dankbit.zones_box_window_hours` at page render via `<t t-esc="zones_box_window_hours"/>` in the label) — switches the trade window the yellow/teal zones boxes are computed from between the default since-00:00-UTC-through-now window and a trailing-`N`-hours one. Backed by a client-side `ZONES_BOX_WINDOW_MODE` (`'midnight'`/`'hours'`); when `'hours'`, `refreshZonesBoxes()` appends `?hours=<N>` to `/api/zones-box/<asset>` (`zones_box_json`/`get_box()`/`_compute_asset()` all gained an optional `hours` param for this — passing it overrides the domain's default `deribit_ts` lower bound with `now - hours` instead of today's UTC midnight); `get_box()` **never persists** regardless of this toggle (it always calls `_compute_asset()` directly, never `get_box_n()`/`_persist_extrema()` — see Models), so switching this never corrupts the shared per-expiry history other viewers/expiries' term-structure lines depend on, and the 4-hourly `compute_snapshot()` cron is completely unaffected either way. `onZonesBoxWindowChange()` re-fetches immediately on change rather than waiting for the next `ZONES_BOX_REFRESH` poll.

## Odoo Gotchas

- **Manifest licence field:** Odoo only accepts a fixed set of strings. Use `"Other OSI approved licence"` (British spelling) for MIT — `"MIT"` alone will fail validation.
- **Every route in this addon is `auth="user"` — login required:** a logged-out request to any URL here — PNG chart, JSON API, or TradingView page — is redirected to `/web/login?redirect=<path>` instead of rendering/responding. This includes the JSON APIs the TradingView pages' own client-side JS calls (`gamma-by-strike`, `gamma-by-strike-until`, `bands`, `klines`, `zones-box`, etc.) — those rely on the browser's session cookie being sent automatically on same-origin `fetch()` calls once the user is logged in to view the page at all, so no separate handling was needed there. Testing this matters: `odoo shell` runs as superuser (no check at all), and a browser tab already logged into the backend renders every route fine regardless of this restriction — only a cookie-less request (e.g. plain `curl`) actually exercises the login redirect.
- **Model writes reached from a controller still need `.sudo()` for non-internal sessions:** every model in this addon except `dankbit.trade` only has `ir.model.access.csv` rules for `base.group_user` (internal users) — no portal grant. `auth="user"` guarantees a logged-in session, but not necessarily an internal one — a portal user would still lack write access. Reading via raw SQL (`request.env.cr.execute(...)`) bypasses this entirely, but any `self.search()`/`self.create()`/`record.write()` call reached from a controller needs an explicit `.sudo()`, or it 403s for a logged-in-but-non-internal session. This is easy to miss in testing since `odoo shell` and an already-logged-in backend browser tab both run as an internal/admin user, masking the gap. See `dankbit.bands._persist_extrema()` for the fix.

## Odoo Module Reload

Python changes to `my_addons/` take effect after `docker compose restart web`. XML/view changes and new model fields require upgrading the module:

```bash
docker compose exec web odoo -d <db_name> -u dankbit --stop-after-init
docker compose restart web
```
