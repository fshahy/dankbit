# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Running the Stack

```bash
# Start all services
docker compose up -d

# Rebuild after code changes
docker compose up -d --build

# Rebuild a single service (e.g. after editing dankbit_ws_batch.py)
docker compose up -d --build dankbit_ws

# Tail logs
docker compose logs -f
docker compose logs -f dankbit_ws

# Restart just the Odoo addon (no rebuild needed for Python changes to my_addons)
docker compose restart web
```

After first startup, install the Dankbit addon via Odoo UI: Apps → search "Dankbit" → Install.

**Required `.env` variables:**
```
DANKBIT_POSTGRES_DB=<db_name>   # any name; db1 is the development default
DERIBIT_KEY=<key>
DERIBIT_SECRET=<secret>
```
The `UID` and `GID` env vars are also needed so the `web` container writes files as the host user (dev default: `1000`/`1000`). `docker-compose.yml` sets `user: "${UID}:${GID}"` on the `web` service and `chown`s the Odoo data dir to that UID/GID before starting — the values must match whichever host user owns the bind-mounted directories, so Odoo's writes (filestore, logs) land with an owner the host can actually manage. Check with `ls -ln <bind-mounted dir>` if unsure. Setting it to `0`/`0` (root) is correct only when the production host directory itself is root-owned — it also means Odoo runs as root inside the container, a broader security surface than an unprivileged UID.

> **Production note:** `config/odoo.conf` contains `dbfilter = ^db1$` — update this to match the actual database name.
> **Production note:** production's `.env` uses `UID=0`/`GID=0` (root), unlike the dev default of `1000`/`1000` — see above for when that's the correct choice.

## Architecture

Two Docker services talk to a shared PostgreSQL database:

```
Deribit WS API ──► dankbit_ws (Python asyncio) ──► PostgreSQL
                                                        │
                   Deribit REST API ◄── Odoo cron       │
                                                        ▼
                                               Odoo 18 (web:8069)
                                                        │
                               ┌───────────────────────┤
                               │                       │
                        PNG charts                TradingView
                  /BTC-<EXPIRY>, /i/BTC-<EXPIRY>  /chart/BTC
```

**Trade data enters two ways:**
1. **WebSocket (primary):** `dankbit_ws_service/dankbit_ws_batch.py` — connects to Deribit, authenticates, fetches all BTC+ETH option instruments, subscribes to `trades.<instrument>.raw` channels in chunks of 400, inserts rows directly into PostgreSQL with `ON CONFLICT IGNORE` on `deribit_trade_identifier`.
2. **REST backfill:** `Trade.get_last_trades()` — runs every minute via cron; ensures no trades were missed by paging through the last few days of Deribit REST API history.

**PNG chart rendering:** Per-expiry routes like `/BTC-7MAY26` aggregate all trades for that expiry, compute a Black-Scholes portfolio delta and dollar-gamma (GEX = Γ × S²) curve, and return a PNG via matplotlib's Agg backend (`Figure + FigureCanvas`, never `pyplot` — server-safe). Overlaid lines: gamma peaks and bottoms (dashed black), delta=0 crossings (solid — green for "supply" [delta negative below, positive above], red for "demand" [delta positive below, negative above]).

**Zones chart (`/<instrument>/zones`):** a distinct plot mode — separates that instrument's trades since 00:00 UTC into buy-side and sell-side (trade domain uses `("name", "=ilike", f"{instrument}-%")`, an anchored/left-prefix match — not a bare `ilike` substring match — so a query for one expiry can never pull in another instrument's trades), builds two independent `OptionStrat` payoff curves via `options.build_zone_curves()` (`long_call`/`long_put` for buys, `short_call`/`short_put` for sells — same payoff math as the combined-portfolio routes, just accumulated into two separate objects, plus the crossing-based auto-zoom below — same function `dankbit.zones.extrema` calls, see Models), and plots them together via `OptionStrat.plot_zones()`: green "Longs" curve, red "Shorts" curve, blue vertical line at index price — `plot_zones()` itself draws no text overlay. The `Short Max`/`Long Min`/`Top Box`/`Bottom Box` info (previously drawn inside the PNG via `ax.text()`) is now rendered as page HTML instead, alongside further per-expiry stats: `chart_png_zones` calls `options.zone_summary(STs, longs_curve, shorts_curve)` (module-level function, same extrema/box-boundary definitions as `dankbit.zones.extrema` — see Models) for `Short Max`/`Long Min`/`Top Box`/`Bottom Box`/`Top Intersection`/`Bottom Intersection`. Current price is not a factor in any of these: `top_box`/`bottom_box` are each curve's own highest/lowest zero-crossing (the Shorts curve's and Longs curve's max feed `top_box`, their min feeds `bottom_box`) rather than "the nearest crossing above/below index price" — a box forms as soon as *either* curve has at least one crossing (a lone curve's crossing yields a degenerate same-price pair, rendered as a single price rather than a dashed range); only `None`/`"n/a"` if neither curve ever crosses zero. Then separately computes 4 gamma extrema restricted to the single nearest expiry among that route's own `trades` (mirroring `dankbit.zones.extrema`'s "next expiry only" restriction) via `gamma.portfolio_gamma()`: `Long Call Gamma Peak`/`Long Put Gamma Peak` (price where the long-call/put gamma curve peaks, `argmax`) and `Short Call Gamma Bottom`/`Short Put Gamma Bottom` (price where the short-call/put gamma curve bottoms out, `argmin` — short positions carry negative gamma, so the relevant extremum is a trough, not a peak). All of this is assembled into `zone_info_lines` (grouped into 4 blocks — price extrema; boxes; gamma peaks/bottoms; intersections — separated by blank spacer lines, a `" "` non-breaking space since a truly empty/whitespace-only `<div>` can collapse to zero height) and passed to the shared `dankbit_page` template's `zone_info_lines` context var, rendered as a fixed top-left overlay div (`.zone-info`, `user-select: text` so it's copyable, only shown when that var is present, so every other PNG route using `dankbit_page` is unaffected). A separate bottom-left annotation (added by `chart_png_zones` itself, still drawn inside the image via `ax.text()`, not moved) shows the long/short trade counts since 00:00 UTC. Restores a feature this repo had (and removed) in its early history (`d416bcd` "add zones" → removed in `6c293d6`), reimplemented against the current architecture rather than reusing the old pyplot-based code. **Auto-zoomed x-axis:** the configured `from_price`/`to_price` settings are only used for an initial wide pass to locate where the Longs/Shorts curves cross (linear interpolation on the sign change of `longs.payoffs - shorts.payoffs`, same technique as the delta=0 crossing finders); the curves are then rebuilt over `[leftmost_crossing - margin_below, rightmost_crossing + margin_above]` at the same `steps` resolution, so the rendered chart is tightly framed around the relevant region instead of the full (much wider) configured range. Margin is asset-dependent: BTC uses ±$2000; ETH uses ±$100 (ETH's much smaller price scale made the ±$2000 BTC margin blow out the auto-zoom). Falls back to the original wide range if the curves never cross.

**Single-leg delta/gamma routes (`/<instrument>/lp`, `/lc`, `/sp`, `/sc`):** four routes, one per leg type (long put, long call, short put, short call) — `_chart_png_single_leg(instrument, leg_key)` is the shared implementation, dispatched via `_LEG_ROUTES[leg_key]` (`direction`/`option_type` domain filter, which `OptionStrat` leg method accumulates trades, route label, and delta-saturation `fraction`/`side` — see below). Same domain convention as `chart_png_zones` (anchored `=ilike` instrument match, trades since 00:00 UTC), but restricted to that one direction+option_type combination and built over the **full configured price range** (`from_price`/`to_price`/`steps`), not the crossing-based zoomed window `/zones` uses — matching `chart_png_hours`/`chart_png_all`'s style instead, including their `r=0.05` risk-free rate for `delta.portfolio_delta`/`gamma.portfolio_gamma` (deliberately *not* `r=0.0`/zoomed-grid like `dankbit.zones.extrema`, a choice reverted after being tried the other way). `obj.plot()` renders the same delta+gamma dual-axis chart as those routes; the gamma peak/bottom + delta=0 crossing overlay is factored into `_annotate_gamma_delta_crossings(ax, STs, market_deltas, market_gammas)` (shared by all 4 routes, so they don't each carry their own copy of that block). A solid green `axvline` additionally marks the delta-saturation price from `options.delta_saturation_price()` — see `options.py` below — labeled with its price, drawn after the standard overlay.

**TradingView chart:** `/chart/BTC` and `/chart/ETH` serve a Lightweight Charts v4 page with live Deribit perpetual futures candles (proxied server-side) and price lines refreshed on `dankbit.refresh_interval`:
- Delta=0 lines (3 sets) — each a single `createPriceLine`, `lineWidth: 1`, `lineStyle: Solid`. Color is by crossing type, not by set: **black** ("S/D" in the legend) for "supply" (delta negative below, positive above), **red** ("Tie-out" in the legend) for "demand" (delta positive below, negative above — the important case). Set identity is shown via the line's `title` only. Every `createPriceLine` call across the chart (delta=0 sets, gamma peak/bottom, ruler) sets `axisLabelColor: 'black'` so the axis label box is always black regardless of the line's own color — only the label background is forced black, not the line stroke — **except** Daily (24H)/Daily+1 (24H) (below), whose axis label is `axisLabelColor: 'orange'` instead, a deliberate one-off deviation from the rest of the chart. `addLineSeries` (Net Call/Put Gamma) has no equivalent per-series label-color override in Lightweight Charts v4, so those labels still match their line color (blue/orange):
  - **Weekly** — sourced from `/api/delta-zero/<instrument>`
  - **Monthly** — sourced from `/api/delta-zero/<monthly-instrument>`
  - **All** — all active expiries; sourced from `/api/delta-zero-all/<asset>`
- Daily (24H) / Daily+1 (24H) delta=0 lines — two distinct `createPriceLine` sets, color-by-crossing-type like Weekly/Monthly/All (black for "supply" — delta negative below, positive above; red for "demand" — delta positive below, negative above), but with an **orange** axis label (`axisLabelColor: 'orange'`, unlike every other line on this chart — see above) and **no footer-legend entry**, titles `Daily (24H)`/`Daily+1 (24H)`, sourced from `/api/delta-zero-tomorrow/<asset>` and `/api/delta-zero-day-after-tomorrow/<asset>` respectively — both thin wrappers around the shared `_delta_zero_for_calendar_day(asset, days_ahead)` helper (`days_ahead=1`/`2`) so the two can never disagree on how a calendar-day expiry/trade-window is computed. Each targets the specific expiry landing `days_ahead` calendar days out (UTC) — computed directly as `today + N days` formatted to Deribit's no-leading-zero day convention (e.g. `7JUL26`), not "nearest active expiry" (`MIN(expiration) >= NOW()`), since the latter can still resolve to *today's* not-yet-happened expiry. Restricted to trades from the trailing 24h (`deribit_ts >= NOW() - INTERVAL '24 hours'`). Gated by `show_daily_lines` (see Settings fields below).
- Gamma peak/bottom lines: Weekly and Monthly, both lineWidth 1 — violet, sourced from `/api/gamma-levels/<instrument>`. Each peak/bottom entry is `{price, delta_positive}` — the endpoints compute `delta.portfolio_delta` over the same `STs`/`agg_trades` used for gamma, then `np.interp(px, STs, d_arr) > 0` at the extremum's price. Title suffix combines gamma sign (implicit from peak `∧` = gamma>0, bottom `∨` = gamma<0, guaranteed by `find_gamma_peaks`/`find_gamma_bottoms`'s threshold logic) with delta sign into standard options-Greeks labels: peak+`delta_positive` → `Long Call`, peak+not → `Long Put`, bottom+`delta_positive` → `Short Put`, bottom+not → `Short Call` (e.g. `Weekly ∧ Long Put`, `Monthly ∨ Short Call`) — gamma is always positive for long option exposure and negative for short, so this combination is a technically valid characterization of the net Greeks profile at that price. `drawGammaLines(data, lineStore, lineWidth, prefix, color)` takes an explicit `color` (defaults to `'violet'`).
- Zones Extrema lines (4) — `Top Intersection` (green, `#2e7d32`, solid), `Bottom Intersection` (red, `#c62828`, solid), `Gamma Band` (light violet, `#c9a0dc`, **dashed** — `LightweightCharts.LineStyle.Dashed`), and `Delta Band` (dark green, `#1b5e20`, **dotted** — `LightweightCharts.LineStyle.Dotted`), `addLineSeries` on the **default/main price scale** (no `priceScaleId` override) since these are plain BTC/ETH price levels, not dollar amounts — they render directly alongside the candles. Sourced from `/api/zones-extrema/<asset>`: one point per **instrument** stored in `dankbit.zones.extrema` (see Models — one continuously-refined-then-frozen record per expiry, not a time-series of polls), positioned at that instrument's own expiration time rather than a poll timestamp (there isn't one anymore) — so these are term-structure-style lines connecting each expiry's final value to the next expiry's, not a history of one expiry's value changing over wall-clock time. `Gamma Band` is the average of that expiry's 4 gamma extrema (Long Call/Put Gamma Peak, Short Call/Put Gamma Bottom — see Models, `dankbit.zones.extrema._compute_asset`'s `gamma_band`), the same 4 numbers the `/<instrument>/zones` PNG page's info overlay shows. `Delta Band` is the average of that expiry's 4 delta-saturation prices — the price where each leg's (long call/put, short call/put) delta curve reaches 90% of its own extreme value in the window and flattens into a straight line (deep enough ITM to "trade like synthetic stock") — computed via `options.delta_saturation_price()` (see `options.py`; relative to the curve's own extreme, not an absolute delta value, since `delta.portfolio_delta` sums `sign*amount*per-contract delta` across every matching trade and its scale depends on total traded size, not a fixed `[-1, 1]` range). The endpoint derives each point's `t` via a single grouped lookup against `dankbit_trade` (`SUBSTRING(name FROM '^[^-]+-[^-]+')` to recover the instrument prefix, `MIN(expiration)`) rather than parsing the instrument string's day-suffix and assuming a settlement hour — raw SQL, so it still finds archived (past-expiry) instruments' trades despite the ORM's implicit `active=True` filter, since those rows must keep contributing a fixed historical point. Connected point-to-point by the line series. Only shown on the **4h** timeframe — `refreshZonesExtrema()` clears all four series (plus the Top/Bottom Intersection markers, below) when `INTERVAL !== '4h'`, and `setTf()`/`init()` call it every timeframe switch. Top/Bottom Intersection points additionally carry a `+`/`-` `setMarkers()` label (via `top_intersection_positive`/`bottom_intersection_positive` — see Models) showing whether the payoff *at* that intersection sits above or below the zero-payoff line, since the crossing's x-price alone doesn't say — `size: 0` hides the marker's shape glyph so only the text renders, colored to match its own line.
- Zones boxes (2, plus a 3rd Short Max/Long Min box — see below) — filled rectangles (`#fbc02d`, 50% opacity fill, matching top+bottom border lines) marking the top and bottom zero-crossing zones for the **nearest** active expiry only — there is no second (blue) box for a later expiry any more (removed; see Models, `get_box_n`/`TRACKED_EXPIRY_COUNT` for how later expiries are still tracked, just without a box). The 4 box-boundary fields themselves are **not** read from stored history — sourced live from `/api/zones-box/<asset>`, which calls `dankbit.zones.extrema.get_box()` to compute fresh on every request via `_compute_asset()` (see Models: nothing reads box-boundary history, only the latest value is ever drawn). As a side effect, `get_box()` *does* upsert that instrument's zones-extrema record (`index_price`/`top_intersection`/`bottom_intersection`/`gamma_band`/`delta_band`/`short_max_price`/`long_min_price`, not the 4 zero-crossing box-boundary fields) on every call — this endpoint's own polling (at `dankbit.refresh_interval`) is the primary path feeding the Zones Extrema lines' per-expiry history, with a 15-minute cron as a fallback for when nobody's viewing the chart (see Models, `_persist_extrema`). The boxes' own visual state is never affected by the cron either way — box-boundary fields are recomputed fresh on whatever request (browser or cron) triggers `_compute_asset()`, but only the browser's request result is ever rendered; the cron's result is only ever persisted, never displayed. Each box is built from two `addBaselineSeries` (`aboveBoxSeries`/`belowBoxSeries` — a flat top-price line filled down to a `baseValue` bottom price) plus a plain `addLineSeries` for the bottom edge (`aboveBoxBottomLine`/`belowBoxBottomLine`), since `addBaselineSeries` only strokes its top edge. Top/bottom of each box are `max`/`min` of that side's two zero-crossing prices (`short_zero_above_price`/`long_zero_above_price` for the top box, `short_zero_below_price`/`long_zero_below_price` for the bottom box) — despite the field names, these are each curve's own highest (`*_above_price`) or lowest (`*_below_price`) zero-crossing, not "the nearest crossing above/below current index price"; names kept for backward compatibility (API/JS consumers), see Models, `_compute_asset`. `setBoxData()` (in `dankbit_templates.xml`) draws the full rectangle when both a top and bottom price exist and differ; if only one curve contributed a crossing, or both coincide (a zero-height box), it instead draws a single line at whichever price is available (reusing the box's own bottom-edge line series) rather than dropping it silently; only clears both when neither curve crosses zero at all (a `0.0` value on **both** sides). Time span is `[snapTo18(computed_at - 2 days), expiration]` — the right edge is that expiry's own real settlement time (Deribit's `expiration_timestamp`, e.g. 08:00 UTC / 10:00 CEST — read off `dankbit_trade.expiration` via `_compute_asset()`, not assumed/hardcoded), so the box now visibly runs all the way out to when the trades it's summarizing actually stop mattering, rather than stopping at "now". The left edge is 2 days back, snapped to 18:00 on that calendar day (`snapTo18()`) rather than keeping `computed_at`'s own time-of-day — anchored to the live lookup's own timestamp (recomputed — and thus re-anchored — on every poll, not fixed to a stored tick). Only shown on the **4h** timeframe, same gating as the Zones Extrema lines.
- Short Max / Long Min box (3rd box, teal) — same mechanism (`addBaselineSeries` + bottom-edge `addLineSeries`, `setBoxData()`'s single-line fallback when top/bottom coincide) and the exact same `fromTime`/`toTime` as the yellow box above, since both come from the same `/api/zones-box/<asset>` response for the nearest expiry — `smlmBoxSeries`/`smlmBoxBottomLine`. Top/bottom are `max`/`min` of `short_max_price` (price where the Shorts payoff curve peaks) and `long_min_price` (price where the Longs curve bottoms out) — restored model fields, see Models, `dankbit.zones.extrema`. No separate endpoint: `short_max_price`/`long_min_price` were already computed by `_compute_asset()` as a side effect of the same `get_box()` call the yellow box uses, just added to `/api/zones-box/<asset>`'s JSON payload.
- No footer legend — removed entirely (`#dz-legend` div and its CSS); all lines/series remain, just without a color-key legend.
- Footer status shows: `<weekly-expiry>  ·  <monthly-expiry>  ·  N trades  ·  Nearest: <expiry>`; trade count = all active trades up to and including monthly expiry (from monthly endpoint). The `Nearest:` segment is the same nearest-active-expiry the yellow zones boxes use (`dankbit.zones.extrema.nearest_expiry()`, a cheap standalone `SELECT DISTINCT expiration ... LIMIT 1` lookup, no curve-building) — shown regardless of timeframe (`refreshNearestExpiry()`, unconditional unlike the 4h-only boxes), via a shared `updateStatusLine()` that both it and `refreshMonthlyDeltaZero()` call so neither overwrites the other's half of the status text.

## Odoo Addon (`my_addons/dankbit/`)

The addon depends only on `website`. Key components:

**Models:**
- `dankbit.trade` — Core model. Fields map directly to Deribit trade fields. `strike` and `option_type` are computed from `name` (instrument name like `BTC-29NOV24-98000-P`). `days_to_expiry` is UTC-safe. `get_hours_to_expiry()` returns continuous time used in Black-Scholes.
- `dankbit.zones.extrema` — **One record per instrument** (e.g. `"BTC-10JUL26"`, unique-constrained on `instrument`), not per snapshot-in-time — deliberately has no timestamp field of its own (a row's position on the TradingView chart is that instrument's own expiration time, looked up from `dankbit_trade.expiration` when the API serves it — see below and TradingView Chart Notes' Zones Extrema lines). Fields: `asset`, `instrument`, `index_price`, `top_intersection`/`bottom_intersection` (the highest/lowest price where the Longs-vs-Shorts payoff curves cross each other — not where either crosses zero, and **not** relative to `index_price` — labels kept for backward compatibility, but with only one crossing that single point can land on either side of price by a trivial amount, which used to make the "other" field silently read `0.0` even though the plot clearly showed one real intersection; same `longs.payoffs - shorts.payoffs` sign-change `build_zone_curves()` finds internally for its own ±$2000 auto-zoom), `top_intersection_positive`/`bottom_intersection_positive` (Boolean — whether the *payoff value* at that intersection, not its price, sits above or below the zero-payoff line; interpolated via `np.interp` at the crossing price against `longs_obj.payoffs`, which equals `shorts_obj.payoffs` at a true crossing — drives the `+`/`-` marker on the TradingView chart, see TradingView Chart Notes), `gamma_band` (average of 4 gamma extrema — Long Call Gamma Peak, Long Put Gamma Peak, Short Call Gamma Bottom, Short Put Gamma Bottom — the same numbers the `/<instrument>/zones` PNG page's info overlay shows, computed here against this same instrument's `trades`/`STs` via `gamma.portfolio_gamma()`, `argmax` for the two Long peaks and `argmin` for the two Short bottoms since short positions carry negative gamma), `delta_band` (average of 4 delta-saturation prices, one per leg — see `options.delta_saturation_price()` under `options.py` — computed via `delta.portfolio_delta()` against the same `trades`/`STs`, not derived from gamma, since portfolio delta and portfolio gamma are numerically related but conceptually distinct measures here), `short_max_price`/`long_min_price` (price where the Shorts payoff curve peaks / where the Longs curve bottoms out, against this same `longs_obj`/`shorts_obj` — this model's original two fields, added in `8c59981`, repurposed away into `top_intersection`/`bottom_intersection` in `8da5a1a`, and since reintroduced as their own fields alongside those rather than replacing them again; same values `options.zone_summary()`'s `short_max_price`/`long_min_price` show on the `/<instrument>/zones` PNG page's info overlay, not exposed via `/api/zones-extrema/<asset>` — model-only, feeds the `/api/zones-box/<asset>` teal box instead, see TradingView Chart Notes). `_compute_asset(asset, expiry_index=0)` is the shared computation core (the `expiry_index`-th soonest active expiry — 0 is nearest, 1 is the one after that, via a raw `SELECT DISTINCT expiration` query so a single expiry's 100k+ trades can't crowd out the next one — trades from the trailing 24h for that expiry (`deribit_ts >= NOW() - INTERVAL '24 hours'`; **not** the same "since 00:00 UTC" window the `/<instrument>/zones` PNG route uses — this model's own window, independent of it), `options.build_zone_curves()`, then `top_intersection`/`bottom_intersection`/`gamma_band`/`delta_band`/`short_max_price`/`long_min_price`, that expiry's own `instrument` string and `expiration` datetime, plus 4 zero-crossing box boundaries — each curve's own highest/lowest zero-crossing, also not relative to `index_price` (same principle as `top_intersection`/`bottom_intersection`; see below) — returns `None` if there's nothing computable (missing index price/expiry at that index/trailing-24h trades) rather than writing a row that would silently look like real data (e.g. the configured price-range floor). Persisted as a side effect of `get_box_n()` (below), called by the live zones-box/zones-extrema-refresh polling that already happens on every TradingView chart page at `dankbit.refresh_interval`, and by a 15-minute fallback cron, `compute_snapshot()`, since live-polling-only would leave a gap for any instrument nobody happens to be viewing (a real missing point in the per-expiry history, not just a staler one). The cron calls the exact same `get_box_n()` the live endpoints call (for both BTC and ETH, every tracked `expiry_index` — see `TRACKED_EXPIRY_COUNT` below), so it can never compute or persist anything a live page view wouldn't have, and never touches the TradingView horizontal price lines (delta=0, gamma peak/bottom) — only this model.
  - `_persist_extrema(data)` — upserts (search by `instrument`, `write()` if found else `create()`) the one record for `data['instrument']`, storing only the historical-line fields (`index_price`/`top_intersection`/`bottom_intersection`/`top_intersection_positive`/`bottom_intersection_positive`/`gamma_band`/`delta_band`/`short_max_price`/`long_min_price`); the 4 zero-crossing box-boundary fields are never persisted. `self.sudo()` internally: `get_box()`/`get_box_n()` are `auth="public"` routes, so this otherwise runs as the anonymous public user, which has no access rights at all on `dankbit.zones.extrema` (`ir.model.access.csv` only grants `base.group_user`) — without the explicit sudo, every real (cookie-less) request 403s even though it works fine from an already-authenticated browser session or `odoo shell`, which is why this went unnoticed when `_persist_extrema` was first introduced. Called from `get_box_n()` for every tracked `expiry_index`, so an instrument's row starts accumulating as soon as it's tracked at all — typically well before it becomes nearest. Either way, this is still enough to build a connected multi-expiry history: while an instrument is tracked, every poll refines its one row right up until it expires and rolls off the active list, at which point a *different* instrument takes over that slot and starts its own new row — the old row is simply never touched again, freezing at its last computed value, which is exactly the final point the chart needs for that expiry.
  - `TRACKED_EXPIRY_COUNT = 3` — how many active expiries (soonest-first, 0 = nearest) get a persisted zones-extrema row at all. Only `expiry_index` 0 (the nearest expiry) gets an actual drawn box on the TradingView chart (see Zones boxes above — the 2nd, "next expiry" box was removed); `expiry_index` 1 and 2 still get tracked (feeding the Top/Bottom Intersection, Gamma Band, and Delta Band term-structure lines, which render whatever rows exist for the asset regardless of whether a box was ever drawn) but only via `get_box_n()`, not a dedicated box endpoint.
  - `get_box_n(asset, expiry_index)` — generic live computation for any tracked `expiry_index`, computed fresh on every call and persisted via `_persist_extrema()`. `get_box(asset)` (below) is a thin wrapper over `get_box_n(asset, 0)`, kept as its own method since `/api/zones-box/<asset>` is the one that actually renders the yellow (and teal) box. `expiry_index` 1 and 2 are called via `/api/zones-extrema-refresh/<asset>/<expiry_index>` instead — no box is drawn from that endpoint's response.
  - `get_box(asset)` — called live by `/api/zones-box/<asset>` on every request; returns `_compute_asset()`'s (nearest-expiry) result including the 4 zero-crossing box fields (**not persisted** — nothing reads box-boundary history, only the latest value is ever drawn, see TradingView chart's Zones boxes; `short_max_price`/`long_min_price` are also in this same result but, unlike the box fields, *are* persisted by `_persist_extrema()`), but does call `_persist_extrema()` on that same result first.
  - `nearest_expiry(asset)` — standalone cheap lookup (no curve-building), returning just the nearest active expiry as a Deribit-style day-string (e.g. `"9JUL26"`); shares `_distinct_expirations()` with `_compute_asset()` rather than duplicating the DISTINCT-expiration SQL. Called by `/api/nearest-expiry/<asset>` for the TradingView footer's `Nearest:` text.
- `res.config.settings` extension — Chart price ranges, refresh interval, Deribit cache TTL, weekly/monthly expiries for BTC and ETH.

**Controllers (`main.py`):**
- PNG routes: `/<instrument>`, `/<instrument>/<hours>`, `/i/<instrument>`, `/<instrument>/s` (slideshow of hours_list `[0, 4, 8, 12, 24]`), `/<asset>/weekly`, `/<asset>/monthly` (bookmarkable shortcuts — read `weekly_expiry`/`monthly_expiry` (or `eth_*` for ETH) from settings and delegate straight into `chart_png_until`, so they always render identically to `/i/<asset>-<expiry>` for whatever expiry is currently configured; 404 for assets other than BTC/ETH), `/<instrument>/zones` (Longs-vs-Shorts payoff zones for that instrument's trades since 00:00 UTC — see below), `/<instrument>/lp`, `/<instrument>/lc`, `/<instrument>/sp`, `/<instrument>/sc` (single-leg delta/gamma routes — see below)
- JSON API: `/api/delta-zero/<instrument>`, `/api/delta-zero-all/<asset>`, `/api/delta-zero-tomorrow/<asset>`, `/api/delta-zero-day-after-tomorrow/<asset>` (calendar day-N expiry, trailing-24h trades only), `/api/gamma-levels/<instrument>`, `/api/klines/<asset>`, `/api/zones-extrema/<asset>` (one entry per stored `dankbit.zones.extrema` instrument-row, `t` = that instrument's own expiration time derived from `dankbit_trade`, not a poll timestamp; `t`, `index_price`, `top_intersection`, `bottom_intersection`, `top_intersection_positive`, `bottom_intersection_positive`, `gamma_band`, `delta_band`), `/api/zones-box/<asset>` (live, non-persisted box fields — calls `get_box()` fresh on every request; `t`, `expiration` (that expiry's own settlement time, ms), `index_price`, `short_zero_above_price`, `long_zero_above_price`, `short_zero_below_price`, `long_zero_below_price`, `short_max_price`, `long_min_price` — the last two *are* persisted, as a side effect of the same `get_box()` call, unlike the 4 zero-crossing fields), `/api/zones-extrema-refresh/<asset>/<expiry_index>` (triggers `get_box_n(asset, expiry_index)` for `expiry_index` 1 upward — no box is drawn from its response, just `{asset, expiry_index, instrument}`; polled purely to keep those rows fresh while the chart is open, same as `/api/zones-box` does for `expiry_index` 0), `/api/nearest-expiry/<asset>` (cheap standalone lookup — `dankbit.zones.extrema.nearest_expiry()`, no curve-building; `expiry` e.g. `"9JUL26"`)
- TradingView page: `/chart/<asset>` (reads weekly expiry from settings, returns 404 for unconfigured assets)
- Key helpers:
  - `find_gamma_peaks(STs, gamma_curve, min_fraction)` — local maxima of the gamma curve
  - `find_gamma_bottoms(STs, gamma_curve, min_fraction)` — local minima of the gamma curve
- `delta.py` / `gamma.py` — Pure Black-Scholes functions. Dollar gamma is `Γ × S²`. No side effects.
- `options.py` — `OptionStrat` class. Accumulates legs, then `plot()` returns a matplotlib `Figure`; `plot_zones(longs_curve, shorts_curve, index_price, title)` renders the Longs/Shorts zones chart from two separately-accumulated `OptionStrat` instances — `title` is accepted (callers pass `"Zones"`) but no longer appears in the rendered title, which is always just `"{name} | {now} UTC"`. Uses Agg backend; do not import `pyplot` here. `build_zone_curves(instrument_name, index_price, trades, from_price, to_price, steps)` is the shared Longs/Shorts curve-building + crossing-based ±$2000 zoom logic (module-level function, not a method) — used by both `chart_png_zones` and `dankbit.zones.extrema._compute_asset`, so the PNG chart, the persisted zones-extrema record, and the live zones-box lookup can never compute different values for the same trades. `find_zero_crossings(STs, curve)` is the shared zero-crossing finder (linear interpolation on each sign change) used by `build_zone_curves`'s own Longs-vs-Shorts crossing and by `_compute_asset`'s per-curve box-boundary crossings. `zone_summary(STs, longs_curve, shorts_curve)` computes `short_max_price`/`long_min_price`, the top/bottom box boundaries, and `top_intersection`/`bottom_intersection` — from an already-built pair of curves, used by `chart_png_zones` to populate the zones-PNG page's top-left info overlay (see Zones chart above). Current price is deliberately not a parameter of this function at all: `top_box`/`bottom_box` are each curve's own highest (`top_box`) or lowest (`bottom_box`) zero-crossing — a box forms from a single curve's crossing alone (returned as a degenerate `(price, price)` pair, rendered by `chart_png_zones` as one number rather than a range) if the other curve never crosses zero; `None` only when neither curve crosses zero at all — and `top_intersection`/`bottom_intersection` are simply the highest/lowest Longs-vs-Shorts crossing, same reasoning (see Models, `dankbit.zones.extrema`). A separate, differently-shaped computation in `dankbit.zones.extrema._compute_asset` (needs the 4 raw box fields individually rather than a summarized pair, and doesn't need `short_max_price`/`long_min_price` at all since it no longer persists them) duplicates the same box/intersection logic rather than sharing this function, but both apply the identical "ignore index price, use each curve's own extremes" rule. `delta_saturation_price(STs, trades, fraction, stop_at)` finds the interpolated price where `delta.portfolio_delta(STs, trades)` first reaches `fraction` of its own extreme value at the `stop_at` edge (`'min'`/`'max'`) of `STs` — the point where a leg's sigmoid-shaped delta curve stops curving and flattens into a straight line (deep enough ITM to "trade like synthetic stock"). Relative to the curve's own extreme, not an absolute delta value, since `portfolio_delta` sums `sign*amount*per-contract delta` across every matching trade — its scale reflects total traded size (can be in the hundreds), not a single option's `[-1, 1]` range, so a fixed absolute threshold would be crossed almost immediately near the start of the curve rather than at any deep-ITM point. Curves are monotonic per leg (see `delta.py`'s sign convention), so the `stop_at` edge value is genuinely the curve's extreme in this window and its sign is inherited automatically — no separate sign parameter needed. Falls back to the `STs` edge on `stop_at` if the extreme is `0` or the curve never reaches that fraction within the window. Shared by `dankbit.zones.extrema` (`delta_band`) and the `/<instrument>/lp,lc,sp,sc` single-leg routes' green marker line, so the two can never disagree on where this point is.

**Settings fields (`res.config.settings`):**
- `from_price`, `to_price`, `steps` — BTC chart price range
- `eth_from_price`, `eth_to_price`, `eth_steps` — ETH chart price range
- `refresh_interval` — page auto-refresh interval (seconds)
- `deribit_timeout`, `deribit_cache_ttl` — Deribit API behaviour
- `weekly_expiry`, `monthly_expiry` — BTC expiry strings (e.g. `BTC-4JUL26`)
- `eth_weekly_expiry`, `eth_monthly_expiry` — ETH expiry strings (e.g. `ETH-4JUL26`)
- `show_daily_lines`, `show_weekly_lines`, `show_monthly_lines` — TradingView chart line visibility toggles (all default `True`). Each gates *every* line carrying that label: `show_weekly_lines`/`show_monthly_lines` control both the delta=0 crossing line and the gamma peak/bottom lines for that label; `show_daily_lines` controls both Daily-family lines: Daily (24H), Daily+1 (24H). The "All" delta=0 line set has no toggle — always shown. These 3 fields deliberately **don't** use the plain `config_parameter=` attribute like every other setting in this file: Odoo's generic config_parameter handling for `Boolean` fields treats a stored `False` the same as "delete the parameter" (`ir.config_parameter.set_param()` special-cases Python `False`/`None` as unset), so unchecking one and saving would silently revert to `default=True` on next read instead of persisting `False`. `ResConfigSettings.get_values()`/`set_values()` override this explicitly, storing/reading the literal strings `"True"`/`"False"` instead (a string doesn't trigger the delete-on-False special case). `chart_tv` reads the params the same way (`icp.get_param(..., default="True") == "True"`) and converts to the literal strings `"true"`/`"false"` when passing into the template (QWeb's `t-att-*` silently drops the attribute for a falsy Python value, so a real boolean can't round-trip through `data-show-daily`), and the template reads `cfg.showDaily`/`showWeekly`/`showMonthly` into `SHOW_DAILY_LINES`/`SHOW_WEEKLY_LINES`/`SHOW_MONTHLY_LINES` consts — settings are baked in at page render, so a change takes effect on next page load, not live.

**Scheduled crons (data/ir_cron.xml, all `active=False` by default — activate manually post-install):**
- Every minute: `get_last_trades()` — ensures no trades were missed (e.g. during brief downtime)
- Daily: `_delete_expired_trades()` — archives expired trades
- Every 15 minutes: `compute_snapshot()` (on `dankbit.zones.extrema`) — fallback for BTC's and ETH's current tracked instrument rows (every `expiry_index` up to `TRACKED_EXPIRY_COUNT`), calling the same `get_box_n()` the live `/api/zones-box`/`/api/zones-extrema-refresh` polling calls (see Models, `_persist_extrema`) — a safety net for when nobody's actually viewing the TradingView chart, not the primary update path.

**Caching:** `trade.py` uses a module-level `_DERIBIT_CACHE` dict (key → `{ts, value}`) for Deribit index price and instrument lookups. TTL is configurable via settings (`deribit_cache_ttl`, default 300s).

**Backend menu:** `Dankbit` root menu (`trade_views.xml`) with two items — `Trades` (`dankbit.trade` list/form) and `Zones Extrema` (`zones_extrema_views.xml`, `dankbit.zones.extrema` list/form/search with BTC/ETH filters).

## WebSocket Service (`dankbit_ws_service/`)

Single file: `dankbit_ws_batch.py`. Authenticates with `DERIBIT_KEY`/`DERIBIT_SECRET` env vars using `client_credentials` grant before fetching instruments. Reconnects every 3 seconds on any failure. Rate-limit (`over_limit`) errors trigger a 0.5s sleep and retry.

The DB connection uses `autocommit=True` — no explicit transaction management needed.

## TradingView Chart Notes

- Candles sourced from Deribit perpetual futures (`BTC-PERPETUAL`, `ETH-PERPETUAL`) via `/api/klines/<asset>` proxy (avoids CORS). Uses `get_tradingview_chart_data` endpoint. Deribit returns oldest-first parallel arrays (`ticks`, `open`, `high`, `low`, `close`); proxy zips and reverses to newest-first `{t,o,h,l,c}` format (t in ms). **Deribit has no native 4h resolution** (`resolution=240` errors with "unsupported resolution" — valid values are 1, 3, 5, 10, 15, 30, 60, 120, 180, 360, 720, `1D`); the `4h` interval fetches native 1h candles and aggregates every 4 into one OHLC bar server-side, bucketed by `t // (4*3600*1000)` (timestamp-based, not positional) so buckets stay calendar-aligned to 00:00 UTC regardless of the fetch window's boundaries.
- Candles refresh every 5 seconds; delta=0 lines refresh on `dankbit.refresh_interval`
- Berlin timezone applied via `berlinOffset` computed from `Intl` API
- Timeframe buttons: 1h / 4h (default) / 1d — visible windows: 1h=30d, 4h=15d, 1d=365d; Deribit resolution: 1h→60, 1d→1D, 4h→aggregated from 1h (see above). The 4h window was deliberately narrowed from an initial 90d/30d — `loadCandles()` always fetches a flat 500 bars regardless of timeframe, and Lightweight Charts renders candle body width as a fixed proportion of `barSpacing`; cramming too many bars into one screen (as 90d did, ~3.2px/bar) made each candle's body-to-gap ratio look like a sparse, "half-empty" chart. 15d keeps `barSpacing` comfortably in the ~17px range. `applyVisibleRange()`'s right edge is `now + FUTURE_MARGIN_DAYS` (5 days, fixed regardless of timeframe) rather than a proportion of the visible window — a proportional margin (the original `days * 0.03`) left only ~10 hours of empty space past the current candle on the 4h view, making the chart look locked to "now" instead of leaving room to see future price-line/box levels laid out ahead of the latest candle.
- Ruler tool: click twice on chart to measure price-to-price percentage
- Crosshair price label (`crosshair.horzLine.labelBackgroundColor`) is set to blue — the library default (`#131722`, near-black) was indistinguishable from the black `axisLabelColor` used on every indicator price line
- Do **not** change `title:` values in `createPriceLine()` calls — the user maintains these manually

## Odoo Gotchas

- **Manifest licence field:** Odoo only accepts a fixed set of strings. Use `"Other OSI approved licence"` (British spelling) for MIT — `"MIT"` alone will fail validation.
- **`auth="public"` routes writing to the ORM:** every model in this addon except `dankbit.trade` only has `ir.model.access.csv` rules for `base.group_user` (internal users) — no public/portal grant. Reading via raw SQL (`request.env.cr.execute(...)`) bypasses this entirely, but any `self.search()`/`self.create()`/`record.write()` call reached from a public route needs an explicit `.sudo()`, or it 403s for genuinely anonymous requests. This is easy to miss in testing: `odoo shell` runs as superuser (no check at all), and hitting the route from a browser tab that's already logged into the backend authenticates as that admin user instead of the public one — only a cookie-less request (e.g. plain `curl`) exercises the real public-user path. See `dankbit.zones.extrema._persist_extrema()` for the fix.
